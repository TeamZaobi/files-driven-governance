# 运行观察与能力晋升

这份 reference 只回答一件事：`runtime -> candidate -> capability` 这条链什么时候可以走，什么时候必须停。

## 总原则

- 当前回合先记运行观察，不热改能力真源。
- 先把事实、证据和历史分开，再决定是否进入候选。
- 展示页只能展示，不得冒充正式激活。
- 任何能力候选都必须有试验条件、失败信号和回退路径。

## 什么时候只记运行观察

只记运行观察，不做晋升判断，适用于下面几种情况：

- 你看到的是一次执行中的现象，还没有足够证据说明这是一条稳定规律。
- 你只能确认“发生了什么”，还不能确认“为什么发生”。
- 当前材料只是截图、日志、口头复述或临时摘要，没有形成可复核的证据包。
- 这轮动作还在运行态，不能把观察直接写回能力真源。

运行观察的目标是保留现场，不是先下结论。

## 什么时候做证据包

当运行观察已经足够稳定，可以把它整理成证据包：

- 观察对象清楚，范围也清楚。
- 证据能复核，且能说明同一现象不是偶然噪声。
- 你已经能写出“发生了什么、来自哪里、为什么值得继续看”。
- 后续是否晋升还没决定，但材料已经够进入正式评审。

证据包的作用是把观察变成可引用材料，而不是直接变成能力定义。

## 什么时候召回历史

当新观察要和旧事实比较时，先召回历史：

- 现象和过往案例相似，但现在还不能确认是否同类问题。
- 你需要检查之前是否已经处理过类似链路、类似失败或类似约束。
- 你怀疑当前观察只是旧结论的重复，而不是新的能力信号。

召回历史的目的不是翻旧账，而是确认新材料到底是在补证，还是在重复已有结论。

## 什么时候拆成项目出口与能力候选

当材料已经说明“这里存在分岔”时，就拆成两条线：

- 项目出口：当前项目要交付、展示或落地的具体结果。
- 能力候选：可以被提炼、试验并可能晋升为稳定能力的部分。

如果不拆开，项目交付会和能力建设互相污染，最后谁都说不清是在交付事实，还是在改本体。

拆分时必须同时明确：

- 哪部分是项目事实，不能改写成能力规则。
- 哪部分是能力候选，只能先进入试验态。
- 哪些内容只是展示，不是正式结论。

## 什么时候进入待试验

能力候选只有在满足下面条件时才进入待试验：

- 已经完成证据包。
- 已经召回过相关历史。
- 已经拆清项目出口与能力候选。
- 能写出试验窗口、成功条件和失败信号。

待试验不是“已经生效”，而是“可以被有限验证”。

## 什么时候允许激活或回退

只有当待试验满足下列条件，才允许激活或回退：

- 试验窗口结束，且结果可判定。
- 成功条件命中，才允许激活。
- 失败信号命中，必须回退。
- 回退路径已经在候选阶段预先声明，不能临场补写。

激活不是展示页上的一句话，而是可以被合同、状态和事件共同证明的结果。

## 这条链的硬约束

- 当前回合不热改能力真源。
- 展示页不能冒充正式激活。
- 能力候选必须先试验，再谈激活。
- 失败信号和回退路径必须先定义，再谈晋升。
