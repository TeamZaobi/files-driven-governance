# 贡献指南

欢迎改进 `files-driven`。

本仓库的维护原则很简单：**技能真源、参考件、仓库包装文档分开维护**。

## 1. 改动前先判断改的是哪一层

### 改 `SKILL.md`

只有在以下情况才优先改这里：

- 技能定位变了
- 触发条件变了
- 核心工作流变了
- 输出要求变了
- 默认边界约束变了

### 改 `references/`

以下情况优先改参考件：

- 新增稳定做法
- 新增判断规则
- 新增共享约定
- 新增跨工具处理规则
- 新增流程说明、对照表或场景手册

### 改仓库包装文档

包括：

- `README.md`
- `docs/`
- `SECURITY.md`
- `CHANGELOG.md`
- `agents/openai.yaml`

这些文件面向仓库读者，不应反过来定义技能真源。

## 2. 贡献时遵守的语言规则

本项目现在统一使用中文主导的语言体系。
新增内容前，请先看 [docs/语言体系规范.md](docs/语言体系规范.md)。

至少遵守这几条：

1. 面向人的说明默认写中文
2. 首次出现的核心概念先写中文，再补稳定键名
3. 不要把英文原词当成默认解释
4. 不要为同一概念发明多个叫法

## 3. 内容层面的贡献原则

1. 优先保持跨工具通用，不把某个工具的局部习惯写成项目原则
2. 优先保持真源清晰，不让包装文档越权
3. 优先补稳定参考件，不把 `SKILL.md` 写成百科全书
4. 优先降低认知门槛，不新增无必要的工程黑话
5. 优先治理项目结构，再谈目录外观

## 4. 提交前检查

建议在提交前执行：

```bash
python3 /Users/jixiaokang/.codex/skills/.system/skill-creator/scripts/quick_validate.py .
```

同时人工检查：

1. `SKILL.md` 与 `references/` 的链接是否仍然正确
2. 新术语是否已经在 [docs/语言体系规范.md](docs/语言体系规范.md) 固定主叫法
3. `agents/openai.yaml` 是否仍与对外定位一致
4. 包装文档是否没有反向定义技能真源

## 5. 合并请求（Pull Request）说明建议

PR 说明至少回答：

1. 这次改的是技能真源、参考件，还是仓库包装
2. 为什么需要这次改动
3. 是否影响触发条件、输出要求或跨工具兼容性
4. 是否需要同步更新 `README.md`、`docs/完整说明书.md` 或 `agents/openai.yaml`
5. 是否引入了新的术语，如果有，中文主叫法是什么
