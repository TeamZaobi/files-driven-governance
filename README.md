# 文档驱动的项目治理（files-driven）

当前版本：`v0.2.7`

`files-driven` 是一个面向 AI Agent 项目、AI 驱动流程项目和 OpenClaw 类项目的治理技能。
它不把“整理文件”理解成目录美化，而是把文档当成项目治理的承载介质，用来回答：

1. 项目的事实到底写在哪
2. 哪些文档在定义事实，哪些只是在总结、展示或搬运
3. 多人、多代理、多工具一起工作时，谁能写、谁先读、谁负责复核

## 这次语言升级做了什么

这个仓库原先长期混用中文、英文原词和软件工程黑话。
本轮调整把语言体系统一成“中文优先、必要时补稳定键名”的写法，目标是：

1. 首次阅读能直接读懂
2. 核心概念有固定中文主叫法
3. 英文只保留给文件名、工具名、稳定键名和外部方法名

统一规则见 [语言体系规范](docs/语言体系规范.md)。

## 它适合什么场景

这个技能主要适合三类项目：

1. 现有仓库诊断
   - README、状态页、任务、规则和流程文档已经开始互相漂移
2. 新项目搭建
   - 想要一套早期够用、后续还能扩展的治理结构
3. 漂移后收口
   - 项目已经出现重复真源、交接失效、状态失真或工具入口越权

## 核心概念

本项目统一把项目结构分成八类：

1. 规则与约束层（`policy_or_rules`）
2. 对象层（`object`）
3. 流程层（`workflow`）
4. 技能层（`skill`）
5. 角色层（`agent`）
6. 过程载体层（`execution_object`）
7. 状态摘要层（`status_projection`）
8. 展示输出层（`display_projection`）

另外还有几个必须固定的基础词：

- 真源：真正定义事实的上游文档
- 当前版本锚点：当前该读哪一版
- 官方读取顺序：先读什么，后读什么
- 工具适配入口：工具侧的包装、启动入口和映射面
- 跨层共享约定：谁写、谁读、怎么同步
- 意图触发约定：短口令如何稳定落地

## 这个技能会做什么

使用 `files-driven` 时，它会优先完成这些事：

1. 先锁定方向与边界
   - 使用场景
   - 首批交付物
   - 核心用户故事
   - 验收或测试用例
   - 非目标
2. 做项目诊断
   - 项目阶段
   - 风险
   - 协作密度
   - 自主度
   - 恢复压力
   - 工具异构度
   - 文档膨胀
3. 划分结构家族
   - 看清每类文档的真源、责任边界、同步顺序和关口
4. 设计跨层共享约定
   - 明确谁能写、谁只能总结、谁负责交接
5. 选择治理组合
   - 规格驱动（`Spec-Driven`）
   - 看板流（`Kanban`）
   - 迭代节奏（`Agile / Sprint-like`）
6. 选择默认流程库
   - 哪些日常默认启用
   - 哪些只在风险上升时启用
   - 哪些当前明确不启用
7. 给出有顺序的治理方案
   - 不停在概念层，而是明确先做什么、后做什么

## 默认工作方式

这个技能的默认做法有几条硬约束：

1. 先诊断，再开药方
2. 先确认边界，再谈目录和治理强度
3. 先分清真源和摘要页，再决定命名和层级
4. 先保证跨工具可迁移，再考虑某个工具的局部便利
5. 输出最小够用方案，不默认展开完整治理大图

它默认支持多工具环境，包括但不限于：

- `Claude Code`
- `Codex`
- `AntiGravity`
- `OpenClaw`

但不会把任何工具名直接当成项目角色名。

## 典型触发句

可以直接这样使用：

- “分析这个仓库里哪些文档是真源，哪些只是状态摘要。”
- “我想给这个 AI Agent 项目设计一套早期够用的文档治理结构。”
- “这个项目的 README、任务和状态页已经互相漂移了，请给出收口顺序。”
- “我们想用‘继续开发’‘开始审计’这类短口令驱动工作，请帮我定义稳定约定。”

## 输出会长什么样

这个技能默认不会一上来给你一套很厚的治理全案。
它先输出核心必答区块，只在确实必要时再补条件区块。

核心必答区块包括：

1. 方向与边界锚点
2. 项目画像
3. 当前主要失真或治理压力
4. 推荐治理模式
5. 推荐经典流程库
6. 项目结构家族图
7. 推荐入口与恢复链
8. 推荐下一步实施顺序
9. 明确不建议的做法

只有在问题确实涉及多人、多代理、多工具、版本漂移或文档膨胀时，才会补：

- 跨层共享矩阵
- 推荐项目结构分层
- 推荐角色控制回路
- 推荐版本与同步纪律
- 对象家族检索与适配策略
- 意图触发与执行约定
- 工具可移植性约束
- 文档生命周期与压缩策略
