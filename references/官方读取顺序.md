# 官方读取顺序

这个文件用于固定五类核心结构的默认读取顺序：

1. `policy_or_rules`
2. `object`
3. `workflow`
4. `skill`
5. `agent`

本文件只在结构家族和真源入口已大致判清后使用。
它负责稳定“先读什么、后读什么”，不负责替代真源判断或结构定位。

## 总规则

先读真源，再读当前版本锚点，再读最近批准的变更，最后才读摘要页、工具入口和历史材料。
读取顺序是为了稳定接手，不是要求把所有入口都机械读完；
先停在能完成当前判断的最短点。

如果某个结构只能从工具启动文件里找到，说明治理还不稳定，应回到真源和定位约定层修正。

## 分家族读取顺序

### 1. `policy_or_rules`

1. 规则索引或当前规则锚点
2. 当前规则主文档
3. 最近批准的规则变更
4. 状态摘要或执行记录
5. 工具入口最后再看

### 2. `object`

1. 对象索引或对象注册表
2. 当前对象定义或模式文档
3. 最近批准的对象变更
4. 消费它的流程文档
5. 摘要页和展示页最后再看

### 3. `workflow`

1. 流程索引或流程注册表
2. 当前流程定义
3. 最近批准的流程修订
4. 看板、状态页或执行脚本
5. 工具入口最后再看

### 4. `skill`

1. 技能索引或技能注册表
2. 技能包根目录
3. `SKILL.md`
4. 技能实际引用的辅助文件
5. 工具包装或外部示例

### 5. `agent`

1. 角色索引或角色注册表
2. 当前角色约定
3. 角色专属边界说明
4. 运行时映射
5. 工具入口最后再看

## 恢复模式下的覆盖规则

如果仓库已经漂移，读取时先加一个前置动作：

1. 先判断哪些入口还可信
2. 先挑出当前真源基线
3. 再按上面的顺序读取

不要在漂移状态下机械执行读取顺序，而不先判断“谁还值得信”。

## 工具侧规则

工具入口、启动说明、包装命令和运行时提示，默认都只算适配入口。
除非项目明确把它们提升成真源，否则它们没有资格改写上游定义。

如果当前还说不清五类核心结构在哪里，先读 [结构家族定位约定](结构家族定位约定.md)。
如果多个角色或工具会共享同一事实，再读 [跨层共享约定](跨层共享约定.md)。
