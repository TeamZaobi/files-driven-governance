# 官方读取顺序

这个文件用于固定五类核心结构的默认读取顺序：

1. `policy_or_rules`
2. `object`
3. `workflow`
4. `skill`
5. `agent`

本文件只在结构家族和真源入口已大致判清后使用。
它负责稳定“先读什么、后读什么”，不负责替代真源判断或结构定位。

## 总规则

先读真源，再读当前版本锚点，再读最近批准的变更，最后才读摘要页、工具入口和历史材料。
读取顺序是为了稳定接手，不是要求把所有入口都机械读完；
先停在能完成当前判断的最短点。

如果某个家族已经同时存在：

1. 解释文档
2. 结构化合同

并且合同已经承担机读控制语义，
默认先读合同，再读解释文档。
不要反过来让解释文档重新定义合同已经冻结的状态、转移或权限。

如果某个受控 workflow 目录里同时出现：

1. `WORKFLOW.md`
2. `workflow.contract.json`
3. `workflow.state.json`
4. `workflow.events.jsonl`
5. `checks/`

默认按家族读取，而不是按共址读取：

1. 先读 workflow 合同和解释文档
2. 再读 `execution_object`
3. 然后才读 `status_projection`
4. `display_projection` 最后再读

不要因为它们放在一起，就把它们当成同一个真源包。

如果某个结构只能从工具启动文件里找到，说明治理还不稳定，应回到真源和定位约定层修正。

## 分家族读取顺序

### 1. `policy_or_rules`

1. 规则索引或当前规则锚点
2. 当前规则合同（如果存在）
3. 当前规则主文档
4. 最近批准的规则变更
5. 状态摘要或执行记录
6. 工具入口最后再看

### 2. `object`

1. 对象索引或对象注册表
2. 当前对象 schema（如果存在）
3. 当前对象定义或模式文档
4. 最近批准的对象变更
5. 消费它的流程文档
6. 摘要页和展示页最后再看

### 3. `workflow`

1. 流程索引或流程注册表
2. 当前 workflow 合同（如果存在）
3. 当前流程定义
4. 最近批准的流程修订
5. 需要判断当前运行时，再继续读 `execution_object`
6. 看板、状态页或执行脚本只当消费方
7. 工具入口最后再看

### 4. `skill`

1. 技能索引或技能注册表
2. 技能包根目录
3. 技能 manifest（如果存在）
4. `SKILL.md`
5. 技能实际引用的辅助文件
6. 工具包装或外部示例

### 5. `agent`

1. 角色索引或角色注册表
2. 当前角色合同（如果存在）
3. 当前角色约定
4. 角色专属边界说明
5. 运行时映射
6. 工具入口最后再看

## 条件扩展：运行实例与投影

只有在需要判断“当前这次到底跑到哪、能不能继续、怎么接手”时，
才在五类核心结构之后继续读这一组。

### 6. `execution_object`

1. 当前运行实例
2. 当前事件流、决策记录或验证记录
3. 当前过程压缩面只作补充，不取代实例

### 7. `status_projection`

1. 当前状态摘要、handoff 包或恢复页
2. 发现状态、关口或下一步和上游不一致时，立即回 workflow 合同与 execution_object
3. 工具状态页最后再看

### 8. `display_projection`

1. 当前汇报版、对外版或展示版
2. 需要核真的地方，回上游状态与实例
3. 不要把展示页当成控制判断的第一入口

## 恢复模式下的覆盖规则

如果仓库已经漂移，读取时先加一个前置动作：

1. 先判断哪些入口还可信
2. 先挑出当前真源基线
3. 再按上面的顺序读取

不要在漂移状态下机械执行读取顺序，而不先判断“谁还值得信”。

## 工具侧规则

工具入口、启动说明、包装命令和运行时提示，默认都只算适配入口。
除非项目明确把它们提升成真源，否则它们没有资格改写上游定义。

如果当前还说不清五类核心结构在哪里，先读 [结构家族定位约定](结构家族定位约定.md)。
如果多个角色或工具会共享同一事实，再读 [跨层共享约定](跨层共享约定.md)。
