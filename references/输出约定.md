# 输出约定

正式回答时，默认按本文件的区块顺序组织内容。
在诊断完成前，不要直接跳去讲目录、模板或文件树。

## 核心必答区块

### 1. 方向与边界锚点

要概括清楚：

- 首批真实使用场景
- 首批交付物
- 一到三个核心用户故事
- 足够判断通过、失败和越界的测试锚点
- 明确非目标
- 只用于校准质量的参考对象

这一段优先用说人话的方式写。
不要先抛架构词、工具名或目录方案。

如果边界还不稳，要把以下内容直接写出来：

- 四到六个说人话确认问题
- 一到三个用户故事草案
- 三到八个测试用例草案
- 至少一个非目标
- 一位验收责任人或一类验收对象

如果边界仍然不稳，要明确说出这一点，并限制后续治理细节，只保留那些即使边界再澄清一轮也不会失效的建议。

### 2. 项目画像

要概括：

- 项目类型
- 当前阶段
- 当前目标
- 主要角色或代理
- 自主度
- 当前协作形态
- 当前理解把握度和主要未知点

### 3. 当前主要失真或治理压力

只点出真正驱动建议的几个压力，不要铺满清单。
常见压力包括：

- 故事边界或验收边界漂移
- 重复真源
- 摘要越权
- 版本不清
- 交接弱
- 过程记录过载
- 状态漂移
- 恢复成本高
- 文档膨胀和读取成本高

### 4. 推荐治理模式

写清楚选了哪种治理模式或哪种组合，并解释为什么它适合当前诊断。

### 5. 推荐经典流程库

至少判断以下流程里哪些应该：

1. 变成默认做法
2. 只在条件触发时启用
3. 当前明确延后

重点判断：

- `low_token_recovery_chain`
- `discussion -> decision_package -> task_or_decision`
- `truth_source -> execution_object -> status_projection -> display_projection`
- `mechanism_review`
- `adversarial_inquiry -> defense -> convergence`
- `proposal -> validation -> shadow/canary -> activation_or_rollback`
- `skill_seed -> package_contract -> active_package`
- `contract_gap -> closure_topic -> downstream_resume`
- `growth_signal -> lifecycle_review -> compact_or_archive`
- `ops_signal -> classify -> state_refresh_or_local_fix_or_dev_change_or_rollback`
- `observation -> mechanism_review -> candidate_or_absorb`

只解释和当前项目真相关的流程。

### 6. 项目结构家族图

对以下八类结构都给出当前或建议形态：

1. `policy_or_rules`
2. `object`
3. `workflow`
4. `skill`
5. `agent`
6. `execution_object`
7. `status_projection`
8. `display_projection`

每一类至少写清：

- 真源
- 主要摘要或工具适配入口
- 责任边界
- 是否需要更强关口

### 7. 推荐入口与恢复链

至少写清：

1. 新接手时先读什么
2. 当前工作主要放在哪
3. 当前可信事实去哪里找
4. 历史材料什么时候再读

顺序一定要短，短到可以真的拿来执行。

### 8. 推荐下一步实施顺序

给出短而明确的顺序。
如果存在边界漂移，优先先稳边界，再做结构细化。

### 9. 明确不建议的做法

点出当前项目最容易犯、且代价最大的几个错误。
常见例子：

- 过早过度设计
- 方向与边界还没稳就扩首批交付物
- 让状态页定义事实
- 结构角色还没分清就先加目录
- 让讨论记录长期兼任任务桶
- 让规则、流程、技能和角色互相改义

## 条件区块

只有在诊断证明确实重要时才加，不要为了齐整而硬补。

### A. 跨层共享矩阵

对重要结构写清：

- 生产者
- 主要消费者
- 可直接写的层
- 只能总结或展示的层
- 可见范围
- 同步触发时机
- 冲突处理规则

启用提示：

- 不止一个人、不止一个代理或不止一个工具入口会碰同一事实时，加这一段

### B. 推荐项目结构分层

把文档系统映射到四层：

1. `truth_source`
2. `execution_object`
3. `status_projection`
4. `display_projection`

同时解释结构家族如何落在这四层上。

启用提示：

- 分层混乱本身就是问题时，加这一段

### C. 推荐角色控制回路

至少写清：

1. 谁观察
2. 谁决策
3. 谁执行
4. 谁复核
5. 谁能回退或持续改进

如果项目用了代理，不要只写角色名，要写清每类角色承担什么文档责任。

启用提示：

- 自主度、角色歧义或复核责任是关键问题时，加这一段

### D. 推荐版本与同步纪律

至少写清：

- 当前版本锚点
- 更新顺序
- 责任边界
- 是否需要 `sync_pending` 之类的标记
- 什么情况下必须先复核再升级
- 哪些边界需要更严格的禁止覆盖规则

启用提示：

- 版本不清、漂移明显、多人并写或升级复杂时，加这一段

### E. 对象家族检索与适配策略

对以下五类核心结构写清：

- `policy_or_rules`
- `object`
- `workflow`
- `skill`
- `agent`

至少包含：

- 真源定位点
- 当前版本锚点
- 官方读取顺序
- 工具适配入口
- 哪些适配入口只能总结，不能改写
- 当前哪些地方还难找

启用提示：

- 五类核心结构难找，或工具入口会遮住真源时，加这一段

### F. 工具可移植性约束

写清：

- 哪些概念不能绑定到单一工具
- 哪些工具入口只是适配，不是真源
- 工具切换时必须保留什么
- 哪些地方最容易被某个工具的局部习惯绑死

启用提示：

- 多工具同时存在，或未来迁移风险明显时，加这一段

### G. 意图触发与执行约定

对主要短口令写清：

- 主意图编号
- 允许的别名
- 修饰项
- 默认读取顺序
- 绑定流程
- 执行角色
- 输出要求
- 歧义处理规则

启用提示：

- 项目希望通过自然语言短口令驱动工作时，加这一段

### H. 文档生命周期与压缩策略

写清：

- 哪些文档还算活跃
- 哪些文档应该降级
- 哪些文档应该压缩
- 哪些文档应该归档
- 压缩时如何不破坏恢复链

启用提示：

- 活跃文档膨胀、陈旧页堆积、读取成本上升时，加这一段

### I. 输入分型与低带宽压缩包

先写清：

- 当前用户输入主要属于 `understanding / constraint / evidence / decision` 哪一类
- 是否做过反例检查
- 如果不是纯理解同步，它实际改写了什么

如果确认主要是 `understanding`，再补一个最小压缩包，优先从下面四种里选：

- `anchor package`
- `decision package`
- `evidence package`
- `route package`

写的时候至少说清：

- 当前锚点或当前路径
- 当前结论依据
- 用户现在真正需要接住的最小信息
- 哪些历史材料这轮先不用展开

启用提示：

- 已出现解释型摩擦
- 用户在要求“讲清楚”“解释一下”“再检查一下”
- 项目本身依赖低带宽短口令推进
- agent 手里掌握的信息明显比用户多很多

### J. 开发/运营双治理结构

写清：

- 开发面哪些对象在定义真源，哪些过程载体负责推进变更
- 运营面哪些过程载体在承接运行、观察、事件、补救与恢复
- 哪些信号只刷新状态，哪些只做局部修复，哪些必须升级成开发变更或回退
- 哪些运行观察只留在候选层，哪些才允许吸收到长期资产

启用提示：

- 项目同时持续开发和持续运行
- 运行信号会持续反推规则、对象、流程或验收边界
- 团队已经明显感到“开发任务”和“运营事件”在抢同一条治理链

## 按起点补充的内容

### 现有仓库

额外强调：

- 当前真源地图
- 当前漂移点
- 当前迁移顺序

### 新项目

额外强调：

- 最小真源集合
- 最小过程载体集合
- 第一条恢复链

### 恢复与重整

额外强调：

- 当前可信事实基线
- 立即止血动作
- 第一轮降级和压缩顺序

## 推理纪律

回答时保持以下纪律：

1. 证据和推断分开写
2. 假设写明白
3. 只展开当前问题真正需要的部分
4. 没读清边界前，不要先下重结论
