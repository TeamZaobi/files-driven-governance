# 输出约定

正式回答时，默认先用最小骨架回答。
不要在诊断完成前，直接跳去讲目录、模板、文件树、schema 清单或完整流程库。

## 默认回答骨架

### 1. 边界与假设

先概括清楚：

- 首批真实使用场景
- 首批交付物
- 一到三个核心用户故事
- 足够判断通过、失败和越界的测试锚点
- 明确非目标
- 只用于校准质量的参考对象

规则：

- 这一段优先用说人话的方式写
- 边界还不稳时，直接写一到三个确认问题，不要先给完整治理蓝图
- 如果本轮把握度仍是 `medium`，明确写出关键假设

### 2. 四层与真源

默认只回答四层：

1. `truth_source`
2. `execution_object`
3. `status_projection`
4. `display_projection`

至少写清：

- 当前可信真源在哪里
- 哪些文件在推进执行或持有实例
- 哪些文件只是摘要或展示
- 哪些投影目前已经越权

只有在责任边界、机读合同或工具适配会受影响时，
才补可选 `source_kind` 标签：

- `policy_or_rules`
- `object`
- `workflow`
- `skill`
- `agent`

不要默认枚举全部 8 类结构家族，更不要把它们当作所有回答的必答题。

### 3. 当前主要失真

只点出真正驱动建议的几个压力，不要铺满清单。
常见压力包括：

- 故事边界或验收边界漂移
- 重复真源
- 摘要越权
- 版本锚点不清
- 交接弱
- 恢复成本高
- workflow 仍停留在 prose-first
- 讨论一直停在聊天层
- 短口令或解释型输入反复滑路

只解释和当前问题真正相关的几项。

### 4. 下一步动作

给出短而明确的一到三条动作，不默认展开全套治理编制。

常见顺序：

- `existing_repo`
  - 先找真源与越权投影
  - 先止漂移，再修命名和目录
- `greenfield`
  - 先稳边界
  - 再定最小真源与恢复入口
- `recovery_or_realignment`
  - 先止血
  - 再选定可信事实并降级旧摘要

不要默认要求：

- 推荐完整治理模式
- 枚举完整经典流程库
- 逐类说明全部 8 个结构家族
- 补全所有条件区块

## 条件加段

只有在诊断证明确实重要时才加，不要为了齐整而硬补。

### A. governed pack / harness / 受控 workflow

出现以下话题时，加这一段：

- `BOUNDARY.md`
- `workflow.contract.json`
- `objects/*.json`
- `workflow.state.json`
- `workflow.events.jsonl`
- `status.projection.json`
- validator、CI、hook、check、harness

至少写清：

1. 为什么 `BOUNDARY.md` 必须先于 workflow 合同
2. 最小合同链的顺序
3. 哪些资产是解释层，哪些是机读控制真源，哪些是实例，哪些是投影
4. validator 与 CI 在哪里承担确定性检查

### B. discussion 收口与晋升

当议题还没到 `task / decision`，或争议已经影响关键决策时，加这一段。

至少写清：

1. 当前停在 `discussion`、`adversarial inquiry`、`decision_package` 还是 `task_or_decision`
2. 为什么现在该停在这一层
3. 何时才需要升级到质询
4. 何时只补 `process_projection`

### C. 共享存储与跨层共享

只有多人、多代理或多工具会碰同一事实时，加这一段。

至少写清：

- 生产者
- 主要消费者
- 哪一层能直接写
- 哪一层只能总结或展示
- 冲突时按什么规则处理

### D. 关口硬化

只有反复出现“该停没停、该问没问、该验没验”时，加这一段。

至少写清：

- 当前失效主要是 `route / evidence / write / stop` 哪一类
- 哪些状态禁止哪些下游输出
- 检查落在脚本、hook、validator 还是固定回归

### E. 官方读取顺序与恢复

只有真源入口不清、版本锚点不清或交接失败明显时，加这一段。

至少写清：

1. 新接手时先读什么
2. 当前可信事实去哪里找
3. 历史材料什么时候再读

顺序一定要短，短到可以真的拿来执行。

### F. 低带宽理解 / 短口令

只有出现解释型摩擦或项目已经决定用短口令做稳定入口时，加这一段。

至少写清：

- 当前属于 `understanding / constraint / evidence / decision` 哪一类
- 这次该补解释、补证据还是重路由
- 哪些短口令已经制度化，哪些还不能靠聊天默契驱动

## 明确不建议的做法

- 方向与边界还没稳就扩首批交付物
- 让 `status_projection` 或 `display_projection` 定义事实
- 把与 workflow 共置的实例或状态页误当成 workflow 真源
- 把 runtime / 工具品牌抬成项目角色
- 把 schema、validator 或流程库变成所有回答的默认主叙事
- 把 discussion 长期留在聊天层，却不给晋升或收口判断
- pack 里跳过 `BOUNDARY.md`，直接堆 workflow / objects / state
