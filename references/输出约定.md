# 输出约定

正式回答时，默认先用最小骨架回答。
不要在诊断完成前，直接跳去讲目录、模板、文件树、schema 清单或完整流程库。
即使要讲方法学，也先把回答翻回“哪份文件算数、今天先做什么、哪些先别改”，
再按需补 `scope`、结构家族、合同链和控制模型。

## 默认回答骨架

### 1. 边界与假设

先概括清楚：

- 首批真实使用场景
- 首批交付物
- 一到三个核心用户故事
- 足够判断通过、失败和越界的测试锚点
- 明确非目标
- 只用于校准质量的参考对象

规则：

- 这一段优先用说人话的方式写
- 边界还不稳时，直接写一到三个确认问题，不要先给完整治理蓝图
- 如果本轮把握度仍是 `medium`，明确写出关键假设
- 如果对方明显几乎没有软件工程基础，首屏先写“这次要做什么 / 先别做什么 / 卡住时找哪里”，不要先抛 `scope / tranche / harness`

### 2. 当前作用域与归属判断

默认先标当前在：

1. `capability_scope`
2. `project_scope`
3. `runtime_scope`

然后再回答这四类归属：

1. `入口规则`
2. `能力规则`
3. `项目规则`
4. `项目实体`

至少写清：

- 当前在什么 scope 说话
- 当前动作先归哪一类
- 为什么它应该放在这一类，而不是其他三类
- 当前是否需要一级关口或二级升级

默认规则：

- 这四类归属默认描述 `project_scope`
- 如果治理对象正好是 `files-driven` 自己，先显式写出 `self-hosting`
- 内部仍然要完成作用域与归属判断，但对零基础协作者，对外优先翻译成“规则问题 / 项目内容 / 工具设置 / 状态说明”

### 3. 四层与真源

只有顶层归属已经判清，才继续回答四层：

1. `truth_source`
2. `execution_object`
3. `status_projection`
4. `display_projection`

至少写清：

- 当前可信真源在哪里
- 哪些文件在推进执行或持有实例
- 哪些文件只是摘要或展示
- 哪些投影目前已经越权

只有在责任边界、机读合同或工具适配会受影响时，
才补可选 `source_kind` 标签：

- `policy_or_rules`
- `object`
- `workflow`
- `skill`
- `agent`

不要默认枚举全部 8 类结构家族，更不要把它们当作所有回答的必答题。
也不要让它们抢走 `Skill` 和项目实体的顶层主语。

### 4. 下一步动作与升级口

给出短而明确的一到三条动作，不默认展开全套治理编制。
优先写成：

`动作触发 -> 一级关口 -> 二级升级`

常见顺序：

- `existing_repo`
  - 先冻结当前动作的归属判断
  - 再找真源与越权投影
  - 先止漂移，再修命名和目录
- `greenfield`
  - 先稳边界
  - 再定最小入口规则与 `Skill` 主容器
  - 最后补第一批项目规则和项目实体
- `recovery_or_realignment`
  - 先止血
  - 再选定可信事实并降级旧摘要

补充规则：

- 对零基础协作者，动作尽量翻成“先看哪个文件 -> 做什么 -> 做完后问什么”，不要只给抽象关口名

不要默认要求：

- 推荐完整治理模式
- 枚举完整经典流程库
- 逐类说明全部 8 个结构家族
- 补全所有条件区块

## 条件加段

只有在诊断证明确实重要时才加，不要为了齐整而硬补。

### A. governed pack / harness / 受控 workflow

出现以下话题时，加这一段：

- `BOUNDARY.md`
- `workflow.contract.json`
- `objects/*.json`
- `workflow.state.json`
- `workflow.events.jsonl`
- `status.projection.json`
- validator、CI、hook、check、harness

至少写清：

1. 为什么 `BOUNDARY.md` 必须先于 workflow 合同
2. 最小合同链的顺序
3. 哪些资产是解释层，哪些是机读控制真源，哪些是实例，哪些是投影
4. validator 与 CI 在哪里承担确定性检查

### B. discussion 收口与晋升

当议题还没到 `task / decision`，或争议已经影响关键决策时，加这一段。

至少写清：

1. 当前停在 `discussion`、`adversarial inquiry`、`decision_package` 还是 `task_or_decision`
2. 为什么现在该停在这一层
3. 何时才需要升级到质询
4. 何时只补 `process_projection`

### C. 共享存储与跨层共享

只有多人、多代理或多工具会碰同一事实时，加这一段。

至少写清：

- 生产者
- 主要消费者
- 哪一层能直接写
- 哪一层只能总结或展示
- 冲突时按什么规则处理

### D. 关口硬化

只有反复出现“该停没停、该问没问、该验没验”时，加这一段。

至少写清：

- 当前失效主要是 `route / evidence / write / stop` 哪一类
- 哪些状态禁止哪些下游输出
- 检查落在脚本、hook、validator 还是固定回归

### E. 官方读取顺序与恢复

只有真源入口不清、版本锚点不清或交接失败明显时，加这一段。

至少写清：

1. 新接手时先读什么
2. 当前可信事实去哪里找
3. 历史材料什么时候再读

顺序一定要短，短到可以真的拿来执行。

### F. 低带宽理解 / 短口令

只有出现解释型摩擦或项目已经决定用短口令做稳定入口时，加这一段。

至少写清：

- 当前属于 `understanding / constraint / evidence / decision` 哪一类
- 这次该补解释、补证据还是重路由
- 哪些短口令已经制度化，哪些还不能靠聊天默契驱动
- 如果对象是非工程背景读者，优先给 `anchor package` 或 `route package`，先别丢完整术语表

### G. self-hosting / 作用域绑定

只有当治理对象正好是 `files-driven` 自己，或同一文件在不同上下文里出现双重身份时，加这一段。

至少写清：

1. 当前在 `capability_scope`、`project_scope` 还是 `runtime_scope`
2. 为什么这次需要显式切作用域
3. 哪个文件或 `Skill` 在跨 scope 复用
4. 这次最终按哪个 `(scope, file_ref)` 做判断

### H. 运行观察 / 候选保留 / 能力晋升

只有当问题明确在问“运行中反复纠偏时该停在哪一层”时，加这一段。

默认入口顺序：

1. 先读 [运行观察与能力晋升](./运行观察与能力晋升.md)
2. 再看 [capture-candidate-activation](../examples/capture-candidate-activation/README.md)
3. 还要追总模型时，再回 [经典治理流程库](./经典治理流程库.md) 和 [项目治理能力模型](../docs/项目治理能力模型.md)

至少写清：

1. 当前材料是运行记录、能力候选还是正式能力真源
2. 为什么这次不能直接热改真源，或者为什么已经可以晋升
3. 需要哪些复核证据或跨场景验证，才可从记录进候选、再从候选进能力
4. 这次先读哪份 reference、再看哪份 example pack
5. 如果暂不晋升，下一轮激活条件是什么

### I. 对话链接 / 宿主内链分流

只有当输出会真的落盘到宿主里的 Markdown 文件，或用户明确在问 Obsidian / wiki 兼容时，加这一段。

至少写清：

1. 当前这条链接是只留在对话层展示，还是会写进宿主里的 Markdown
2. 如果落盘目标是 Obsidian vault 内 `.md`，内部引用应写成 `[[笔记名]]` 或 `[[相对路径/笔记名|显示名]]`
3. 如果目标是库外本地文件或外部 URL，继续使用普通 Markdown 链接
4. 不要把桌面端可点击绝对路径文件链接直接落盘进项目 Markdown
5. “能点开文件”不等于“进入宿主的双向链接 / backlink / graph 体系”

## 明确不建议的做法

- 方向与边界还没稳就扩首批交付物
- 让 `status_projection` 或 `display_projection` 定义事实
- 把与 workflow 共置的实例或状态页误当成 workflow 真源
- 把 runtime / 工具品牌抬成项目角色
- 把 schema、validator 或流程库变成所有回答的默认主叙事
- 把 discussion 长期留在聊天层，却不给晋升或收口判断
- pack 里跳过 `BOUNDARY.md`，直接堆 workflow / objects / state
- 在对话层验证可点击后，就把绝对路径文件链接原样写进宿主里的 Markdown
