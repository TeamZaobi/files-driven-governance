# 执行面判定与 CLI 生产策略

当问题已经确定需要补控制，但还没判断主线程、`subagent`、宿主 workflow、`CLI`、runner 应该怎么分工时，用这份 reference。

它回答两件事：

1. 当前动作应该落在哪个执行面
2. `CLI` 到底什么时候能算生产工具

## 执行面对照表

| 执行面 | 最适合做什么 | 主要收益 | 不适合做什么 |
| --- | --- | --- | --- |
| 宿主主线程 | 短任务、轻判断、即时分析 | 最轻、上下文连续、协作成本最低 | 长链路治理、可恢复编排 |
| 宿主 `subagent` | 有边界的并行子任务、认知隔离 | 上下文隔离、并行探索 | 需要明确退出码和文件级编排 |
| 宿主原生 `workflow / approvals` | 阶段化推进、宿主内 resume / checkpoint | 低耦合复用宿主能力 | 跨宿主迁移、外部脚本拼接 |
| `CLI` | 有明确输入输出的节点执行 | 可拼接、可落文件、可取退出码 | 目标不清的探索性讨论 |
| 脚本 runner | 多节点编排、state/events、恢复与审计 | 外置控制、跨工具稳定执行 | 低风险一次性小任务 |

## 主线程 / `subagent` / `CLI` / runner 最小判定表

### 用主线程

当满足下面多数条件时，直接用主线程：

- 任务短
- 边界清楚
- 风险低
- 不需要回放
- 不需要中断恢复

### 用 `subagent`

当满足下面多数条件时，优先宿主 `subagent`：

- 需要换个干净上下文
- 需要并行看几个问题
- 子任务足够独立
- 结果主要是认知产物，不是严格 runtime artifact

### 用 `CLI`

当满足下面多数条件时，优先 `CLI`：

- 需要明确输入输出
- 需要退出码
- 需要产物路径
- 需要被 shell / runner / CI 编排
- 需要从宿主会话中切出一个可执行节点

### 用脚本 runner

当满足下面多数条件时，优先 runner：

- 需要多节点串联
- 需要 state / events / projection
- 需要恢复、回放或审计
- 需要跨会话稳定续跑
- 需要为未来独立 agent/runtime 迁移保留执行面

## `CLI` 什么时候能算生产工具

`CLI` 不是默认生产工具。
它只有在下面条件成立时，才适合承担生产角色：

1. 产物边界清楚
2. 产物需要落文件或稳定 stdout
3. 可验证，不靠纯主观自报成功
4. 有失败退出码或等价失败信号
5. 可以被宿主 workflow、runner 或 shell 稳定调用

满足这些条件时，`CLI` 可以是：

- 节点 producer
- checker
- actuator

这里把词收回到本项目现有语言里：

- `producer`：当前节点直接产出 `execution_object` 或明确交付物
- `checker`：执行校验、validator 或 gate 判断，不新增项目事实
- `actuator`：执行明确动作，把既定命令或流程推进落地，但不改写 control truth

它们只是执行面角色说明，不是新的顶层结构家族。

## `CLI` 什么时候只该做 checker / actuator

如果下面情况更明显，就不要让 `CLI` 直接当 producer：

- 目标还在探索
- 产物主要是讨论或判断
- 没有明确成功条件
- 没有稳定输入输出边界
- 需要频繁人工即时澄清

这时更合理的角色通常是：

- checker：执行校验、lint、validator、diff check
- actuator：执行明确的脚本动作、命令、同步、导出、落盘

## `CLI` 什么时候根本不该用

下面几类任务默认不要刻意上 `CLI`：

- 早期方向讨论
- 用户故事和边界还没稳
- 只是为了上下文隔离
- 宿主原生 `subagent` 已经足够
- 只是想“看起来更工程化”

## 宿主 workflow 与 `CLI` 的嵌套价值

宿主原生 workflow 和 `CLI` 不是二选一。
很常见也很有价值的形态是：

1. 宿主负责 plan / approval / checkpoint
2. `CLI` 负责节点执行
3. 必要时再由 runner 把多个 `CLI` 节点串起来

因此默认判断不是“用宿主还是用 `CLI`”，而是：

**宿主负责什么，`CLI` 补什么。**

## 迁移价值

如果项目未来可能迁移到独立于 `Codex`、`AntiGravity`、`Claude Code` 的 agent/runtime，
`CLI` 执行面会特别有价值，因为它更容易被：

- 外部脚本调用
- CI/cron 调用
- 独立 agent runtime 接管

但如果项目长期就只在一个宿主里运行，且宿主原生能力已经足够，
不要为了迁移假设过早上重实现。

## 一句话收口

**没有编排需求时，不要为了 CLI 而 CLI；一旦需要可拼接执行面，CLI 就应该被设计成稳定节点。**
