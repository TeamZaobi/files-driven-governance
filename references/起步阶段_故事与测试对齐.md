# 起步阶段：先用故事和测试把边界说清楚

当项目还早、方向还在变，或者一次小小的边界改动就会引发后续大量结构变更时，先读这个文件。
这份 reference 只负责先把边界锚住，
不替代后续的治理模式选择、结构划分或正式输出约定。

## 什么时候启用

出现以下情况时，优先启用：

1. 项目还在早期，首批交付物不够具体
2. 最近几轮沟通已经把范围悄悄带偏
3. 好几种“第一阶段要做什么”听起来都像对的
4. 如果现在定错边界，后面的结构设计会被整体带偏

## 先锚定什么

先做一个“方向与边界锚点”资料包，至少包括：

1. 首批真实使用场景
2. 首批交付物
3. 一到三个核心用户故事
4. 三到八个验收或测试用例，其中至少一个是越界或失败边界
5. 明确非目标
6. 影响质量预期的参考对象
7. 一位验收责任人或一类验收对象

资料包保持短、保持具体、保持能让真实使用者快速判断。
不要让架构词替代范围描述。
如果用户没有直接给出来，就先起草一版，再请对方纠偏。
如果这些内容要落进受控 asset pack，默认把它们收成 pack 根目录的一份 `BOUNDARY.md`。

需要可直接照着走的问题集和模板时，直接用 [说人话需求确认工具包](说人话需求确认工具包.md)。

## 提问策略

默认先问四到六个短问题。
优先顺序如下：

1. 谁会在什么场景下真正使用
2. 这次第一阶段到底交付什么
3. 什么算成功
4. 什么这次明确不做
5. 哪些输入或资料必须先支持
6. 谁需要提前对齐

提问要求：

- 问题要具体、指向结果
- 说用户会自然回答的话，不要说内部设计黑话
- 问完以后，先起草故事和测试，不要直接跳到结构设计

## 好问题长什么样

推荐问法：

- “第一批真正会用这个的人是谁？他们会在什么场景下打开它？”
- “这次第一阶段到底交付什么？尽量只说一件主要交付物。”
- “什么变化出现后，你会觉得这次真的做对了？”
- “有哪些东西虽然相关，但这次明确不做？”

不推荐问法：

- “你的最终系统架构是什么？”
- “打算采用什么 workflow 引擎？”
- “你想要哪种 agent topology？”

## 好的验收锚点长什么样

好的锚点至少能回答：

1. 谁在什么情境下使用
2. 期望得到什么结果
3. 什么才算通过
4. 什么虽然相关，但这次明确不交付

如果一条锚点只能说明“希望更好”，却说不清什么算通过，它就还不够好。

## 故事和测试的细化规则

故事和测试至少要细到另一个人或另一个代理能判断：

1. 这次范围是什么
2. 通过线在哪里
3. 越界线在哪里

如果一条故事会自动带出多条彼此独立的交付物，就应该拆分。
如果测试只能验证“理想路径”，还要继续补边界和失败路径。

## 常见漂移信号

看到以下情况，说明边界还没有稳：

- 每次沟通都会多出一个“顺手也做”的能力
- 参考对象从“质量校准”变成“默认抄范围”
- 用户故事一直停在标题级别
- 测试用例只会写主路径
- 谁来验收始终说不清

## 什么时候才能继续放大到治理设计

在满足以下条件前，不要把对话升级成结构治理、流程设计或多工具适配：

1. 有一段简短边界描述
2. 有一到三个用户故事
3. 有三到八个测试用例
4. 至少有一个非目标
5. 有一位验收责任人或一类验收对象

只要这些条件还没满足，优先继续补边界，而不是扩治理蓝图。
