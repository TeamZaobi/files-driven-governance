# 场景手册

先判断项目属于哪种起点，再用对应打法。
但在三种起点里，都先做同一件事：

1. 判当前动作是什么
2. 判它先落在 `入口规则`、`能力规则`、`项目规则` 还是 `项目实体`
3. 判它需要一级关口还是二级升级

如果对方是第一次接触项目，先把三种起点翻成：
`现有项目 / 新项目 / 止血收口`，
不要先把 `existing_repo / greenfield / recovery_or_realignment` 当成首句。

本文件只在起点已经判成 `existing_repo`、`greenfield` 或 `recovery_or_realignment` 后读取。
它负责给出最低检查范围、核心最低交付物和常见起手流程，不替代主 `SKILL`、[输出约定](输出约定.md) 或 [经典治理流程库](经典治理流程库.md)。

## 现有仓库

### 最低检查范围

至少检查：

1. 当前首批交付物，以及最近几轮沟通是否已经把它悄悄带偏
2. 主 README 或等效入口
3. 当前状态页或起始入口
4. 活跃的任务、讨论和复核载体
5. 任何版本定位点、注册表或规格路径
6. 任何可能越权的网站、仪表盘或说明页
7. 任何定义短口令或操作口径的工具启动文档

### 核心最低交付物

至少给出：

- 当前可信真源与四层文档分类
- 主要漂移点
- 需要先降级的摘要页、展示页或工具入口
- 推荐治理模式
- 推荐最小流程集合
- 简短迁移顺序

### 按条件补充

- 如果边界仍在变化，再补方向与边界锚点
- 共享关系已经影响同步时，再补跨层共享关系
- 多角色碰同一事实或审批边界不清时，再补角色回路草图
- 如果项目依赖短口令，再补意图触发约定
- 如果文档过厚，再补压缩顺序

### 常见失效模式

- 多个文件都在定义同一条边界
- 状态页写了上游真源没有写的事实
- README 之所以最新，只是因为大家习惯先看它
- 讨论记录同时承担任务和决策
- 没有可靠交接链
- 历史材料仍躺在活跃页面里
- 最近几轮澄清悄悄扩大了首批交付物
- 同一句短口令在不同工具里含义不一样

### 优先第一步

如果边界还在漂移，先临时冻结故事和测试锚点。
然后先回答：

1. 这次高频动作是什么
2. 它先归 `Skill` 还是归项目实体
3. 哪个入口规则必须本轮直接生效

再画出真源地图，把不可信的摘要页先降级，再谈目录和命名。

### 常见流程起手式

通常先启用：

- `low_token_recovery_chain`
- `discussion -> decision_package -> task_or_decision`
- `truth_source -> execution_object -> status_projection -> display_projection`
- 动到治理对象且会改结构边界时，再补 `mechanism_review`

有条件再加：

- 高争议话题才加 `adversarial_inquiry -> defense -> convergence`
- 治理面或公开面变更才加 `proposal -> validation -> shadow/canary -> activation_or_rollback`
- 文档膨胀明显时，加 `growth_signal -> lifecycle_review -> compact_or_archive`

## 新项目

### 最低设计范围

只定义当前阶段真需要的内容：

1. 首批交付物的方向与边界锚点
2. 慢变量对应的最小真源集合
3. 最小结构家族集合
4. 少量过程载体
5. 一个恢复入口
6. 一条跨层变更的版本与同步规则
7. 如果要用短口令，再补最小意图集合

### 核心最低交付物

至少给出：

- 方向与边界锚点
- 推荐治理模式
- 最小可行文档集合
- 最小结构家族集合
- 第一条入口与恢复链
- 第一组默认流程

### 按条件补充

- 多人、多代理或多工具会共享同一事实时，再给第一版跨层共享约定
- 多角色边界从第一天就重要时，再补第一条角色回路
- 如果要用短口令，再给第一版意图触发约定
- 风险、自主度或回退成本偏高时，再补第一条复核或升级规则

### 常见失效模式

- 直接照搬成熟项目的目录树
- 澄清几轮后首批交付物越变越大，却没回写边界
- 在还没有实际工作前就发明太多对象类型
- 把战略、任务和状态混在一个文件里
- 风险还没到，却先上重审批
- 短口令越来越多，但主意图没定稳

### 优先第一步

先锁定一到三个用户故事和少量验收样例。
再定义：

1. 最小入口规则
2. `Skill` 主容器里第一批必须稳定的能力规则
3. 第一批项目规则和项目实体

最后才补最小真源集合，以及一条清楚的工作与恢复路径。

### 常见流程起手式

通常先启用：

- `low_token_recovery_chain`
- `discussion -> decision_package -> task_or_decision`
- `truth_source -> execution_object -> status_projection -> display_projection`

通常先延后：

- 每个话题都上敌意质询
- 低风险工作也走完整验证阶梯
- 没有直接升级证据就开多角色隔离讨论
- 文档还没长起来就设计复杂归档规则

## 恢复与重整

### 最低止血范围

至少判断：

1. 哪些文档还值得信任
2. 哪些文档应降级成摘要或历史
3. 哪些活跃过程载体需要重新指定责任人
4. 哪些入口正在干扰交接
5. 哪些结构家族目前混在一起
6. 上一次可信的交付物边界停在哪
7. 哪些短口令还有稳定含义，哪些已经失真

### 核心最低交付物

至少给出：

- 下一阶段的恢复后边界锚点
- 立即止血顺序
- 当前可信事实与真源重置方案
- 短期状态恢复链
- 最小修复后的流程集合

### 按条件补充

- 多角色仍会碰同一事实时，再补最小可用共享约定
- 如果项目依赖短口令，再修意图触发约定
- 文档膨胀明显时，再补第一轮文档降级和压缩顺序
- 止血后仍需持续收口时，再补长期重整方向

### 常见失效模式

- 还没选定可信事实就继续加新文件
- 先改目录名，再补责任边界
- 想把每一条历史信号都保留成当前事实
- 上游还没修好，就急着重做网站和仪表盘
- 因为以后“可能还有用”，就把历史页面继续留在活跃层
- 澄清里提过一次的相邻能力，全都继续留在当前范围
- 在流程和状态都还没稳住前，就急着统一短口令

### 优先第一步

先停掉新的结构扰动，选定当前可信事实，重锚下一阶段交付物。
然后优先回答：

1. 当前哪些能力规则还能继续信任
2. 哪些项目实体已经越权或混写
3. 哪个一级关口要先恢复

最后再恢复最短可用交接链。

### 常见流程起手式

先启用：

- `low_token_recovery_chain`
- `truth_source -> execution_object -> status_projection -> display_projection`
- `mechanism_review`

真源重新可信后，再考虑恢复：

- `adversarial_inquiry -> defense -> convergence`
- `proposal -> validation -> shadow/canary -> activation_or_rollback`
- `skill_seed -> package_contract -> active_package`

文档膨胀明显时，止血阶段通常同时启用：

- `growth_signal -> lifecycle_review -> compact_or_archive`
