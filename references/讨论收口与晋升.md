# 讨论收口与晋升

这份 reference 只回答一件事：一段 `discussion` 什么时候该开、该留、该升、该收。
它面向直接执行的人，不面向内部术语说明。

## 这是什么

`discussion` 是临时协商容器，不是任务桶，也不是长期记录仓。

只要还有争议点、证据缺口、归属未定或出口未清，就用 `discussion`。
一旦只剩结论、执行或存档，就不要继续把它叫 `discussion`。

## 什么时候开 discussion

也就是：什么时候开正式 `discussion`。

在下面这些场景开：

1. 问题还没说清，但又不能直接下 `decision`
2. 争议点还没分开，或者不同人对事实、口径、边界的理解不一致
3. 还没有稳定 owner，或者 owner 需要先看清选项和代价
4. 这件事可能会升级成 `issue_ledger`、`decision_package`、`task`、`decision` 或 `archive`，但现在还不确定落点

不要在下面这些场景硬开 `discussion`：

- 已经有明确结论，只差执行
- 只是重复已经确认过的信息
- 只是想把长期任务继续挂在聊天里

## 最低要留什么

至少留下五件事：

1. 议题一句话
2. 当前倾向或临时结论
3. 还没解决的关键问题
4. 需要补的证据或事实
5. 下一步去哪个出口

更稳一点的最小包，还要加：

- 负责人或裁定人
- 相关链接或来源锚点
- 时间点或版本锚点

如果连这些都留不住，就先补记录，再谈讨论。

## 何时升级到质询

把 `discussion` 升成质询，不是因为“聊得多”，而是因为：

1. 争议会直接影响任务、决策、规则或风险边界
2. 同一个问题已经反复出现，但没有被逐点回答
3. 需要把反对点、答辩和裁定写清楚
4. 风险太高，礼貌讨论不足以暴露问题

升级后要把内容改成：

- 主张是什么
- 反对点是什么
- 哪些问题已回答，哪些没回答
- 最终是通过、延后、打回还是拆题

## 何时压成 decision_package

也就是：什么时候压成 `decision package`。

当目标已经不是“继续讨论”，而是“把一个可裁定的问题压给决策人”时，就收口成 `decision_package`。

适合压包的信号：

1. 事实面基本稳定
2. 备选项已经排过
3. 需要的是清晰的 decision surface，不是继续拉长聊天
4. 后续只需在一个地方做取舍

`decision_package` 里最少要有：

- 推荐方案
- 不选方案
- 关键依据
- 残余风险
- 需要谁来定

## 何时压成 task

当 choice 已经定了，剩下的是执行，就压成 `task`。

`task` 里至少保留：

- 任务名
- owner
- 完成标准
- 截止时间或触发条件
- 依赖与阻塞

判断标准很简单：

- 如果需要的是“做什么、谁做、什么时候算完”，那是 `task`
- 如果需要的是“选哪个、为什么选”，那还是 `decision_package`

## 何时压成 decision

当要保留的核心不是动作，而是裁定本身，就压成 `decision`。

`decision` 里至少保留：

- 被选中的选项
- 为什么选它
- 还接受了什么风险
- 这条 `decision` 影响哪些后续对象

如果执行已经另有 `task` 承接，不要再把 `decision` 写成任务清单。

## 何时归档

归档不是删掉，而是把它从当前活跃链挪走。

满足下面任意一条，通常就可以 `archive`：

1. 当前已完成，后续不会再改写事实
2. `decision` 或 `task` 已经接住后续动作
3. 只需要历史恢复，不再需要日常读取
4. 当前页面会继续误导读者，把它当成活跃项

归档前要先确认：

- 当前真源在哪里
- 当前恢复链还能不能找到这件事
- 归档页里是否保留了一句话总结和去向

## 最短收口顺序

通常按这个顺序推进：

`discussion -> issue_ledger -> decision_package -> task_or_decision -> archive`

其中：

- `discussion` 管澄清和分叉
- `issue_ledger` 管问题列表和缺口
- `decision_package` 管决策面压缩
- `task_or_decision` 管最终落点
- `archive` 管历史保留

## 一条硬规则

如果 `discussion` 已经稳定成“只剩执行”，就不要继续把它扩写成新讨论。
如果它已经稳定成“只剩裁定”，就不要继续把它扩写成争议记录。
