# 理解型输入与低带宽压缩包

这个文件回答一个实际协作问题：

当用户说“讲解一下”“再检查一下”“我还是担心”“为什么会这样”时，
到底是在补新信息，
还是在试图建立一个可操作的理解模型。

这份 reference 只在已经出现解释型摩擦或低带宽推进信号时启用，
不替代正常诊断、正常澄清或正式输出约定。

结论不是“很多提问都没有价值”。
更稳的结论是：

1. 确实存在一类 `understanding` 型回合
2. 这类回合常由 `display_projection` 或 `status_projection` 不适合人的认知带宽触发
3. 但很多看起来像解释性追问的输入，实际在改写 `constraint / evidence / decision`

所以处理顺序必须先分型，再决定是压缩解释，还是改写问题定义。

## 一、四类输入

### 1. `understanding`

特点：

- 主要目标是建立当前局面的可操作理解
- 常见表达是“讲解一下”“这个是什么意思”“为什么会这样”
- 往往不直接引入新事实，但可能要求更合适的输出形态

### 2. `constraint`

特点：

- 在改范围、边界、禁区、制度口径或验收条件
- 常见表达是“我更倾向于……”“不要这样写”“这里不该……”
- 这类输入不是解释摩擦，而是治理边界改写

### 3. `evidence`

特点：

- 提供新线索、指出 artifact defect、补关键来源或校正引用
- 常见表达是“这里写错了”“这个路径不对”“这个材料要回看”

### 4. `decision`

特点：

- 直接表达 owner 判断、取舍或否决
- 常见表达是“就按这个走”“这个不接受”“先停在这里”

## 二、先做反例检查

不要因为一句话听起来像解释型追问，
就直接把它归成 `understanding`。

至少先检查它是否在改写下面这些东西：

1. `scope boundary`
2. `approval anchor`
3. `owner decision`
4. `entry contract`
5. `evidence gate`

如果答案是“是”，
就优先按 `constraint / evidence / decision` 处理，
而不是给一大段解释。

## 三、标准处理顺序

推荐固定成这个顺序：

`input_classify -> counterexample_check -> low_bandwidth_package_or_reframe`

含义：

1. 先判断用户输入属于哪一类
2. 再检查它是否表面像理解、实际在改边界
3. 只有确认是 `understanding` 时，才优先补低带宽压缩包
4. 如果不是，就明确说出它改写了什么，并切换到新的问题定义

## 四、四种低带宽压缩包

### 1. `anchor package`

适用：

- 用户在问“现在到底在说哪个对象、哪个版本、哪个主线”
- 常见缺口是 `anchor_missing`

最少包含：

1. 当前锚点是什么
2. 当前可信真源在哪里
3. 这轮判断建立在哪个对象上
4. 现在不需要读哪些材料

### 2. `decision package`

适用：

- 用户在问“为什么这么做”“为什么不是另一条路”
- 常见缺口是 `decision_surface_missing`

最少包含：

1. 当前推荐路径
2. 不走的替代路径
3. 当前判断依据
4. 现在真正需要用户决定的只有什么

### 3. `evidence package`

适用：

- 用户在问“依据是什么”“这件事怎么证明”
- 常见缺口是 `evidence_surface_missing`

最少包含：

1. 关键结论
2. 对应证据或文件锚点
3. 还没补齐的证据缺口
4. 如果继续验证，下一步看什么

### 4. `route package`

适用：

- 用户在问“接下来怎么走”“为什么卡在这里”
- 常见缺口是 `route_missing`

最少包含：

1. 当前阶段
2. 下一到三步
3. 每一步的产物
4. 当前阻塞点或停点

## 五、给非工程背景读者的最小压缩形状

如果对象几乎没有软件工程基础，
不要先把完整术语表、完整模型或所有 reference 一次性丢出去。

默认先压成 4 句：

1. 现在哪份文件算数
2. 你现在先做哪一步
3. 哪些文件现在先别改
4. 卡住时先回看哪个入口

这 4 句已经足够让多数解释型摩擦先降下来。

## 六、何时优先启用

下面这些场景优先用这套规则：

1. 同类解释型问题重复出现
2. 用户使用低带宽短口令推进工作
3. agent 掌握的信息明显比用户多很多
4. 已经出现“我还是没明白”“再讲清楚一点”这类摩擦信号

如果当前输入已经明显在改 `constraint / evidence / decision`，就不要回到这套压缩包逻辑。

## 七、不要做的事

1. 不要把“我还是担心”“再检查一下”自动当成纯理解同步
2. 不要在没做反例检查前，就判断用户没有提供有效增量
3. 不要把 `display_projection` 只当成展示美化层
4. 不要一上来转储整份真源；先给可操作的压缩包
5. 不要在本该重定问题时继续顺着旧问题解释
