# 基本原则

这个技能默认按三个视角依次判断：系统视角、信息视角、控制视角。
它们不是抽象哲学，而是为了帮助你做出实际的治理选择。
这份 reference 只负责解释判断基点，不替代场景路由、治理模式选择或输出清单。

## 系统视角

先问这些问题：

1. 项目边界是什么
2. 哪些层是慢变量，哪些层变化快
3. 哪些角色、代理、工具或流程会碰同一组事实
4. 哪些文档在定义系统，哪些文档只是在描述系统
5. 哪些耦合一旦写错，就会把错误传播出去

在谈目录前，先把结构家族分清：
正式分类与定位仍以主 `SKILL` 和 [结构家族定位约定](结构家族定位约定.md) 为准，这里只解释为什么要先分清。

1. 规则与约束层（`policy_or_rules`）
2. 对象层（`object`）
3. 流程层（`workflow`）
4. 技能层（`skill`）
5. 角色层（`agent`）
6. 过程载体层（`execution_object`）
7. 状态摘要层（`status_projection`）
8. 展示输出层（`display_projection`）

系统视角下的设计原则：

- 慢变量放进稳定真源
- 高频变化放进过程载体
- 恢复入口和导航层不要混成真源
- 不要让一个文件同时承担太多角色
- 先分清结构家族，再谈目录细化
- 先让人看清谁定义事实，谁只是消费或展示事实
- `agent` 是角色职责与控制边界
- `skill` 是可复用方法与步骤包

## 信息视角

接着问这些问题：

1. 每个重要事实从哪里产生
2. 哪些文件在复制、总结或展示它
3. 现在哪些地方已经出现重复、滞后或漂移
4. 一个新接手的人最短恢复路径是什么
5. 哪些页面看起来权威，其实不是真源

信息视角下的设计原则：

- 一个事实优先只有一个明确真源
- 优先用引用，不要靠复制粘贴同步
- 优先提供当前版本锚点，不要让人“全仓库通读”
- 优先提供短恢复链，而不是依赖完整历史回放
- 让结构关系可读：规则指向哪条流程，流程依赖哪些对象，技能用了哪些参考，状态页在总结哪些上游事实
- 如果对项目基本情况的理解把握度还低，先补问题，再冻结诊断

## 控制视角

最后问这些问题：

1. 项目怎么观察当前状态
2. 谁能决策，谁能执行，谁负责复核
3. 什么事件会触发升级、版本变更、回退或重分类
4. 项目怎么发现自己已经失真
5. 项目怎么在不停工的前提下恢复秩序

至少画出一条明确回路：

1. `observe`
2. `decide`
3. `act`
4. `review`
5. `rollback_or_improve`

控制视角下的设计原则：

- 默认用轻量关口
- 只有在自主度、风险或回退成本上升时，才增加正式复核
- 状态摘要层尽量从上游对象推导出来
- 多个角色都能碰同一事实时，要写清责任边界和更新顺序
- 为每个角色明确文档责任：谁管规则，谁管流程，谁管过程载体，谁管摘要页，谁能授权回退或重分类

## 实际落地时怎么映射

- 系统视角负责边界、结构家族、责任面和文档分层
- 信息视角负责真源、引用链、同步顺序、版本锚点和恢复入口
- 控制视角负责角色回路、升级规则、复核节奏、回退路径和例外处理

需要按起点排顺序时，读 [场景手册](场景手册.md)。
需要选择治理力度和组合时，读 [治理模式选择对照表](治理模式选择对照表.md)。
需要组织正式回答时，读 [输出约定](输出约定.md)。

如果仓库还很早期：

- 模型保持小
- 入口保持少
- 每个大主题只给一条明确真源路径

如果仓库已经漂移：

- 暂停继续加结构层
- 先挑出当前可信事实
- 把陈旧页面降级成摘要或历史
- 先恢复责任边界，再重构分类

如果仓库正在扩张：

- 按功能拆分文档
- 增加责任与同步纪律
- 让摘要层保持轻
- 让结构家族边界足够清楚，不让角色和工具互相改义

## 红旗信号

出现以下情况，通常说明治理已经开始失效：

- 状态页说得比规格和决策还像真源
- README 默默替代了正式真源
- 同一条规则被复制到多个目录
- 当前工作只活在聊天记录里
- 每次出问题都靠再加一个顶层目录解决
- 没人说得清楚该先读什么
- 规则、角色、流程和技能互相改写，没有稳定责任模型
