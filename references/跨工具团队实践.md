# 跨工具团队实践

当用户问“多工具团队该怎么协作”时，使用这个参考件。
这份 reference 只在问题重心确实是多工具协作、并行分工、证据入口或工具切换成本时启用，
不替代项目自身的结构划分、流程定义或输出约定。

目标不是复制某个工具的命令习惯，而是提炼跨工具仍然成立的做法。

## 值得保留的通用做法

### 1. 并行工作时分上下文

如果团队经常同时推进多件独立事情，就用分离的工作上下文：

- worktree
- 独立检出目录
- 独立工作区
- 独立代理会话

核心原则：

- 并行工作要隔离
- 减少上下文污染
- 需要时保留一个专门读日志和指标的上下文

### 2. 复杂工作先做计划

复杂变更默认先计划，再实施。
如果中途方向变化，再重新计划。

核心原则：

- 计划不是额外负担
- 复杂验证本身也可能需要计划

### 3. 把纠错经验写进长期文档

每次重复错误或重要修正出现后，不要只靠记忆。
把它写进项目的长期文档、规则或技能包。

核心原则：

- 纠错知识应写一次、复用多次
- 衡量标准是错误率下降，不是文档数增加

### 4. 把重复工作打包

同一做法重复出现时，优先打包成：

- 技能
- 命令
- 清单
- 受控脚本

核心原则：

- 重复行为应该从聊天习惯升级成项目资产

### 5. 让代理从证据出发

排错和定位问题时，优先从这些证据出发：

- CI 失败
- 日志
- 议题记录
- 指标
- 复现步骤

核心原则：

- 让代理扎在可观察事实里
- 少做无依据的上下文切换

### 6. 让代理承担质询和解释

代理不只用来执行，也可用来：

- 质询方案
- 比较版本差异
- 解释为什么这样改
- 产出学习材料

### 7. 把工作环境做得容易切换

不管是终端还是 IDE，关键都不是品牌，而是：

- 当前上下文清楚
- 分支和工作区可见
- 切换任务成本低

### 8. 工具支持时，用次级代理处理边任务

如果工具支持子代理或并行工作者，就把边界清楚的小任务分出去。
如果不支持，就用独立会话或工作区代替。

### 9. 数据和指标入口要可控

日志、指标和分析数据优先走这些入口：

- CLI
- API
- MCP 类连接器
- 受控技能包

核心原则：

- 数据访问要可复用、可审计
- 不要依赖临时手抄

### 10. 学习产物也是项目资产

图示、说明稿、原理解释和经验沉淀，都属于项目韧性的一部分。

## 代理参与团队流程时要怎么调

### 推荐责任面

可以先用这三类责任面思考，而不是生硬复制名字：

- 规格责任人
- 代理治理责任人
- 质量关口责任人

### 推荐最小流转状态

一个通用的最小流转可以是：

1. `Backlog`
2. `Spec Ready`
3. `Agent Run`
4. `Evidence Pending`
5. `Human Review`
6. `Merge or Release`
7. `Observe or Feedback`

这是跨工具团队的通用示例，不是每个项目都要直接照搬的默认流程。

### 在制品上限规则

上限按“人能复核多少”来定，不按“代理能产出多少”来定。

### 证据规则

不要把“代码写完了”当成完成。
更好的关口是“证据已经齐了，可以复核了”。

### 工具通用性规则

项目治理不要绑在某个工具的界面习惯上。
哪怕工具切换，以下内容也应保留下来：

- 角色定义
- 真源
- 复核关口
- 证据要求
- 交接包

## 面向代理团队的写作建议

1. 交接前先消歧义
2. 规格要能对照证据复核
3. 稳定约束和任务临时记录分开写
4. 优先写短、准、能操作的话
5. 写给“下一位接手者”看，而不是写给当前自己看

## 反模式

避免以下做法：

1. 把工具偏好当成治理原则
2. 把某个产品的口令体系强行升格为团队正式流程
3. 把关键知识只留在聊天习惯里
4. 允许并行产出，却不设计并行复核
5. 把每个提效技巧都当成通用规则
