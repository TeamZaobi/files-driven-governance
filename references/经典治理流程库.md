# 经典治理流程库

这个文件用来判断：哪些流程值得变成默认做法，哪些只应在特定条件下启用。
这份 reference 只在治理力度和项目起点已经大致判清后使用，
不替代场景路由、治理模式选择或正式输出约定。

## 怎么用

不要把整份流程库原样塞给每个项目。
先根据项目阶段、风险、协作密度和恢复压力，选一个最小流程集合。
如果只是个别争议需要反方视角，优先下钻 [反方质询与收敛回路](反方质询与收敛回路.md)；
如果是在问多工具团队如何协作，再下钻 [跨工具团队实践](跨工具团队实践.md)。

## 值得优先保留的默认流程

### 1. 轻量恢复链

`low_token_recovery_chain`

用途：

- 让新接手的人快速找到当前目标、当前状态和下一步
- 降低必须通读全历史的成本

### 2. 讨论 -> 决策包 -> 任务或决策

`discussion -> decision_package -> task_or_decision`

用途：

- 不让讨论记录长期兼任任务桶
- 让需要升级的议题进入更清楚的容器

### 3. 真源 -> 过程载体 -> 状态摘要 -> 展示输出

`truth_source -> execution_object -> status_projection -> display_projection`

用途：

- 固定信息流方向
- 防止状态页和展示页反向改写真源
- 多工具过程不透明时，可先从 `execution_object` 派生 `process_projection`，
  再把它落到 `status_projection` 或 `display_projection`，
  用统一语义说明“这次到底跑了什么”
- 如果复杂性主要来自显式 `workflow` 或多 agent 协作，
  再条件补一个 `topology_supplement`，
  只解释节点、流转、分支、交接和收敛

补充约束：

- `process_projection` 只压缩过程，不定义事实
- 它不替代原始 `walkthrough / trace / decision / handoff`
- 不同工具的投影冲突时，回到共同上游过程载体裁决
- `topology_supplement` 只补执行拓扑，不替代过程摘要本身

### 4. 机制复核 -> 修补或拆分

`mechanism_review -> repair_or_split`

用途：

- 每次碰治理对象时，顺手检查是否需要补规则、拆层或修共享约定

## 条件触发的升级流程

### 5. 反方质询 -> 答辩 -> 收敛

`adversarial_inquiry -> defense -> convergence`

适用：

- 争议大
- 题目重要
- 需要制度化反方视角

### 6. 隔离式多角色讨论

`isolated_multi_role_deliberation`

适用：

- 多角色意见需要分开展开
- 直接混聊会互相污染判断

### 7. 提案 -> 验证 -> 影子/灰度 -> 激活或回退

`proposal -> validation -> shadow/canary -> activation_or_rollback`

适用：

- 治理规则、公共表面或高风险改动
- 回退代价高的改动

### 8. 技能种子 -> 包约定 -> 活跃包

`skill_seed -> package_contract -> active_package`

适用：

- 某项做法已经重复出现
- 需要从聊天习惯升级成可复用技能

### 9. 约定缺口 -> 闭环议题 -> 下游恢复

`contract_gap -> closure_topic -> downstream_resume`

适用：

- 下游工作暴露出上游约定缺口
- 不想让临时补丁长期留在下游

### 10. 膨胀信号 -> 生命周期复核 -> 压缩或归档

`growth_signal -> lifecycle_review -> compact_or_archive`

适用：

- 活跃文档变厚
- 陈旧页面仍在冒充当前页面
- 恢复成本明显上升

### 11. 输入分型 -> 反例检查 -> 低带宽压缩包或重定问题

`input_classify -> counterexample_check -> low_bandwidth_package_or_reframe`

适用：

- 对话里反复出现“讲解一下”“再检查一下”“我还是担心”
- 用户使用低带宽短口令推进工作
- agent 明显掌握比用户更多的上下文

用途：

- 先判断输入是在补理解，还是在改边界
- 避免把真实 `constraint / evidence / decision` 误判成低价值解释性追问
- 对纯理解同步，优先补 `anchor / decision / evidence / route package`

### 12. 运营信号 -> 分类 -> 状态刷新 / 局部修复 / 开发变更 / 回退

`ops_signal -> classify -> state_refresh_or_local_fix_or_dev_change_or_rollback`

适用：

- 项目同时持续开发和持续运行
- 运行事件、人工反馈或观察信号会持续进入当前协作面
- 团队需要明确哪些信号只改状态，哪些必须升级

用途：

- 防止运营信号直接改写开发真源
- 把状态刷新、局部修复、开发变更和回退分成不同出口
- 让开发面和运营面通过清楚接口衔接，而不是混成一条队列

### 13. 运行观察 -> 机制复核 -> 候选保留或吸收

`observation -> mechanism_review -> candidate_or_absorb`

适用：

- 运行中反复出现同类问题、补救动作或观察结论
- 团队在讨论“这次经验要不要变成长期资产”
- 需要阻止“每次做完都自动升级成 skill”

用途：

- 先把运行观察和正式技能真源分开
- 只让重复、稳定、验证过的做法被吸收到长期资产
- 让候选保留成为正式步骤，而不是口头默契

## 场景化选择建议

### 小型或早期项目

通常先保留：

- `low_token_recovery_chain`
- `discussion -> decision_package -> task_or_decision`
- `truth_source -> execution_object -> status_projection -> display_projection`

### 已有仓库且开始漂移

通常先保留：

- `low_token_recovery_chain`
- `truth_source -> execution_object -> status_projection -> display_projection`
- `mechanism_review -> repair_or_split`

必要时再加：

- `growth_signal -> lifecycle_review -> compact_or_archive`
- `adversarial_inquiry -> defense -> convergence`

### 治理强度较高或多代理项目

通常会比默认集合多启用：

- `proposal -> validation -> shadow/canary -> activation_or_rollback`
- `isolated_multi_role_deliberation`
- `skill_seed -> package_contract -> active_package`
- `input_classify -> counterexample_check -> low_bandwidth_package_or_reframe`

### 开发/运营并存项目

通常优先判断：

- `ops_signal -> classify -> state_refresh_or_local_fix_or_dev_change_or_rollback`
- `observation -> mechanism_review -> candidate_or_absorb`
- `truth_source -> execution_object -> status_projection -> display_projection`

## 不要照搬的地方

1. 不要因为流程看起来完整，就默认全开
2. 不要把重流程当成熟标志
3. 不要让流程名字代替真实判断
4. 不要让项目背负超出当前需要的治理负担
