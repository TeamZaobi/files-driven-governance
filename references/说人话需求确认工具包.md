# 说人话需求确认工具包

当项目还在早期、边界还在漂移，或者用户需要一套能直接照着走的确认流程时，用这个参考件。
这份 reference 只在需要直接提问、起草故事和测试模板时启用，
不要求每个项目都完整走一遍工具包，也不替代后续治理设计和输出约定。

目标很简单：
在治理设计、架构扩展或流程细化之前，先把“这次第一阶段到底交付什么”用日常语言说清楚。

## 工具包包含什么

1. 默认问题集
2. 默认用户故事模板
3. 默认测试用例模板
4. 最小完成规则
5. 相关方预对齐检查

## 默认使用顺序

除非用户已经给出稳定边界，否则按这个顺序来：

1. 先问四到六个说人话的问题
2. 用一小段话重述首批交付物
3. 起草一到三个用户故事
4. 起草三到八个测试用例
5. 明确写出非目标
6. 指出谁需要在更大范围规划前先对齐

如果前两三步已经足够锁定边界，就不要为了“走完整流程”继续机械补齐后面所有动作。

不要从提问直接跳到架构。
不要只凭一个模糊故事就直接扩成治理方案。

## 默认问题集

提问时用日常语言，不要让用户先翻译产品黑话或架构黑话：

1. “第一批真正会用这个的人是谁？他们会在什么场景下打开它？”
2. “这次第一阶段到底交付什么？尽量只说一件主要交付物。”
3. “他们现在最卡的地方是什么？如果这次不做，会继续损失什么？”
4. “这次做好以后，你最想看到什么具体变化？什么才算真的有用？”
5. “有哪些东西虽然相关，但这次明确先不做？”
6. “有没有谁的意见会直接影响方向？谁需要先提前对齐？”

如果还不够清楚，再补这些追问：

1. “首批必须支持哪些输入或材料？哪些可以晚一点？”
2. “有没有一个参考作品只是帮我判断质量，而不是把范围一起带进来？”
3. “如果只能保留一个最重要的结果，你宁愿保留哪一个？”

## 用户故事模板

先用中文白话写故事。
除非用户自己先引入，否则不要一上来写实现词。

```markdown
### 用户故事 [编号]

**谁在用：**
[具体到一个真实角色，不要写“所有用户”]

**在什么场景下：**
[描述真实触发时刻]

**他/她现在想完成什么：**
[一个清楚动作或结果]

**为什么这件事对当前阶段重要：**
[说明当前阶段的核心价值]

**这次完成后，用户应该看到什么变化：**
[一个具体、可观察的变化]

**这次明确不包含什么：**
[避免把相邻能力一起带进来]
```

### 故事质量检查

只要出现以下任一情况，这个故事就还不够稳：

1. 只写了解法，没有写真实场景
2. 角色写成“用户”或“团队”这种过宽描述
3. 只说“更方便”“更智能”，没有说具体变化
4. 说不清当前明确不做什么
5. 一条故事同时带出多个首批交付物

## 测试用例模板

每个故事都要配能区分“成功、失败、越界”的测试用例。

```markdown
### 测试用例 [编号]

**对应故事：**
[故事编号]

**前提：**
[用户状态、已有输入、当前环境]

**当：**
[触发动作]

**那么：**
[交付物或系统应该表现出的结果]

**通过条件：**
[怎么判断真的通过]

**这次明确不要求：**
[虽然相关，但这次不算缺失的内容]

**失败/越界边界：**
[哪些结果出现就说明偏题、越界或失败]
```

### 测试最少组合

每个故事至少包含：

1. 一条主路径测试
2. 一条边界或错误路径测试
3. 一条越界或非目标测试

如果首批交付物还很模糊，就继续补测试，不要急着补结构设计。

## 最小完成规则

在满足以下条件前，不要把边界当成已经锁定：

1. 有一段简短的首批交付物描述
2. 有一到三个用户故事
3. 至少有三条测试用例
4. 至少有一个明确非目标
5. 有一位验收责任人或一类验收对象

## 相关方预对齐检查

在把边界锚点扩成治理方案前，先确认：

1. 谁能批准或改向首批交付物
2. 谁会判断结果够不够好
3. 谁最可能提出反对或扩大范围
4. 他们需要明确接受什么取舍

这些还不清楚时，不要把边界当成已定。
