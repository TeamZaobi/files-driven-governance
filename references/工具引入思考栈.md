# 工具引入思考栈

当用户提到“引入 / 接入 / 安装 / 换 / 用 / 整合 / 配置 / 上某个工具”时，用这份 reference。

这里的“工具”包括宿主可发现的任意 skill、plugin、connector、app、MCP、CLI、workflow runtime、
knowledge workspace 或外部 sidecar，不只包括某几个新装能力。

这里的目标不是先安装工具，而是先判断：

1. 这个工具到底要解决什么问题
2. 官方手册、社区实践和已安装实例里是否已经有成熟解法
3. 它会不会改变项目真源、写权、流程、状态、恢复或审计
4. 它应该作为原生能力、被动缓存、薄 adapter、命令节点、runtime，还是项目本地 workflow 进入系统

如果只是一次性执行当前任务，而且宿主原生能力已经足够，默认不升级成工具引入。
如果当前已经绕了两轮、反复试错，或方案明显依赖真实工具机制，默认先查官方手册和成熟实践，不要继续靠想象设计。

## 触发信号

这些说法默认触发工具引入判断：

- “把 X 接进来”
- “全局安装 X”
- “用 X 调度 / 管理 / 编排”
- “换成 X”
- “接 MCP / API / CLI / plugin / skill / agent / workflow”
- “这个 skill / plugin / connector 应该怎么用、怎么接、怎么组合”
- “浏览器自动化 / 前台操作 / daemon / server / tunnel”
- “让多个工具一起用同一套规则 / 状态 / 入口”
- “网上有没有现成方案 / 官方推荐怎么做 / 社区一般怎么接”

但触发判断不等于触发安装。

## 先问的问题

最小入口只问四件事：

1. `job`：这个工具解决的是缺能力、缺纪律、缺隔离、缺编排、缺数据入口，还是缺展示面？
2. `known practice`：官方手册、真实案例、社区反模式或本机已安装实例有没有更短路径？
3. `authority`：它会不会接触项目事实、写权、流程状态、恢复链或审计链？
4. `posture`：它应该默认启用、按需触发、项目内 opt-in，还是只作为 passive cache？

如果 `job` 说不清，先停在讨论或调研，不要进入 install / register / repair。
如果 `known practice` 为空，而工具又不是纯本地小脚本，先补调研，不要直接造 adapter。

## 手册与实践调研

工具接入前默认先做一轮轻量调研，除非用户明确要求离线、当前工具完全本地可见，或任务只是一次性命令执行。

调研顺序：

1. 官方机制：官方文档、官方源码、官方 CLI help、官方配置示例、release notes
2. 已安装实例：本机或项目里已有配置、命令、workflow、adapter、日志和失败痕迹
3. 社区实践：常见接法、成熟模板、反模式、issue / discussion 里的真实故障
4. 当前推断：把无法确认的部分明确标成推断，不伪装成官方建议

触发升级调研的信号：

- 已经绕路或重复试错
- 工具涉及鉴权、网络、daemon、foreground UI、数据写入、team mode、telemetry
- 工具有成熟生态，手写方案很可能低于现成实践
- 用户要求“手册 / 最佳实践 / 网上方案 / 社区做法”
- 当前要把工具提升成全局能力、项目控制面或可复用 skill

调研不是无限研究。够用标准是：

1. 找到官方推荐入口或确认官方没有这种入口
2. 找到至少一个成熟实践或反模式
3. 知道当前本机 / 项目 live 实例是否已经存在
4. 能把“官方机制 / 社区经验 / live 现状 / 当前推断”分开说

## 工具形态

优先按形态理解工具，不按品牌理解工具。

| 形态 | 典型角色 | 默认处理 |
| --- | --- | --- |
| 宿主原生能力 | 主线程、subagent、approval、workflow | 优先复用，避免重复造控制面 |
| 已发现 skill / plugin / connector | 专用能力入口、方法层、宿主 app 能力 | 任务匹配时直接用；要改安装、触发、同步或分发时交给 `skills-master` |
| `CLI` / 命令包 | checker、actuator、producer 节点 | 需要明确输入、输出、退出码和产物路径 |
| `MCP` / API / server / service | 数据入口、能力入口、远端执行入口 | 先定权限、数据边界、鉴权和失败模式 |
| workflow runtime / runner | 多节点编排、恢复、回放、审计 | workflow 定义应归项目，不归工具品牌 |
| skill / agent / prompt bundle | 方法层、触发层、适配层 | 交给 `skills-master` 处理生命周期和发现验证 |
| browser / foreground UI adapter | 人机界面自动化、不可 API 化操作 | 默认显式 opt-in，必须说明会前台驱动 |
| knowledge workspace / report surface | Notion、Obsidian、Docs、Sheets、Slides 等 | 先分清真源、索引、状态、投影和展示 |
| passive sidecar | 纪律、质询、长任务、辅助 workflow | 默认不抢主 owner，只在阈值满足时接入 |

同一个品牌可能跨多个形态。不要让品牌名直接变成项目角色。

## 权责分配

当工具进入讨论时，先把 owner 拆开：

- `MyWay`：识别用户是不是在提工具引入，做低噪声 Prelude，不静默改写成安装任务
- `skills-master`：安装、链接、上游 pin、薄 adapter、多宿主投影和 live discovery 验证
- `files-driven`：判断工具输出能否进入项目真源、写权、流程合同、状态、恢复和审计
- 项目自身：持有领域事实、对象定义、流程定义和最终交付物
- runtime / security owner：持有 secrets、网络、权限、daemon、tunnel、telemetry 和前台控制边界

如果一个工具同时命中多个 owner，先按写权和真源归 `files-driven`，再把安装动作交给 `skills-master`。

## 决策栈

按这个顺序判断，不要从工具名字直接跳到方案：

1. `problem fit`
   - 当前问题是否真的需要新工具
   - 还是宿主原生能力、现有脚本、现有 skill 已经够用
   - 如果只是命中已存在 skill 的正常使用场景，直接按该 skill 工作，不升级成治理问题
2. `manual and practice`
   - 官方手册、源码、CLI help 或配置示例怎么建议接入
   - 社区成熟做法和常见反模式是什么
   - 当前项目或本机是否已有可复用实例
   - 当前方案哪些是事实，哪些只是推断
3. `scope`
   - 只影响本轮运行态：`runtime_scope`
   - 影响一个项目的规则、状态、流程或产物：`project_scope`
   - 影响可复用能力、全局 skill 或多宿主发现：`capability_scope`
4. `role`
   - reader、checker、producer、actuator、orchestrator、projection、sidecar
   - 一个工具可以有多个角色，但每次接入先指定主角色
5. `truth and write`
   - 它能读什么
   - 它能写什么
   - 它绝对不能写什么
   - 产物进入真源前需要谁接受
6. `default posture`
   - 宿主原生优先
   - broad suite 默认 passive
   - foreground、network、telemetry、auto-update、team mode 默认显式 opt-in
   - 项目内 workflow 必须由项目采纳后才成为控制面
7. `risk`
   - secrets、凭证、网络、远端执行、共享目录、破坏性命令、前台 UI 驱动、隐式更新
   - 风险越高，越需要最小权限、审计记录和回退口
8. `state and evidence`
   - 是否要有 receipt、logs、events、state、artifact path
   - 是否需要冷启动恢复或跨会话接续
9. `validation`
   - 正向 smoke：工具该触发时能触发
   - 近失 smoke：工具不该触发时保持安静
   - 权限 smoke：不能越界读写
   - fresh-session / live discovery：宿主真的看到了新入口
10. `rollback`
   - 怎么禁用
   - 怎么移出 discovery path
   - 怎么恢复原入口
   - 生成物和状态如何标记为 projection 或废弃

需要落成判断时，用这张卡，不需要每次都完整展开。

```yaml
tool_intake:
  tool: ""
  request: ""
  job_to_be_done: ""
  research:
    official_manual: []
    live_instance: []
    community_practice: []
    inference: []
  tool_shape: ""
  owner:
    primary: ""
    secondary: []
  scope: "runtime_scope | project_scope | capability_scope"
  role: "reader | checker | producer | actuator | orchestrator | projection | sidecar"
  default_posture: "native-first | passive | opt-in | project-local | always-on"
  read_scope: []
  write_scope: []
  deny_scope: []

    positive_smoke: ""
    near_miss_smoke: ""
    boundary_smoke: ""
  rollback: ""
  next_action: "discuss | research | use-existing | install | adapt | register | defer"
```

## 和其他 reference 的关系

- 工具已经要在多个宿主里暴露同一套结构：继续读 [工具适配对照表](工具适配对照表.md)
- skill / plugin / connector 的安装、链接、触发、同步、上游 pin、分发或 live discovery：交给 `skills-master`
- 已经确定需要主线程 / subagent / `CLI` / runner 分工：继续读 [执行面判定与CLI生产策略](执行面判定与CLI生产策略.md)
- 工具会接触全局 rules、项目 rules、pack rules 或适配入口：继续读 [多规则工具治理与精准修改协议](多规则工具治理与精准修改协议.md)
- `MyWay` 挂载的 sidecar 输出要进入项目事实或流程合同：继续读 [MyWay Sidecar Governance Integration](myway-sidecar-governance.md)
- Obsidian、Notion、Docs、Sheets、Slides 的页面职责冲突：继续读 [../docs/宿主化知识工作场景矩阵.md](../docs/宿主化知识工作场景矩阵.md)

## 禁止事项

- 把“引入工具”直接等同于“立刻全局安装”
- 明明可以查官方手册和成熟实践，却在本地反复绕路试错
- 因为技能很多，就把每轮普通任务都升级成 catalog 盘点
- 把工具品牌写成项目角色、流程阶段或真源 owner
- 因为工具能生成文件，就默认允许它写项目真源
- 因为安装成功，就宣称宿主 live surface 已经接入
- 把 browser foreground、telemetry、auto-update、network tunnel 或 team mode 静默打开
- 把外部 README、prompt、workflow 示例直接提升成项目合同
- 为了一次轻任务引入重编排、重审计或多 agent 仪式

## 一句话收口

工具引入先是调研和治理判断，再是安装动作；先判 `job / known practice / authority / posture`，再谈 adapter、runtime 和 workflow。
