# 理解把握度与澄清规则

在把仓库现状冻结成治理方案前，先判断自己到底理解了多少。
不要因为“已经看了不少文件”就默认自己理解充分。
这份 reference 只负责判断要不要先提问、先假设还是先收窄未知点，
不替代八维诊断、场景路由或正式输出约定。

## 什么算项目基础信息

至少要能说清：

1. 项目边界与当前目标
2. 主要角色、代理和工具入口
3. 当前真源与当前版本锚点
4. 当前协作形态与风险
5. 用户真正要解决的使用场景
6. 用户当前最关心的首批交付物

## 把握度分级

### `high`

说明：

- 主要边界已经清楚
- 核心真源和入口能对上
- 治理选择大概率不会因为一次澄清就推翻

处理方式：

- 可以继续给治理方案
- 但仍要把证据和推断分开

### `medium`

说明：

- 还剩一到两个关键假设没锁死
- 这些假设可能影响结构分类、工具适配、控制回路或验收边界

处理方式：

- 如果这些假设真的会改结论，就先补一到三个问题
- 如果不会改主结论，就明确假设后继续

### `low`

说明：

- 项目边界、首批交付物、真源、主要工具入口或用户目标仍然模糊

处理方式：

- 先问问题，不要出完整方案
- 先缩小未知点，再冻结诊断

## 什么时候必须先提问

出现以下任一情况，优先提问：

1. 你说不清当前首批交付物是什么
2. 你说不清当前可信事实到底落在哪
3. 你分不清工具入口和项目真源
4. 你不确定当前建议会不会因为边界变动而整体失效

如果这些问题已经被上游材料回答，就不要重复追问用户。

如果项目还在早期，且首批交付物还在变：

- 先启用方向与边界锚点
- 先用说人话需求确认工具包
- 先问一组围绕使用场景和交付预期的短问题

## 提问设计规则

1. 只问会改变建议的问题
2. 优先问使用场景、交付边界和成功标准
3. 不要让用户复述仓库里已经能直接读到的事实
4. 问题数量保持小，但每个问题都要高杠杆
5. 问完以后，要把答案真正回写进判断里

如果已经能给出稳健的最小建议，就不要为了“看起来谨慎”继续堆问题。

## 推荐问法

- “这次第一阶段到底交付什么？尽量只说一件主要交付物。”
- “谁会在什么场景下最先用到它？”
- “如果这次做好，你最想看到什么具体变化？”
- “有哪些东西虽然相关，但这次明确先不做？”

## 输出时怎么处理不确定性

如果把握度是 `medium`：

- 明确写出关键假设
- 只给对这些假设稳健的建议

如果把握度是 `low`：

- 先输出问题和暂存判断
- 不要把猜测包装成定论

规则很简单：
没搞清楚的地方，就把不确定性写出来，而不是靠语言强度掩盖。

边界还在早期漂移时，继续读 [说人话需求确认工具包](说人话需求确认工具包.md)。
需要组织正式回答时，回到 [输出约定](输出约定.md)。
