# 关口硬化与稳定放行

当项目反复出现“该停没停、该问没问、该验没验”，
或者明明已经判成 `partial / blocked`，却还是继续往下游动作滑时，
用这个参考件。

它解决的不是“记忆不够”，而是“控制回路没有真正落地”。
如果约束只活在 prompt、聊天习惯或状态摘要里，
它就不是稳定 gate。

## 什么时候启用

出现以下任一情况，就应该认真考虑启用：

1. 同一类越级动作已经重复出现
2. 自然语言短口令会在 `audit / explain / modify / publish` 之间滑动
3. 已经存在 `partial / blocked / ready` 之类状态语义，但下游不服从
4. 共享 Skill、Workflow 或公共表面会被多人、多代理、多工具重复调用
5. 一次越级动作的回退代价明显高于补一个检查

如果只是偶发失误、低风险、单人一次性任务，
先补轻量澄清和最小读取顺序，不必立刻上完整硬化包。

## 核心判断

先把问题定性清楚：

1. 这是 `memory` 问题，还是 `control` 问题
2. 是入口混路，还是证据不足，还是写入越权，还是结束前没复核

默认优先按 `control` 问题处理。
只要“已经知道规则，却还是继续越级”，
主因通常不是没记住，而是没有真正的 `route / write / stop` 关口。

## 最小稳定解

不要把“关口硬化”理解成多加几句提醒。
最小稳定解至少分四道关：

### 1. `route_gate`

先判断当前输入到底触发哪一类动作。
至少区分：

- `understanding`
- `audit`
- `design`
- `modify`
- `publish`

如果短口令可能跨多类动作，就不能直接执行，
必须先落到显式主意图和修饰项。

### 2. `evidence_gate`

动作升级前，先写清：

1. 当前动作类需要哪些证据
2. 现在哪些证据已经有
3. 哪些证据还缺
4. 缺失证据是否足以阻断升级

关键点：

- 没补齐证据，不等于“先做一点下游动作也行”
- `partial` 只表示“当前回路还能继续”，不表示“可以升级下游”

### 3. `write_gate`

当状态仍是 `partial / blocked` 时，
明确禁止哪些输出类型进入下游。

至少区分：

- 可继续产出：分析、证据、问题地图、澄清问题、验证记录
- 暂不允许产出：替换文本、执行脚本、发布结论、正式放行、面向外部的最终输出

真正要防的是：

- `next_step` 偷偷升级
- `process_projection` 把“建议下一步”写成“可直接执行”
- 状态页或综合报告提前给出下游动作选项

### 4. `stop_gate`

结束前再复核一遍：

1. 这轮实际动作类是什么
2. 有没有越级产出下游输出
3. 有没有把 `partial / blocked` 说成“可继续执行”
4. 有没有绕过既定回退或批准边界

如果已经越级，不要靠“下一轮修正”掩盖，
应直接回到上一层重判。

## 最小关口包应该落在哪

`files-driven` 里，关口硬化不是漂在空中的提示。
至少要落到这些结构：

### `policy_or_rules`

放“什么情况下必须拦”“哪些状态禁止哪些输出”。
这是 gate 规则真源。

### `object`

放稳定状态语义和动作类语义。
最少建议有：

- `action_class`
- `gate_state`
- `required_evidence`
- `missing_evidence`
- `allowed_next_steps`
- `forbidden_outputs`

### `workflow`

把路径正式写成：

`route_gate -> evidence_gate -> write_gate -> stop_gate`

必要时与：

- `discussion -> decision_package -> task_or_decision`
- `proposal -> validation -> shadow/canary -> activation_or_rollback`

组合使用。

### `execution_object`

至少要有能承载 gate 结果的过程载体，
例如：

- `decision_package`
- `validation_record`
- `review`
- `process_projection`

### `status_projection`

只负责摘要当前 gate 状态，
不能擅自把 `partial` 总结成“可以开始改”。

### `display_projection`

只能转述已放行结果，
不能生成新的放行结论。

## 推荐最小状态语义

如果项目之前没有稳定状态语义，
建议至少从这三档开始：

- `blocked`
  - 当前不得升级下游动作
  - 只能补证据、补澄清、重路由或回退
- `partial`
  - 当前回路可继续
  - 但仍不得进入更下游动作类
- `ready`
  - 仅对明确写出的 `allowed_next_steps` 生效
  - 不等于“后面任何动作都可以做”

不要把 `partial` 当成“基本可以了”。
不要把 `ready` 写成无限放行。

## 推荐最小产物

如果项目已经被 gate 问题反复打穿，
建议至少补这三样：

1. 一份 gate 真源页
   - 写清动作类、状态语义、允许升级条件、禁止输出类型
2. 一份 gate 状态对象
   - 让当前项目实例能显式声明 `blocked / partial / ready`
3. 一套最小检查
   - 哪怕只是脚本、清单或固定复核段落，也要能在结束前检查“有没有提前放行”

如果工具支持 hooks、permissions、validator 或 stop checks，
优先把检查接到这些执行点；
如果不支持，至少放进受控脚本或固定 `self-audit`。

技术落地时，建议把检查协议也按四类 gate 对齐：

- `route`
- `evidence`
- `write`
- `stop`

并明确：

- check 只返回观测结果、证据缺口或阻断信号
- check 不授予放行权
- 合法转移仍由 workflow、rules 和 approval 边界共同决定

## 评测与回归

只补规则，不补回归用例，通常会再次漂回聊天默契。
最少应有两类回归：

### 1. 路由混淆用例

验证：

- “开始审计”
- “继续推进”
- “讲解一下”
- “帮我修改”

这些短口令会不会误触发到错误动作类。

### 2. 阻断失效用例

验证：

- 已判 `partial / blocked` 时
- `next_step`
- `process_projection`
- 综合报告

会不会偷偷提前给出下游执行动作。

如果项目是共享基础设施，
再补：

- 邻近 Workflow 的 route confusion 用例
- 旧状态语义回流用例
- 放行条件被弱化的回归用例

## 常见失效模式

1. 用 memory 或摘要替代硬 gate
2. 只有 `gate_state`，没有 `forbidden_outputs`
3. 只有入口判断，没有结束复核
4. `partial` 看起来很谨慎，实际却能一路滑进修改或发布
5. 状态页、综合报告或 `next_step` 成了旁路放行口
6. 多工具入口各自发展出不同的动作升级规则

## 在本技能里的推荐用法

当用户抱怨：

- “为什么又提前开始改了”
- “为什么明明没审完却想发布”
- “为什么 `blocked` 了还继续”
- “为什么同一句口令每次走向不一样”

默认不要只给抽象反思。
应直接把它当成一个正式治理问题，
给出关口硬化方案：

1. 当前是哪类 gate 失效
2. 真源应该落在哪
3. 当前缺哪几个最小产物
4. 哪条流程要升格成默认流程
5. 哪些检查要变成脚本、hook、validator 或固定回归
