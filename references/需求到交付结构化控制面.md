# 需求到交付结构化控制面

## 适用场景

当一个 existing repo 不再只是写 PRD，而是要把需求提交、PRD、story、工程移交、开发记录、测试证据、release gate 和 closure 串成可检查链路时，启用本参考。

这不是产品内容能力。它是 `files-driven` 的治理控制面能力。

## 默认分工

1. `files-driven`
   - 定义 metadata profile、状态词表、ID 规则、schema、validator、写权和 stop gate。
   - 注册控制面资产的位置和层级。
   - 判断 traceability index 是 `execution_object`、`status_projection` 还是正式合同面。
2. `product-engineer`
   - 提出产品内容侧字段需求。
   - 按项目已经定义的 metadata profile 填写候选卡、PRD patch、story、acceptance 和 handoff。
   - 缺 profile 时输出 `files-driven collaboration request`，不自行发明 schema。
3. 工程 skill
   - 只消费已接收需求、handoff refs 和工程 surface。
   - 回写 execution、test、evidence、release 和 closure refs。

## 最小 profile

下游项目的最小需求 lifecycle profile 应回答：

1. 需求入口在哪里：raw request、candidate intake、accepted PRD、stage PRD、handoff。
2. 哪些状态允许出现：candidate、needs clarification、accepted、merged、handed off、implemented、verified、blocked、closed。
3. 哪些 ID 必须稳定：request id、requirement id、story id、acceptance id、handoff id、execution id、evidence id、release gate id。
4. 哪些引用可以为空，哪些引用必须在 handoff 前补齐。
5. cross-repo handoff 是否能直接生效；如果不能，项目内 acceptance mapping 放在哪里。
6. schema / validator 放在哪里，失败时是 stop、warning 还是人工 review。
7. 工程完成后回写到哪些文件、index 或 manifest。

## 层级规则

1. Markdown 正文承载人读判断，不因补了 metadata 而失去产品真源或候选正文身份。
2. JSON / frontmatter metadata 承载机器可检查的身份、状态和引用，不替代正文解释。
3. JSON schema 和 validator 是控制面，不能自动接受需求。
4. traceability map 如果存在，默认是 `execution_object / control index`，不是新的 PRD 或需求总账真源。
5. `status_projection` 只能摘要状态，不得反向改变需求、PRD 或 release gate。

## ProductEngineer 协作请求模板

```md
files-driven collaboration request

- requested control surface: `<metadata_profile | traceability_index | validator | downstream_adapter>`
- why needed:
- product-side fields suggested:
- downstream project adapter needed: `<yes/no>`
- blocked until files-driven verdict: `<yes/no>`
```

## Validator 只做什么

1. 检查必填 metadata 是否存在。
2. 检查状态值、架构耦合 verdict 和 route target 是否在允许词表中。
3. 检查 `depends_on`、`supersedes`、`superseded_by` 指向的对象是否存在。
4. 阻止 `candidate`、`superseded`、`invalid_merge` 被误当成工程 handoff。
5. 提醒 cross-repo handoff 需要本项目 acceptance mapping 后才可在本仓计划中生效。

Validator 不做产品 acceptance，不替代 PRD review，不自动改状态。

## 多 Agent 授权决策链

当需求到交付链路由多个 Agent 持续推进时，不能把所有判断都收回一个中心线程，也不能让一线 Agent 凭感觉直接放行。
默认采用两层决策：

1. `frontline agent`
   - 离具体 lane 最近，先给现场默认方案。
   - 负责提交证据、风险、替代方案、rollback、测试计划和下一步草案。
2. `project owner agent`
   - 代表项目 Owner 维护开发计划控制面并审查依据。
   - 在授权范围内可以签字、驳回、暂缓或要求人工裁决。
   - 可以通过留痕的 `harness_action` 调整下属 Agent 的 prompt、schedule、status 或批次深度。
3. `dashboard / projection agent`
   - 只投影项目真源、一线报告、计划控制和 verdict。
   - 不裁决、不签字、不把展示状态反向写成真源。

这条链的目标是让“听见炮声的人”先判断，同时让签字有证据、有范围、有回退。

多 Agent 心跳或长期自动推进不应靠三个越来越长的 prompt 堆规则。默认先建立项目级共享控制合同，再让各 Agent prompt 保持薄入口。

共享控制合同至少固定：

1. 指挥链：谁发 `agent_directive`，谁提交 `frontline_default` / `owner_verdict_dossier`，谁只投影。
2. 上下文继承：每个 Agent 每轮必须读取哪些线程、真源和上一轮输出。
3. 固定输出包：`plan_control`、`agent_directive`、`owner_verdict`、`harness_action`、`dashboard_card` 等。
4. 轻 / 重循环：无变化时静默或降频，有阻隔 / 新证据 / 重复短促汇报时升级。
5. 停损规则：越界写入、证据不足签字、投影冒充真源、负责人没有计划控制等都必须停或改。

## Workflow-backed trigger 与 Harness 拓扑

当心跳或长期自动推进需要跨阶段、循环、审批、并行 review、恢复或跨入口复用时，可以把它下沉成 workflow-backed trigger。Archon 这类 workflow engine 的可吸收思想是：把流程写成 DAG / loop / approval / cancel / artifact chain，让 Agent 只在需要判断的节点发挥作用，验证、停止、恢复和投影尽量由可审计 harness 承担。

默认映射：

1. `trigger`：只负责启动、恢复或改写 workflow run。
2. `workflow node`：承载具体 Agent 角色、专业定位、工具权限和输出格式。
3. `loop`：必须有边界、完成信号、最大轮数和尽量确定性的退出检查。
4. `approval / owner_verdict`：只处理授权签收、驳回、暂缓或要求补证据。
5. `cancel / pause`：用于前置条件不满足、越权、重复无效循环或证据不足，状态应是有意停损，不是模糊失败。
6. `artifact chain`：承载计划、实现证据、validator、review、rollback 和 handoff，供 fresh-context Agent 继续读取。
7. `adapter`：CLI、Web、IM、GitHub、Webhook 只负责入口和回传；不因可见就成为真源。

Harness 拓扑原则：

1. 能力平面近似扁平：同一代基础模型的推理和编码能力差异不应被当作主要组织依据。
2. 控制平面必须分层：计划、执行、专业质询、投影、验收、恢复需要不同上下文、工具、权限、证据标准和 gate。
3. 多 Agent 博弈不按投票收口；并行 review 产出 findings，随后由 synthesis / owner verdict 节点判断证据是否足以推进。
4. workflow YAML、run log、dashboard 和 approval 记录默认是执行证据；只有被项目真源、注册表、verdict 或受控 pack 接收后，才成为项目事实。

## 计划控制与调度

项目负责人 Agent 不能只做二层评论者。默认每轮先维护 `plan_control`，再发指令或签字。

`plan_control` 建议字段：

1. `lane_queue`：当前开发线、优先级、状态。
2. `dependency_map`：哪些能并行，哪些必须等集成、签字或证据。
3. `current_focus`：本轮推进的唯一或主 focus。
4. `batch_policy`：`single_step`、`work_batch` 或 `deep_push`。
5. `gate_map`：当前硬门、软门和返回 gate。
6. `evidence_debt`：缺哪些 E2E、validator、rollback、test plan 或 signoff。
7. `tool_routing`：哪些用 files-driven / product-engineer / gstack / E2E / 本仓脚本。
8. `dashboard_policy`：哪些变化需要投影，哪些应 `DONT_NOTIFY`。

调度动作默认通过 `harness_action` 留痕。调频不是越权，但必须写清：

1. `target_automation`
2. `action`
3. `from_state`
4. `to_state`
5. `reason`
6. `expected_effect`
7. `restore_or_recheck_condition`

心跳频率不限制单轮工作深度。3 分钟可以只是唤醒 / 接力频率；一线 Agent 仍可执行一个 `work_batch`，包括重新读取范围、开发 / 落档 / 证据构造、自审、反思和下一轮交接点。只有遇到 hard gate、权限边界、缺少指令或越界风险时，才退回 `single_step`。

## 授权 verdict 类型

建议最小 verdict 词表：

1. `owner_signed`
   - 依据充分，范围属于本项目内部授权边界，可以生效。
2. `owner_rejected`
   - 依据充分，但方案不通过；必须说明返回 gate 和重新提交条件。
3. `owner_deferred`
   - 方向可能成立，但缺关键证据；必须说明补证据路径。
4. `owner_needs_human`
   - 超出项目负责人授权范围，或用户明确保留人工裁决。
5. `decision_basis_insufficient`
   - 证据不足，不允许签字。

每个 verdict 必须写清：

1. 签字对象。
2. 签字范围。
3. 生效层级。
4. 决策依据。
5. 限制和禁止事项。
6. 返回 gate。
7. 何时失效。
8. 是否仍需 downstream / release / human signoff。

## 决策依据分流

不同争议不能用同一种“讨论”解决：

| 争议类型 | 首选依据 | 不能替代什么 |
| --- | --- | --- |
| 当前主线、写口、依赖、禁止事项 | 项目真源 / inventory | 不替代 Owner verdict |
| 需求拆合、PRD patch、handoff | `product-engineer` + 项目 profile | 不定义 schema / validator / 写权 |
| 真源、gate、状态、恢复、审计 | `files-driven` | 不写产品 acceptance |
| 架构耦合、refs / mapping、准入、release-risk | 多专家质询 / `gstack` | 不代签、不证明跑通 |
| 行为、实现、fixture、runtime、UI、terminal、回归 | E2E / validator / 本仓脚本 | 不自动升级为 release / showcase admission |
| 本项目内部签收、驳回、暂缓 | 项目负责人 Owner verdict | 不替代下游仓库、release owner 或用户保留裁决 |

如果多专家质询与 E2E 证据冲突：

- E2E / validator 决定“是否跑通”。
- 多专家质询决定“风险是否可接受、解释是否完整、是否需要补 gate”。
- 是否升格为正式 accepted / admitted / released 仍回到对应 Owner verdict 或 release gate。

## 最小产物

### `frontline_default`

一线 Agent 每轮先提交：

- lane
- 推荐下一步
- 依据
- 风险
- 替代方案
- 需要项目负责人挑战或放行的点

### `owner_verdict_dossier`

需要项目负责人代表 Owner 签字或驳回时，一线 Agent 必须提交：

- 签字对象
- 范围
- 项目真源 refs
- hard gate 表
- 风险和 rollback
- test plan / validator / E2E 证据
- 多专家质询结论，如果争议需要
- 推荐 verdict
- 仍需升级给用户、下游 owner 或 release owner 的事项

### `decision_basis_review`

项目负责人审查：

- 依据是否足够
- 使用多专家质询还是 E2E / validator
- 是否在授权范围内
- 是否可以签字、驳回、暂缓或必须升级

### `decision_contest_packet`

两层判断冲突时，必须记录：

- 一线默认方案
- 项目负责人挑战点
- 争议类型
- 双方依据
- 用什么证据解决
- 最小下一步
- 是否需要用户、下游 owner 或 release owner 裁决

## Stop rules

以下情况必须停在当前 gate，不得进入下游实现或 release 叙事：

1. 没有 `frontline_default`，项目负责人直接微观指挥。
2. 没有 `owner_verdict_dossier`，项目负责人直接签字。
3. 只有多专家讨论，没有 E2E / validator，却声明实现或行为已通过。
4. 只有 E2E / validator，没有 Owner verdict，却声明 accepted / admitted / released。
5. Dashboard、status projection 或聊天摘要被当成真源或签字面。
6. 项目负责人越权替下游仓库、release owner、真实数据正式服务 owner 或用户保留事项签字。
