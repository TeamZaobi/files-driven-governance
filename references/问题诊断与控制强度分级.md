# 问题诊断与控制强度分级

当用户已经意识到“流程不稳”“多人多工具协作容易漂”“是不是该脚本化”，
但还没判断清问题到底出在哪里、需要多强控制时，用这份 reference。

它不替代底层能力模型，也不直接替代 workflow 设计。
它只回答两件事：

1. 当前真正的问题是什么
2. 当前需要多强的控制能力

## 先诊断，不要先上脚本

优先用下面这张表，先把主要失控点停准。

| 观察到的现象 | 更可能的失控点 | 第一优先动作 | 先不要做什么 |
| --- | --- | --- | --- |
| `README`、状态页、汇报稿慢慢变成事实来源 | 真源不清 | 先冻结真源、投影和读取顺序 | 不要一上来就补 runner |
| 同一流程每轮都走歪、跳步、越权 | 控制不足 | 先补 gate、控制强度和执行面分工 | 不要只靠“换个更强模型” |
| 多个工具说法不一致、规则互相打架 | 角色 / 工具适配边界不清 | 先补工具适配和共享真源判断 | 不要把品牌差异当治理本体 |
| 新人或新 agent 只能靠聊天恢复现场 | runtime 可恢复性不足 | 先补状态摘要、事件留痕和恢复入口 | 不要先扩完整 DSL |
| 节点执行没问题，但串起来总失真 | 编排层不足 | 先补宿主 workflow 或脚本拼接 | 不要立刻重做全部文档 |
| 一上来就想补大量 schema / template / script | 设计强度失配 | 先把问题分型，再选控制强度 | 不要为“看起来完整”而过度设计 |

## 控制强度分级

默认不要问“要不要脚本化”，而要问：

**当前场景应该停在哪一级控制强度。**

### `L0` 主线程直做

适用：

- 目标清楚
- 步骤短
- 风险低
- 不需要回放、恢复、审计

默认工具：

- 宿主主线程

不要升级到更重实现，除非已经出现偏航或交接问题。

### `L1` 宿主 `subagent`

适用：

- 需要上下文隔离
- 子任务边界清楚
- 主要收益是认知分流，不是运行时治理

默认工具：

- 宿主 `subagents`

这一级的核心收益是“换个干净上下文”，不是“形成可回放 workflow”。

### `L2` 宿主原生 `workflow / approvals`

适用：

- 流程比单轮更长
- 需要阶段化推进
- 宿主已经能稳住计划、审批、resume 或 checkpoint

默认工具：

- 宿主原生 workflow
- 宿主 approvals / checkpoints

只要宿主原生能力足够，优先停在这一级，不要为了 `CLI` 而 `CLI`。

### `L3` `CLI` 节点执行

适用：

- 需要明确输入输出
- 需要退出码、产物路径或命令编排
- 需要把节点从主线程中切出来，形成可调用执行面

默认工具：

- `CLI`
- 宿主 workflow 调用 `CLI`

这一级的核心是：

**把节点变成可编排执行面。**

### `L4` 脚本 runner + `CLI` + audit/replay

适用：

- 流程复杂且偏航成本高
- 需要恢复、回放、审计
- 需要跨工具、跨会话甚至跨宿主稳定执行
- 未来可能迁移到独立于宿主的 agent/runtime

默认工具：

- 脚本 runner
- `CLI`
- 显式 state / events / projection

这一级不是默认级别。
只有当宿主和轻量拼接都兜不住时，才进入。

## 最小判断问题集

在决定控制强度前，至少回答这 6 个问题：

1. 当前真正的问题是出在真源、控制、角色边界还是 runtime？
2. 如果这轮流程再走歪，代价到底有多高？
3. 这件事是否需要恢复、回放或审计？
4. 宿主原生 `workflow / subagents / approvals` 是否已经足够？
5. 是否存在明确的节点输入输出和产物边界？
6. 这条能力未来是否需要迁移到独立于宿主的 agent/runtime？

如果前四个问题还说不清，就不要直接升到 `L3 / L4`。

## 反过度设计约束

1. 不要把脚本化当默认答案
2. 不要把控制强度选得高于真实风险
3. 不要为未来假设过多提前造平台
4. 不要把宿主已经做得很好的能力重复实现一遍
5. 只有当控制失稳真实出现时，才补更重的 runtime

## 一句话收口

**先诊断问题，再分级控制；控制强度只升到刚好够用。**
