# 跨层共享约定

当多人、多代理或多工具会碰同一份事实时，使用这个参考件。
这份 reference 只在同一事实确实会被多方共享或消费时启用，
不要求每个项目默认写满共享矩阵，也不替代真源判断、默认动作顺序、读取顺序或角色定义。
如果和 [项目治理能力模型](../docs/项目治理能力模型.md) 冲突，以基线为准；这份文档只解释共享专题。

如果当前只是感觉“项目在漂”“状态页和规则打架”“不同工具说法不一致”，
先不要把问题抛成共享矩阵；
先把它翻回：

1. 哪份文件算数
2. 谁能直接改
3. 哪些页面只能总结
4. 冲突时回哪份上游

只有这些问题已经成立，才继续展开共享约定。

核心目标是防止三种失真：

1. 大家都在写，但没人知道谁有最终写权
2. 状态摘要或展示页反过来改写真源
3. 工具入口各自发展出不同版本的事实

## 核心规则

对当前确实跨层共享的重要结构，至少明确：

1. 谁生产
2. 谁消费
3. 哪一层能直接写
4. 哪一层只能总结或展示
5. 谁能看见
6. 什么事件会触发同步
7. 冲突时谁优先
8. 交接时最少带哪些信息

如果项目已经引入：

1. `workflow.contract.json`
2. `workflow.state.json`
3. `workflow.events.jsonl`
4. `checks/`

先判当前动作、`scope` 和一级关口，再确认归属，最后才展开共享矩阵。
必要时再补结构家族和四层定位。
共置不等于归属；
能一起找，不等于能互相改写。

## 可裁剪共享矩阵

只实例化当前项目真的涉及的结构家族，不要默认八类全开。

| 结构家族 | 生产者 | 主要消费者 | 可直接写的层 | 只可总结的层 | 常见同步触发 |
| --- | --- | --- | --- | --- | --- |
| `policy_or_rules` | 规则责任人 | 全部角色与代理 | 真源层 | 状态摘要、展示层、工具入口 | 规则批准、规则修订 |
| `object` | 对象责任人 | 流程、技能、角色 | 真源层 | 渲染页、校验辅助页 | 对象变更、字段调整 |
| `workflow` | 流程责任人 | 角色、技能、执行者 | 真源层 | 看板、状态页、脚本包装 | 关口变化、流转变化 |
| `skill` | 技能维护者 | 执行者、代理 | 技能包真源 | 命令包装、工具入口 | 技能升级、绑定参考件变化 |
| `agent` | 角色责任人 | 工具运行时、协作者 | 角色真源 | 运行时映射、工具包装 | 角色边界变化 |
| `execution_object` | 当前执行者、流程执行面 | 状态摘要、展示层、复核者、接手 agent | 实例层 | 状态摘要、展示层、工具入口 | 节点切换、证据到达、验证完成、决策落盘 |
| `status_projection` | 状态维护者、交接整理者、自动摘要器 | 人类协作者、接手 agent、工具入口 | 状态投影层 | 展示层、对外镜像、工具入口 | 实例变化、关口刷新、阶段交接 |
| `display_projection` | 展示维护者、汇报自动化 | 对外读者、汇报界面、公共入口 | 展示层 | 外部镜像、二次传播入口 | 状态同步、对外发布、阶段汇报 |

## 条件补充：受控 workflow 的共置资产

当受控 workflow 目录里同时出现合同、实例、状态页和检查器时，
默认按以下归属理解：

1. `WORKFLOW.md` 与 `workflow.contract.json` 属于 `workflow`
2. `workflow.state.json` 与 `workflow.events.jsonl` 属于 `execution_object`
3. `checks/` 属于 `skill` 或工具适配执行面
4. `status.projection.json`、handoff 包、状态页、恢复页属于 `status_projection`
5. 汇报页、说明页、deck、对外公告属于 `display_projection`

目录共置只是为了降低查找成本，
不是为了让某个家族顺手夺走别的家族的写权。

## 条件补充：`process_projection`

当不同工具各自保留自己的 `walkthrough / trace`，
但人和下游 agent 需要统一理解“这次到底跑了什么”时，
可以实例化一个 `process_projection` 共享约定。

它不是新的结构家族，也不是新的真源；
它只是从 `execution_object` 派生出的统一过程投影。
默认不为简单单 agent 直线执行启用；
只有多工具过程已经不透明时才启用。

最少写清：

1. 上游来源：哪些 `discussion / review / decision / handoff / walkthrough / trace` 会被纳入
2. 生产者：当前执行者、工具适配器，或两者组合
3. 主要消费者：人类协作者、接手 agent、讨论或汇报界面
4. 可直接写的层：只允许写上游过程载体；`process_projection` 只允许 `summarize / display`
5. 默认落点：给接手面时归入 `status_projection`；给讨论或汇报时归入 `display_projection`
6. 最小字段：`goal / actions / findings / decisions / artifacts / status / handoff_focus`
7. 常见同步触发：阶段完成、关键决策、路线切换、关键证据到达、过程过长、发生交接

如果上游已经有 `allowed_next_step_refs`、放行结论或 gate 合同，
`process_projection` 只能引用、压缩或显示，
不能自己发明“下一步已经获准”。

如果复杂性主要来自显式 `workflow` 或多 agent 协作，
再条件补一个 `topology_supplement`，至少写清：

1. 参与节点：哪些 agent、工具、流程节点在场
2. 流转关系：谁触发谁、谁依赖谁、哪里并行、哪里合流
3. 关口：哪些节点承担批准、复核、回退或收敛职责
4. 默认落点：按需要附着在 `process_projection` 的 `status_projection` 或 `display_projection` 版本中
5. 边界：只补拓扑，不替代 `process_projection` 的目标、动作、结论和状态字段

## 最小权限动词

为了降低歧义，建议至少区分：

- `read`
- `write`
- `approve`
- `summarize`
- `display`
- `archive`

不要把“能看”和“能改”混成一个权限。
不要把“能总结”和“能批准”混成一个权限。

## 可见范围

常见范围包括：

- 仅责任人可见
- 项目内部可见
- 多工具共享可见
- 对外公开可见

可见范围越大，越要强调“只能总结，不能改写真源”。

## 同步触发时机

同步通常在这些时机发生：

1. 真源被批准变更
2. 当前版本锚点更新
3. 状态摘要需要刷新
4. 展示层需要对外同步
5. 发生交接

不要用“有空再同步”当默认规则。

## 冲突处理规则

至少明确：

1. 真源优先于摘要页
2. 已批准版本优先于未批准草案
3. 责任边界清楚的一方优先于无主文档
4. 工具入口冲突时，回到上游真源裁决
5. `process_projection` 冲突时，回到共同上游过程载体裁决
6. `topology_supplement` 冲突时，回到共同上游流程定义或角色边界裁决

## 交接包最少包含什么

一份可用交接至少包含：

1. 当前目标
2. 当前可信事实入口
3. 当前活跃过程载体
4. 当前卡点
5. 下一步动作
6. 需要避开的越界点

## 多工具规则

如果同一项目有多个工具入口：

1. 每个入口都应指回同一套上游真源
2. 每个入口都应遵守同一套读取顺序
3. 每个入口都应共享同一套短口令主语义
4. 每个入口都应共享同一套合同键名、实例语义和状态含义

如果共享前提还没判清，先回到 [项目治理能力模型](../docs/项目治理能力模型.md) 确认动作、`scope` 与一级关口，再补看 [结构家族定位约定](结构家族定位约定.md) 和 [官方读取顺序](官方读取顺序.md)。

## 红旗信号

看到这些现象，通常说明共享约定不够清楚：

- 不同工具展示的“当前状态”不一致
- 摘要页比真源更新得更快
- 交接时只能靠聊天回放恢复上下文
- 没人说得清谁有最终写权
