# 工具适配对照表

当同一个项目需要在多个工具里暴露同一套结构时，用这个参考件。
这份 reference 只在工具入口已经明显影响读取、启动或协作语义时启用，
不要求每个项目先写一整套工具适配方案，也不替代真源判断、读取顺序或角色定义。

如果当前只是发现“不同工具说法不一致”“AI 改着改着就飘了”“不知道哪个入口算数”，
先不要从品牌差异或适配表开始讲；
先把真源、下一步和越权点讲清，再回来看工具入口到底在哪里放大了问题。

目标不是让每个工具看起来一样，而是让它们都服务于同一套真源。

如果当前困惑不是来自 `Codex / Claude Code / CLI / MCP` 这类运行入口，
而是来自 `Obsidian / Notion / Docs / Sheets / Slides` 里的项目页、索引页、状态页、`Canvas`、`Bases` 或 database view，
不要在这里硬套“工具程序适配”。
先读 [宿主化知识工作场景矩阵](../docs/宿主化知识工作场景矩阵.md)，
把页面职责翻回真源、投影、索引和展示，再决定是否回到这里看工具适配。

## 对照表

| 关注点 | 真源层应该做什么 | 工具适配入口可以做什么 | 绝对不要做什么 |
| --- | --- | --- | --- |
| 规则与约束 | 定义规则本身 | 暴露入口、摘要、跳转 | 私自新增规则 |
| 对象层 | 定义对象结构 | 渲染字段、提供查找 | 私造对象结构 |
| 流程层 | 定义路径、关口和控制合同 | 暴露状态、提供启动入口、执行检查 | 用工具脚本替代流程定义或重写流程合同 |
| 技能层 | 定义可复用做法 | 提供命令、包装或启动方式 | 让包装文案取代技能真源 |
| 角色层 | 定义职责边界 | 绑定运行时、工具参数 | 让工具名冒充角色名 |

## 该怎么理解“适配”

适配入口通常做这些事：

1. 给出进入项目的入口
2. 把上游真源映射成工具友好的形态
3. 帮用户更快找到当前该读什么
4. 提供启动命令、包装命令或界面入口
5. 在不改写真源的前提下执行合同检查

适配入口不应该做这些事：

1. 私自新增项目概念
2. 把未批准的变化写成既成事实
3. 成为唯一可读入口
4. 偷偷改变短口令含义
5. 把结构化合同里的状态、转移或权限重新解释一遍

## 常见工具类别

可以按这几类来理解工具，而不是按品牌来理解：

1. 对话与开发工具
2. 启动与运行时包装
3. 自动化与调度工具
4. 数据与指标入口
5. 展示与汇报界面

只覆盖当前项目真的存在的工具类别，不要为了完整性把所有类别都展开。
这份表的作用是压住品牌噪音，
不是把用户第一步的问题改写成“先认工具分类”。
同一品牌可能跨多个类别，不要把品牌名直接当成角色或结构家族。

## 常见命名工具示例

以下名字通常都应被视为工具适配入口，而不是真源：

- `Claude Code`
- `Codex`
- `AntiGravity`
- `OpenClaw`
- 各类 CLI 包装
- 各类 MCP 入口

这里要额外区分一层：

- 这些程序、命令和入口通常属于 runtime 或适配层
- 但仓库里被这些 runtime 共同读取的项目级规则文件，不应自动跟着降格成适配层

在 `AI-Native` 同构团队里，如果下列文件实际承担项目运行规则：

- `AGENT.md`
- `CLAUDE.md`
- `CODEX.md`
- `SKILL.md`
- `RULES.md`

那么它们通常应按职责归入：

- `agent`
- `skill`
- `policy_or_rules`

中的真源，而不是被简单视为“某个工具的启动文案”。

## 适配设计规则

1. 每个工具入口都应明确指回真源
2. 工具切换后，读取顺序仍应尽量稳定
3. 工具入口只负责“怎么进入”，不负责“什么算事实”
4. 同一短口令不能因工具不同而偷偷变义
5. 如果项目已经引入结构化合同，工具入口只能消费、执行和展示合同，不能改写合同

## 准备度检查问题

在把某个入口称为“适配入口”前，先问：

1. 不开这个工具，用户还能找到真源吗
2. 工具入口是否只是总结上游，而不是改写上游
3. 当前版本锚点是否在工具外也能找到
4. 工具切换后，角色、规则和流程是否还能保持同一含义

如果还说不清核心结构在哪里，先读 [结构家族定位约定](结构家族定位约定.md)。
如果还说不清先读什么，先读 [官方读取顺序](官方读取顺序.md)。
如果多个入口共享同一事实，继续读 [跨层共享约定](跨层共享约定.md)。
