# 规则入口共享层判定卡

状态：`working draft`

这份 reference 只回答一件事：
**当当前冲突集中在 rules、入口文件或工具语义时，先判断这条规则属于执行者本地、项目共享，还是局部 pack / 子树。**

它不是新的能力模型真源，
不替代 [项目治理能力模型](../docs/项目治理能力模型.md)，
也不替代 [多规则工具治理与精准修改协议](多规则工具治理与精准修改协议.md)。

## 1. 一句话用法

不要一看见 `AGENTS.md / CLAUDE.md / CODEX.md / GEMINI.md / SKILL.md / RULES.md`，
就直接把所有内容都叫成“入口规则”。

先分三层：

1. `runtime_global_rules`
2. `repo_project_rules`
3. `subtree_or_pack_rules`

只有已经确认落在 `repo_project_rules`，
再继续讨论它属于 `入口规则`、`Skill` 路由、`能力规则` 还是 `项目规则`。

## 2. 四个快问

如果当前拿不准一条规则放哪，先问：

1. 这条规则如果只活在我本地，换个人或换台机器后，项目结果会不会变？
2. 这条规则如果要改，是否应该经过仓库 review，而不是我自己悄悄改？
3. 这条规则是整个仓库共享的，还是只对某个子目录 / pack / workflow 成立？
4. 这条规则约束的是我个人的工作方式，还是这个仓里所有执行者的共享语义？

最短判断：

- 只约束个人默认值：优先 `runtime_global_rules`
- 会改变共享结果：提升到 `repo_project_rules`
- 只对局部模块成立：落 `subtree_or_pack_rules`

## 3. 三层判定

| 层级 | 回答什么问题 | 典型承载 | 不该承载什么 |
| --- | --- | --- | --- |
| `runtime_global_rules` | “我这个执行者默认怎么工作？” | 个人输出习惯、工具偏好、本地安全默认值、个人严格模式 | 项目真源、共享写权边界、共享短口令语义、共享 Skill 路由 |
| `repo_project_rules` | “这个仓里大家共同先看到什么、共同服从什么？” | 项目共享入口、共享读取顺序、共享 route / stop 纪律、共享写权边界、共享 Skill 路由 | 只对某个 pack 成立的局部特例、执行者个人偏好 |
| `subtree_or_pack_rules` | “这个子树 / pack / workflow 里有什么局部特化？” | `BOUNDARY.md`、子目录 `AGENTS.md`、局部 workflow / rules / agent 合同 | 反向改写仓库根的共享主语义 |

## 4. 默认放置规则

下面这些内容，默认只应放在 `runtime_global_rules`：

- 回答风格或语言偏好
- 个人常用命令偏好
- 本地工具 allowlist / convenience 约定
- 不影响仓库共享结果的个人严格模式

下面这些内容，默认必须进入 `repo_project_rules`：

- 项目共享读取顺序
- 项目共享写权边界
- 项目共享 stop / review / route 纪律
- 项目共享短口令主语义
- 会影响“哪个 Skill 主导这个请求”的共享路由规则

下面这些内容，默认进入 `subtree_or_pack_rules`：

- 某个 pack 的边界、对象、workflow 局部规则
- 某个子目录的特化写法或更严格限制
- 只对某个模块成立的例外规则

## 5. Skill 路由怎么判

`Skill` 路由最容易被误塞进个人全局 rules。
默认按下面判断：

1. 如果它只是在强化“我个人更喜欢先找哪个 Skill”，但不会改变团队共享结果，可以暂留在 `runtime_global_rules`
2. 如果它会影响“这个仓里谁拥有解释权、谁主导请求、短口令命中谁”，它就属于 `repo_project_rules`
3. 如果它只影响某个 pack / 子目录里的局部路由，才进入 `subtree_or_pack_rules`

一句话收口：

**只要 `Skill` 路由会改变共享协作结果，它就不是个人偏好，而是项目共享语义。**

## 6. 红旗信号

看到下面这些现象，通常说明共享层已经混了：

1. 规则只活在某个工具本地全局文件里，但它实际上在定义项目语义
2. 不同 runtime 各自带着一套互相冲突的项目规则
3. 子树 / pack 规则开始偷偷改写仓库根的共享口径
4. launcher、状态页、README 或投影页慢慢变成实际规则来源
5. 团队讨论一直停在“哪个工具更好用”，没人能说清项目共享入口

## 7. 最小回答形状

当用户问“这条规则到底放哪”时，最少回答 4 句：

1. 当前在哪个 `scope` 说话
2. 这条规则先落在哪个共享层
3. 它为什么不该落在另外两层
4. 下一步该改哪一类文件
