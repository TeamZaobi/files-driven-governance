# 文档生命周期与压缩

当活跃文档越来越厚、越来越难接手时，用这个参考件。

## 核心原则

不是所有写过的文档都应该永远停在活跃层。
一份文档一旦不再承担当前事实、当前工作或当前恢复功能，就应该降级、压缩或归档。

## 推荐生命周期状态

建议至少区分：

1. 当前真源
2. 当前活跃过程载体
3. 当前状态摘要
4. 当前展示输出
5. 历史参考
6. 已归档

## 膨胀信号

看到这些现象时，通常说明该做生命周期复核了：

- 同一主题需要翻很多页才能确认当前状态
- README 或状态页越来越长
- 历史记录和当前记录混在一起
- 很多旧页面看起来仍像当前页面
- 新接手的人找不到最短恢复链

## 常见压缩动作

1. 把旧过程记录压成简短结论页
2. 把历史页面从活跃层移到历史层
3. 把多份重复摘要合并成一份
4. 给仍需保留的旧页面明确加“非当前版本”标记

## 活跃层规则

活跃层应只保留：

- 当前真源
- 当前活跃工作
- 当前状态摘要
- 当前对外展示

不要把“也许以后还会用到”当成停留在活跃层的理由。

## 历史层规则

历史层可以保留证据和背景，但必须满足：

1. 不再冒充当前状态
2. 不再承担当前写入职责
3. 能从当前入口被清楚地区分出来

## 安全压缩顺序

建议按这个顺序压缩：

1. 先确认当前真源和当前版本锚点
2. 再判断哪些页面已经只是历史
3. 再合并重复摘要
4. 最后归档不再需要直接读取的材料

## 不要这样做

1. 还没确认当前真源就先归档
2. 只删旧页，不留下去向说明
3. 让历史页继续出现在默认恢复链里
4. 把压缩理解成“简单删文件”

## 在本技能里的推荐用法

当你已经判断：

- 活跃文档膨胀
- 读取成本上升
- 陈旧页堆积

就把这套策略作为条件区块补进治理方案。
