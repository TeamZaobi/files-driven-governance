# 文档生命周期与压缩

当活跃文档越来越厚、越来越难接手时，用这个参考件。
这份 reference 只在文档膨胀、历史页堆积或恢复链变长时启用，
不要求项目一开始就设计复杂的生命周期制度，也不替代真源判断。

## 核心原则

不是所有写过的文档都应该永远停在活跃层。
一份文档一旦不再承担当前事实、当前工作或当前恢复功能，就应该降级、压缩或归档。
压缩的对象是活跃读取链、写权误导和重复调度信号，不是单纯追求更短文本。
边界说明如果正在保护真源写口、停止线或责任归属，应该保留、融合或回指到正确真源，而不是为了“更瘦”直接删除。

新增控制文件前，先做奥卡姆检查：

1. 能不能融合进已有真源？
2. 能不能归档为历史证据？
3. 能不能冻结旧文件的活跃读取、调度或签批权？
4. 能不能复用现有真源或只改现有一行？
5. 新文件是否会替代、降级或退役一个旧入口？
6. 只有明显错误、未生效或无保留价值时，才删除旧文件。

如果答案都是否，默认不新增。

## 推荐生命周期状态

建议至少区分：

1. 当前真源
2. 当前活跃过程载体
3. 当前状态摘要
4. 当前展示输出
5. 历史参考
6. 已归档

## 膨胀信号

看到这些现象时，通常说明该做生命周期复核了：

- 同一主题需要翻很多页才能确认当前状态
- README 或状态页越来越长
- 历史记录和当前记录混在一起
- 很多旧页面看起来仍像当前页面
- 新接手的人找不到最短恢复链

## 常见压缩动作

1. 把旧过程记录压成简短结论页
2. 把历史页面从活跃层移到历史层
3. 把多份重复摘要合并成一份
4. 给仍需保留的旧页面明确加“非当前版本”标记

## 活跃层规则

活跃层应只保留：

- 当前真源
- 当前活跃工作
- 当前状态摘要
- 当前对外展示

不要把“也许以后还会用到”当成停留在活跃层的理由。

## 历史层规则

历史层可以保留证据和背景，但必须满足：

1. 不再冒充当前状态
2. 不再承担当前写入职责
3. 能从当前入口被清楚地区分出来

## 安全压缩顺序

建议按这个顺序压缩：

1. 先确认当前真源和当前版本锚点
2. 再判断哪些页面已经只是历史
3. 再合并重复摘要
4. 最后归档不再需要直接读取的材料

## 不要这样做

1. 还没确认当前真源就先归档
2. 只删旧页，不留下去向说明
3. 让历史页继续出现在默认恢复链里
4. 把压缩理解成“简单删文件”

## 在本技能里的推荐用法

当你已经判断：

- 活跃文档膨胀
- 读取成本上升
- 陈旧页堆积

就把这套策略作为条件区块补进治理方案。
如果当前连真源和当前版本锚点都还没判清，先不要做压缩设计。
