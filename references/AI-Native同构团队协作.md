# AI-Native 同构团队协作

当项目已经进入“所有参与者都通过 AI 工具工作”，并且项目级规则文件跟仓共享的阶段时，使用这个参考件。
它不是 `files-driven` 的通用首屏定义，也不适合第一次接触项目的人直接从这里开始。
如果用户当前只是在问“项目为什么会漂、哪份文件算数、今天先做什么”，
先不要从这里起讲；先把主诉答清，再把“共享存储”作为第二层机制解释。
它帮助你避免把问题误判成传统“多人、多工具、多人类角色异构协作”，
转而先按“多个执行上下文共享同一份存储”来设计治理。

这份 reference 不替代结构家族定位、官方读取顺序或跨层共享约定；
它只负责回答一个更具体的问题：
在 AI-Native 同构团队里，多人协作应该先按什么来理解。

## 核心判断

在这个模式下，`多人协作` 可以先做一阶近似：

`多人协作 = 多个执行上下文共享同一 Git 存储`

这里的“执行上下文”可以是：

- 一个人在两个工作站上同时工作
- 两个人各自使用 `Codex / Claude Code / AntiGravity`
- 同一仓库中的多个 agent 会话
- 一个主 agent 加若干并行子 agent

如果它们都读取同一仓库里的项目级规则文件，
并且都围绕同一目录树推进工作，
那么第一性问题通常不是“大家会不会共享不到同一事实”，
而是“同一份存储里哪些路径谁能写，哪些只能读、生成或复核”。

## 适用前提

只有以下前提大致成立时，才默认按这个模式理解：

1. 团队成员主要通过 AI 工具工作，而不是主要靠手工页面协作
2. 项目级规则文件跟仓共享，而不是散落在个人本地配置里
3. Git 仓库承担目录级全量共享底座
4. 工具品牌差异主要体现为 runtime 差异，而不是项目语义差异

如果这些前提不成立，回到更一般的多角色、多工具协作判断。

## 先看什么，不先看什么

在这个模式下，默认先看：

1. 共享仓库根路径
2. 项目级规则文件
3. 当前版本锚点
4. 路径写权
5. worktree / branch 隔离方式

不要一上来先重做：

1. 复杂共享矩阵
2. 人工沟通仪式
3. 工具品牌差异表
4. 过重的 handoff 话术

这些东西不是永远不需要，
而是默认不应先于“共享存储与规则真源”成为第一入口。

## 哪些文件在这里属于真源

在 AI-Native 同构团队里，
仓库中的项目级 AI 规则文件往往不是“工具说明”，而是正式治理真源的一部分。

常见例子：

- `AGENT.md`
- `CLAUDE.md`
- `CODEX.md`
- `SKILL.md`
- `RULES.md`
- 其他被多个 runtime 共同消费的项目级规则文件

判断规则很简单：

如果某个文件实际定义了以下内容，它就不该只被算作“适配入口”：

1. 执行边界
2. 读写约束
3. 审核或停止条件
4. 输出格式纪律
5. 项目特有的稳定工作方式

这类文件应归入：

- `agent`
- `skill`
- `policy_or_rules`

中的对应真源，而不是被笼统压成工具包装文案。

相反，真正属于适配层的通常是：

- 工具程序本身
- 启动命令
- CLI 包装
- UI 入口
- runtime 参数映射

一句话区分：

- `Codex / Claude Code / AntiGravity` 这些程序通常是 runtime 或适配入口
- 仓库里的 `AGENT.md / CLAUDE.md / CODEX.md / SKILL.md` 可能是项目真源

## 默认治理重心

默认先把协作问题改写成以下几项：

1. 哪些目录是真源
2. 哪些目录允许 AI 直接写
3. 哪些目录只能由上游生成
4. 哪些目录需要通过分支或 worktree 隔离并发
5. 当前版本以哪一组规则文件为准

推荐先落清这些边界：

1. 规则文件的优先级
2. 路径级写权
3. 生成物是否可再生
4. 并发隔离单位
5. 回写上游的纪律

## 最小落地顺序

建议先按这个顺序搭：

1. 固定共享仓库与默认读取根路径
2. 固定项目级规则文件集合及优先级
3. 固定 branch / worktree 并发策略
4. 固定哪些目录是直接真源，哪些目录是可再生投影
5. 只有在复杂性真的出现时，再补 workflow、run-state、status-projection

对很多 AI-Native 团队，最早期真正必需的往往不是复杂流程资产，
而是这几件事已经稳定：

1. 大家都从同一仓库读
2. 大家都服从同一组规则文件
3. 大家都知道哪些目录能直接改
4. 生成型资产不会偷偷升格成真源

## 什么时候再升级成“真正复杂的多人协作”

以下情况明显出现时，再从“共享存储视角”升级到更完整的多人协作治理：

1. 不同成员拥有不同批准权
2. 存在正式的外部责任或审计要求
3. 不同 runtime 对同一规则文件出现稳定语义偏差
4. 存在高成本公共发布、回退或事故补救
5. 已跨仓库、跨组织或跨权限域协作

这时再补：

- 跨层共享矩阵
- 角色控制回路
- 更强的 workflow 合同
- 结构化的运行实例与状态投影

## 红旗信号

看到以下现象，通常说明项目虽然自称 AI-Native，
但还没有真正把“共享存储 + 规则真源”做稳：

1. `AGENT.md`、`CLAUDE.md` 只存在于个人本地，不跟仓
2. 不同 runtime 各自携带一套互相冲突的项目规则
3. 生成出来的 docs 或 slides 被长期手工改，却不回写上游
4. 工具品牌名直接替代项目角色名
5. 共享仓库里没人说得清哪些路径允许直接写

## 和其他 reference 的关系

通常按这个顺序使用：

1. 先用本文件判断当前是否属于 AI-Native 同构团队模式
2. 再回 [结构家族定位约定](结构家族定位约定.md) 判清真源归属
3. 需要固定接手顺序时，读 [官方读取顺序](官方读取顺序.md)
4. 只有复杂共享问题真的出现时，再读 [跨层共享约定](跨层共享约定.md)
5. 只有 runtime 差异开始影响语义时，再读 [工具适配对照表](工具适配对照表.md)
