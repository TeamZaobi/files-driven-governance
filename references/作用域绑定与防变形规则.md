# 作用域绑定与防变形规则

这份 reference 是从 [项目治理能力模型](../docs/项目治理能力模型.md) 派生出来的 `v2.1` 专题页。
它只聚焦作用域绑定与防变形，不独立定义整套世界观。

## 1. 这份 reference 解决什么问题

当你已经接受：

1. `v1` 解决了真源、分层和受控 workflow 的基础问题
2. `v2` 解决了 `files-driven = 项目总监` 与工作对象边界

接下来还要解决一件事：

**同一套模型在不同作用域里怎样不变形。**

## 2. 三个默认作用域

### `capability_scope`

`files-driven` 自己作为治理能力主容器的作用域。
这里持有“如何治理”的能力资产，例如分类法、触发法、读取顺序、升级规则、schema、validator、template。

### `project_scope`

被治理项目的作用域。
这里持有项目里的 `Skill`、项目规则、项目实体、真源与投影、写权边界和当前漂移状态。

### `runtime_scope`

当前会话、工具、代理的作用域。
这里持有本轮路由判断、临时读取栈、派生图谱、恢复缓存和 stale 标记。

## 3. 默认绑定规则

除非显式进入 `self-hosting`，以下四分法默认描述的是 `project_scope`：

1. `入口规则`
2. `能力规则`
3. `项目规则`
4. `项目实体`

因此：

1. `files-driven` 默认站在 `capability_scope` 发起治理，不自动把自己当成工作对象
2. 项目里的 `Writer Skill`、`Coder Skill` 和其他项目 `Skill`，在 `project_scope` 里属于能力对象
3. 小说、软件、任务实例、状态页、展示页，在 `project_scope` 里默认属于项目实体
4. `runtime_scope` 的临时判断和缓存，不能回写成长期真源

## 4. `self-hosting` 规则

当治理对象正好是 `files-driven` 自己时，必须先显式声明 `self-hosting`。

这时要先说清：

1. 现在讨论的是 `capability_scope`
2. 还是把 `repo.files-driven` 当成一个被治理的 `project_scope`

不要把能力仓资产和工作对象资产混成同一层。

## 5. 同一文件为什么会有双重身份

同一文件可以跨作用域拥有不同身份。
因此真正的判断主键不是文件名，而是：

`(scope, file_ref)`

例如：

1. `skill.files-driven / SKILL.md` 可以是 `capability_scope` 下的能力资产
2. `repo.some-project / SKILL.md` 可以是某个被治理项目中工作 `Skill` 的能力规则

所以不要给一个文件打全局唯一身份标签。

## 6. 防变形检查单

实际判断时，至少检查这 5 件事：

1. 有没有先标当前 `scope`
2. 有没有把能力仓资产冒充成项目事实
3. 有没有让项目事实反过来直接定义治理能力
4. 有没有让运行时缓存越级改写真源
5. 同一文件跨作用域时，这次到底在替谁说话

如果你发现“谁在治理谁”已经开始混，
或者当前场景涉及 `self-hosting`、双重身份、跨作用域冲突，
请先回到 [项目治理能力模型](../docs/项目治理能力模型.md) 的作用域章节，再用这份 reference 做专项聚焦。
