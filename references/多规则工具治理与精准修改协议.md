# 多规则工具治理与精准修改协议

当同一个项目同时存在多套规则载体，并且用户希望真正做同步、升级、对齐、迁移或收口修改时，使用这个参考件。
它服务的不是“先判断真源在哪”这一步，而是更进一步回答：

1. 这次修改到底应该改哪些文件
2. 怎样避免只改一个工具入口或一份摘要页
3. 怎样让改动既精准又完整，而不是大面积重写或局部漏改

这份 reference 不替代结构家族定位、官方读取顺序或跨层共享约定。
它只处理“多规则载体已经存在，现在要怎么改才不残缺”。

## 适用信号

出现以下任一情况时启用：

1. 同一项目同时有工具全局 rules、跟仓项目 rules 和局部 pack / 子树 rules
2. 用户要求“统一 Codex / Claude Code / AntiGravity / CLI 包装”的项目语义
3. 用户要求同步修改 `AGENT.md / CLAUDE.md / CODEX.md / SKILL.md / RULES.md` 一类规则文件
4. 一次改动会同时影响读取顺序、写权边界、短口令、审批条件或输出纪律
5. 过去经常出现“只改了一个入口，别的入口还在讲旧规则”

以下情况通常不必启用：

1. 单工具、单文件、纯措辞微调
2. 只修投影页排版，不改规则语义
3. 只补一个新的入口链接，但不改变项目语义

## 先分清五类载体

这里讨论的是“范围与承载点”，不是新的结构家族。
结构家族仍按 `policy_or_rules / skill / agent / workflow / object` 等原有体系判断。

### 1. `runtime_global_rules`

工具或用户本地的全局 rules。
常见特点：

- 不一定跟仓
- 常常只被某一个 runtime 读取
- 更像环境默认值，而不是项目共同真源

### 2. `repo_project_rules`

仓库根或项目根共享的项目级规则文件。
常见例子：

- `AGENTS.md`
- `CLAUDE.md`
- `CODEX.md`
- `SKILL.md`
- `RULES.md`

如果这些文件被多个 runtime 共同消费，并实际定义执行边界、读写约束或输出纪律，
它们就属于项目真源的一部分。

### 3. `subtree_or_pack_rules`

子目录、模块、pack 或 workflow 作用域内的局部规则。
常见例子：

- 某个子目录下的 `AGENTS.md`
- `BOUNDARY.md`
- `WORKFLOW.md`
- `rules.contract.json`
- `agent.contract.json`

它们的作用域比项目根更窄，不能静默反向改写上层共享语义。

### 4. `adapter_surface`

工具入口、启动脚本、包装命令、runtime 提示和 MCP 入口。
它们负责“怎么进入”，不负责“什么算事实”。

### 5. `projection_or_handoff`

状态摘要、handoff 包、恢复页、对外说明页、展示页。
它们可以压缩和显示规则结果，但不能反向授予新权限或偷偷改义。

## 判定原则

1. 同一语义先找共同上游真源，不要先改最顺手的入口。
2. 共享项目语义优先落在 `repo_project_rules`，不是落在单工具全局 rules。
3. 局部规则只能收紧、细化或局部特化；如果它开始重写上层主语义，说明分层有问题。
4. `adapter_surface` 只能暴露、映射、执行或跳转，不能私自补一套项目规则。
5. `projection_or_handoff` 只能总结和恢复，不能把“建议”写成“已经批准”。
6. 如果当前只有工具本地全局 rules 承载了项目事实，不要假装问题已经解决；要明确指出治理仍未稳定。

## 修改前必须先做的盘点

在真正动手改之前，至少盘一遍下面这些载体：

1. 仓库根或项目根的共享规则文件
2. 当前任务作用域内的子树 / pack / workflow 规则文件
3. 各工具入口、launcher、CLI 包装和运行时提示
4. 状态页、handoff 页、恢复页和对外说明页
5. 如果已有结构化合同，再盘 `workflow.contract.json`、`rules.contract.json`、`agent.contract.json`

不要找到一个入口就停止。
至少要给自己形成一张最小拓扑表：

| 载体 | 范围 | 是否跟仓 | 是否真源 | 主要消费者 | 本次是否应改 |
| --- | --- | --- | --- | --- | --- |
| `CODEX.md` | 项目级 | 是 | 可能是 | Codex / 其他 agent | 是 / 否 |
| `CLAUDE.md` | 项目级 | 是 | 可能是 | Claude Code / 其他 agent | 是 / 否 |
| 某 pack 的 `BOUNDARY.md` | 子树级 | 是 | 是 | pack 执行者 | 是 / 否 |
| launcher 脚本 | 适配层 | 可选 | 否 | 特定工具入口 | 是 / 否 |
| `status.projection.json` | 投影层 | 是 | 否 | 接手者 | 是 / 否 |

## 先判本次改的是哪一层语义

不要一上来就进入编辑。
先判断这次需求改变的是哪一层：

1. 项目级共享语义
2. 子树 / pack / workflow 局部语义
3. 工具入口文案或启动方式
4. 状态摘要或交接表达
5. 纯本地环境默认值

常见判断：

- 改写权边界、审批条件、默认读取顺序、统一短口令主语义：通常是项目级共享语义
- 只修改某个 pack 的边界、对象、workflow：通常是子树 / pack 语义
- 只调整某个工具如何暴露入口：通常是适配层
- 只刷新状态页或 handoff：通常是投影层

## 写集判定规则

### 1. 改项目级共享语义

至少检查并通常需要修改：

1. `repo_project_rules`
2. 受影响的 `subtree_or_pack_rules`
3. 受影响的 `adapter_surface`
4. 受影响的 `projection_or_handoff`
5. 相关示例、validator、checks 或合同引用

### 2. 改子树 / pack / workflow 局部语义

至少检查并通常需要修改：

1. 当前子树的局部规则
2. 当前子树的 workflow / object / agent 合同
3. 当前子树对应的入口和说明
4. 上层索引或读取顺序是否需要补链接或版本锚点

### 3. 只改工具入口

只在同时满足以下条件时，才允许只改 `adapter_surface`：

1. 上游共享语义完全不变
2. 只是入口呈现、命令包装或跳转方式变化
3. 改完后不会导致不同工具对同一语义产生不同理解

### 4. 改状态摘要或交接包

只在上游真源不变时，才允许只改投影。
如果摘要内容已经和上游语义冲突，先回真源，不要只修投影文案。

### 5. 只改工具全局 rules

只有在用户明确要求“这次只做本地 / 全局环境默认值调整”时才这样做。
如果用户目标是项目统一语义，而你只改了某个工具的全局 rules，这通常算未完成。

## 精准且完整的修改顺序

推荐顺序：

1. 固定共同上游真源和本次语义增量
2. 先改共享真源
3. 再改局部例外或 pack 级规则
4. 再改工具入口和运行时映射
5. 最后刷新状态页、handoff、示例、测试和 validator

这里的“精准”指的是：

1. 不为了求稳而把整套文档全改一遍
2. 不把无关工具和无关 pack 一起重写
3. 不在投影层偷偷补规则

这里的“完整”指的是：

1. 同一语义的 sibling files 没有留旧版本
2. 下游入口都还能正确指回上游
3. 新旧优先级、写权和读取顺序没有互相打架

## 完整性检查清单

交付前至少检查：

1. 同一组项目级规则文件里，是否还有兄弟文件保留旧语义
2. 如果“规则文件集合及优先级”变了，是否同步更新相关入口
3. 如果新增或移除了真源入口，是否更新读取顺序
4. 如果改了写权边界，是否同步更新共享约定或交接入口
5. 如果改了短口令主语义，是否同步更新意图入口和示例
6. 如果改了合同或 gate 语义，是否同步更新 validator、checks 或 smoke 示例
7. 如果这次只改了本地全局 rules，是否明确说明这是局部生效，不是项目统一生效

## 常见失误模式

1. 只改 `CODEX.md`，没有检查 `CLAUDE.md / AGENTS.md / SKILL.md`
2. 只改 launcher 或启动提示，却没改项目真源
3. 只改状态页，让它看起来“已经一致”
4. 改了 pack 局部规则，却忘了上层索引、读取顺序或真源入口
5. 改了共享语义，却没检查结构化合同、checks 和 validator
6. 把“本地方便”误当成“项目统一完成”

## 必须停下来说明的情况

遇到以下情况，不要假装已经完整修好：

1. 当前说不清哪份文件才是真源
2. 同一层级有多份规则互相冲突，且仓库里没有裁决基线
3. 用户想统一项目语义，但你当前只能接触某个工具的本地全局 rules
4. 这次需求会改动项目级语义，但仓库里没有共享规则入口
5. 你只能修适配层，无法回写上游真源

这时要明确说清：

1. 哪一层已经改了
2. 哪一层还没法改
3. 为什么还不能算“项目已统一”

## 推荐回答骨架

如果要让大模型输出更稳定，建议按这个顺序组织回答：

1. 本次修改触及的规则层级与假设
2. 当前规则载体拓扑
3. 本次应改文件集合
4. 实际已改内容与未改原因
5. 一致性检查结果与剩余风险
