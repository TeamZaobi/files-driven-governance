# 意图触发约定

当项目希望通过“继续开发”“开始审计”“反思”“推进”这类短口令触发工作时，用这个参考件。
这份 reference 只在项目已经明确要把短口令当成稳定入口时启用，
不要求每个项目默认设计意图体系，也不替代流程定义、角色定义或状态真源。

目标是让短口令在跨工具、跨会话、跨交接时都保持同一语义。

## 核心规则

1. 一个常用短语只对应一个主意图
2. 易用性放在别名和修饰项层，不放在偷偷变义上
3. 工具可以暴露不同入口，但不能改写主语义

## 三层模型

### 第一层：主意图

这是稳定语义层。
例如：

- 继续当前开发
- 开始审计
- 进入反思
- 按当前主线推进

### 第二层：别名层

这是自然语言友好层。
例如：

- “继续开发”
- “接着做”
- “继续这个”

都可以落到同一个主意图。

### 第三层：修饰项层

这是范围和约束层。
例如：

- “快速”
- “只看当前任务”
- “先不要发版”

## 直接动作与路由动作

建议区分：

- `direct_action_intent`
- `route_intent`

直接动作意图语义固定，不能因上下文偷偷切换。
路由动作意图可以根据项目的既定路由规则决定下一步，但前提是路由规则本身已经写在上游。

## 最小可移植意图集合

一个项目如果确实要用短口令，常见起始集合是：

1. 继续当前开发
2. 开始审计
3. 进入反思
4. 查看当前状态
5. 推进主线

不必一开始全开。
只把当前确实高频、跨工具、跨会话也要保持稳定的主意图正式化。

具体名称可以不同，但语义边界必须稳定。

## 每个意图至少要写清什么

1. 意图编号
2. 中文主叫法
3. 允许的别名
4. 是否属于直接动作或路由动作
5. 可用修饰项
6. 默认读取顺序
7. 绑定流程
8. 执行角色
9. 输出要求
10. 歧义、停止和回退规则

## 修饰项家族建议

常见修饰项包括：

- 范围
- 对象
- 深度
- 速度
- 风险约束
- 发布约束

不要让修饰项偷偷改写主意图，只让它收紧边界或补条件。

## 推荐话语模式

推荐口令长相：

- 主意图 + 修饰项
- 主意图 + 目标对象
- 主意图 + 范围 + 约束

例如：

- “继续开发，先只看当前任务”
- “开始审计，快速过一遍关键风险”

## 默认读取与执行顺序

触发后通常按这个顺序处理：

1. 读当前状态摘要
2. 找当前活跃过程载体
3. 回到相关真源
4. 按绑定流程执行
5. 把结果回写到状态层或过程层

这是常见默认，不是替代项目正式读取顺序。
如果项目已经有稳定读取顺序或状态入口，优先服从上游约定。

## 歧义与停止规则

出现以下情况时，应先停下来问清楚：

1. 一个短语可能对应多个主意图
2. 当前主线不明确
3. 当前活跃对象不明确
4. 修饰项之间互相冲突

## 最小部署包

要让短口令真正可迁移，至少要有：

1. 主意图清单
2. 别名清单
3. 修饰项规则
4. 默认读取顺序
5. 绑定流程
6. 输出与状态同步规则

只为当前真的要迁移的主意图落这些字段，不要先把低频口令一并制度化。

## 最小示例

```yaml
intent_id: continue_mainline
label: 继续当前开发
kind: direct_action_intent
aliases:
  - 继续开发
  - 接着做
default_reads:
  - status_projection
  - active_execution_object
  - related_truth_source
```

## 准备度测试

在正式启用前，至少问：

1. 换一个工具后，这个短口令还能保持同一语义吗
2. 只看上游约定，不看聊天历史，也能解释它吗
3. 它会不会因为不同人理解不同而变成多个动作

## 常见失效模式

1. 同一短口令在不同工具里意义不同
2. 主意图没有写清，只有别名
3. 修饰项被拿来偷偷改主语义
4. 语义只活在启动提示里

如果当前流程和状态入口都还没稳定，先回到 [场景手册](场景手册.md) 和 [输出约定](输出约定.md)，
不要先把短口令制度化。
