# v0.4.1 版本说明（草案）

状态：`Unreleased`

## 这次版本真正要解决什么

`v0.4.1` 不是继续给 `files-driven` 叠更多 runtime 包装，
也不是为了证明“脚本化一定比宿主原生 workflow 更先进”。

这次版本真正要解决的问题是：

**当复杂流程长期依赖 LLM 在对话内部自行维持流转时，控制容易失稳。**

对应的设计目标不是“把一切都脚本化”，而是：

**强化控制能力，让用户更容易识别问题、解决问题。**

## 这次版本的关键纠偏

前一阶段容易把“控制面外移”“Workflow 脚本化”“CLI backend 化”说成长期路线。
这一轮要把目标、能力和实现重新拆开：

1. 目标层
   - 帮助用户识别问题，解决问题
2. 能力层
   - 强化控制能力
3. 实现层
   - 视复杂度与风险，选择宿主原生能力、`subagents`、`CLI`、脚本 runner、审批或审计

也就是说：

- 长期路线不是“控制面外移”
- 长期路线是“强化控制能力”
- 控制面外移只是当前阶段的一类实现策略

## 为什么不能把脚本化写成长期本体

因为 `Codex`、`AntiGravity`、`Claude Code` 这类宿主工具本身也在快速增强：

- plan / workflow
- subagents
- approvals
- checkpoints
- session state
- runtime trace

如果我们把“自建脚本 runtime”写成长期本体，就很容易产生三类问题：

1. 和宿主原生能力重叠
2. 为今天的短板过拟合
3. 未来变成冗余设计

所以当前口径应该是：

**宿主原生能力优先，脚本和 CLI 作为补强。**

## 这一版怎样理解 CLI

`CLI` 在这一版里主要有两个价值：

1. 上下文隔离
   - 把局部节点从主对话里切开，减轻长上下文污染
   - 这一点和宿主 `subagents` 有明显重叠
2. 可被脚本拼接
   - 有明确输入、输出、退出码、产物路径和执行留痕
   - 适合被 runner、shell、CI、cron 或外部 orchestrator 编排

因此 `CLI` 不是默认生产面，而是：

**可编排执行面。**

最小判断规则：

1. 只需要认知隔离：优先宿主 `subagents`
2. 需要脚本编排、恢复、回放、审计：优先 `CLI`
3. 两者都不需要：直接在宿主主线程处理

## 这一版怎样理解脚本化 Workflow

脚本化 workflow 的价值不是“比宿主更高级”，而是：

1. 当宿主 workflow 还不够稳时，提供一条更硬的控制路径
2. 帮复杂流程留下可回放、可恢复、可审计的 runtime artifacts
3. 让能力更容易迁移到独立于宿主的 agent/runtime

因此，对这类项目应优先考虑脚本化：

- 未来可能脱离单一宿主
- 偏航成本高
- 需要恢复、回放或审计
- 需要跨工具或跨会话稳定执行

## 本轮先补哪些脚手架

当前方向已经明确，这一轮先补判断与迁移脚手架，而不是继续加重 orchestrator。

已经先落出的最小脚手架有：

1. [references/问题诊断与控制强度分级.md](../references/问题诊断与控制强度分级.md)
   - 先判断失控点在真源、流程、角色边界还是 runtime
   - 再把控制强度收在 `L0 -> L4`
2. [references/执行面判定与CLI生产策略.md](../references/执行面判定与CLI生产策略.md)
   - 明确主线程、`subagent`、宿主 workflow、`CLI`、runner 怎么分工
   - 也明确 `CLI` 什么时候能算 producer，什么时候只该做 checker / actuator
3. [docs/外部项目Workflow改造脚手架.md](./外部项目Workflow改造脚手架.md)
   - 帮用户把外部项目现有 workflow 变成“宿主优先、脚本补强”的最小 starter

仍然保留为下一步的，是更完整的 benchmark / migration 自动化，而不是立刻继续加重 runtime。

## 一句话收口

`v0.4.1` 要收的，不是“脚本化至上”，而是：

**先识别问题，再选择控制强度；先复用宿主，再用脚本和 CLI 补强。**
