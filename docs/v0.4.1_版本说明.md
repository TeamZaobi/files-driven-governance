# v0.4.1 版本说明（草案）

状态：`Unreleased`

## 这次版本真正要解决什么

`v0.4.1` 不是继续给 `files-driven` 叠更多 runtime 包装，
也不是为了证明“脚本化一定比宿主原生 workflow 更先进”。

这次版本真正要解决的问题是：

**当复杂流程长期依赖 LLM 在对话内部自行维持流转时，控制容易失稳。**

对应的设计目标不是“把一切都脚本化”，而是：

**强化控制能力，让用户更容易识别问题、解决问题。**

## 这次版本的关键纠偏

前一阶段容易把“控制面外移”“Workflow 脚本化”“CLI backend 化”说成长期路线。
这一轮要把目标、能力和实现重新拆开：

1. 目标层
   - 帮助用户识别问题，解决问题
2. 能力层
   - 强化控制能力
3. 实现层
   - 视复杂度与风险，选择宿主原生能力、`subagents`、`CLI`、脚本 runner、审批或审计

也就是说：

- 长期路线不是“控制面外移”
- 长期路线是“强化控制能力”
- 控制面外移只是当前阶段的一类实现策略

## 为什么不能把脚本化写成长期本体

因为 `Codex`、`AntiGravity`、`Claude Code` 这类宿主工具本身也在快速增强：

- plan / workflow
- subagents
- approvals
- checkpoints
- session state
- runtime trace

如果我们把“自建脚本 runtime”写成长期本体，就很容易产生三类问题：

1. 和宿主原生能力重叠
2. 为今天的短板过拟合
3. 未来变成冗余设计

所以当前口径应该是：

**宿主原生能力优先，脚本和 CLI 作为补强。**

## 这一版怎样理解 CLI

`CLI` 在这一版里主要有两个价值：

1. 上下文隔离
   - 把局部节点从主对话里切开，减轻长上下文污染
   - 这一点和宿主 `subagents` 有明显重叠
2. 可被脚本拼接
   - 有明确输入、输出、退出码、产物路径和执行留痕
   - 适合被 runner、shell、CI、cron 或外部 orchestrator 编排

因此 `CLI` 不是默认生产面，而是：

**可编排执行面。**

最小判断规则：

1. 只需要认知隔离：优先宿主 `subagents`
2. 需要脚本编排、恢复、回放、审计：优先 `CLI`
3. 两者都不需要：直接在宿主主线程处理

## 这一版怎样理解脚本化 Workflow

脚本化 workflow 的价值不是“比宿主更高级”，而是：

1. 当宿主 workflow 还不够稳时，提供一条更硬的控制路径
2. 帮复杂流程留下可回放、可恢复、可审计的 runtime artifacts
3. 让能力更容易迁移到独立于宿主的 agent/runtime

因此，对这类项目应优先考虑脚本化：

- 未来可能脱离单一宿主
- 偏航成本高
- 需要恢复、回放或审计
- 需要跨工具或跨会话稳定执行

## 这次版本还缺什么脚手架

当前方向已经明确，但还需要补判断脚手架，而不是继续加重 orchestrator。

优先要补的有：

1. 问题诊断脚手架
   - 先判断失控点在真源、流程、角色边界还是 runtime
2. 控制强度分级脚手架
   - 明确何时只靠宿主，何时上 `subagents`，何时上 `CLI`，何时上 runner
3. 主线程 / `subagent` / `CLI` / runner 判定表
   - 把工具选择从感觉判断变成工程判断
4. 外部项目 workflow 改造 starter
   - 帮用户把现有 workflow 变成“宿主优先、脚本补强”的最小可行形态
5. benchmark / migration 脚手架
   - 用实际样本验证遵循度、恢复性和迁移性是否真的提升

## 一句话收口

`v0.4.1` 要收的，不是“脚本化至上”，而是：

**先识别问题，再选择控制强度；先复用宿主，再用脚本和 CLI 补强。**
