# 分支推进决策流程

本文只回答一个问题：在当前 `files-driven` 的两条工作分支之间，下一步该如何决策与推进。

适用对象：

- 当前稳定基线分支：`main`
- 当前治理能力重构分支：`codex/governance-capability-v1`

## 当前状态快照

以当前工作树为准：

- `origin/main`：远端公开基线
- `main`：本地稳定基线，可能领先 `origin/main` 若干稳定提交
- `codex/governance-capability-v1`：当前 v1 集成线，通常会领先 `main` 多笔重构提交

精确计数不要手写在本文里，直接以命令结果为准：

```bash
git rev-list --left-right --count origin/main...main
git rev-list --left-right --count main...codex/governance-capability-v1
```

这意味着目前不是“三条线并行”，而是两层推进：

1. `main` 承担稳定基线
2. `codex/governance-capability-v1` 承担 v1 重构集成

## 分支职责

### `main`

只承接下列改动：

- 已经稳定、可独立解释、可直接公开的仓库级说明或小修
- 不依赖治理能力模型 v1 合同体系的修订
- 不要求 project pack、schema、validator、smoke asset 一起演进的改动

默认不在 `main` 上继续推进：

- `workflow.contract.json`、`workflow.state.json`、`status.projection.json` 相关结构演进
- validator 语义扩展
- smoke / example governed pack 的设计推进
- 与能力模型 v1 强绑定的文档和流程

### `codex/governance-capability-v1`

承接下列改动：

- 能力模型 v1 的 schema、validator、example、migration、quickstart
- governed pack 的目录、合同、实例、检查执行面
- `workflow` 作为控制载体的实现收口
- 与上述实现强绑定的文档同步

默认不在该分支做：

- 与 v1 无关的普通 README 润色
- 可独立发布到 `main` 的小修、小补和紧急热修

## 决策流程

### 第一步：先判断改动属于哪一类

如果命中任意一条，继续在 `codex/governance-capability-v1` 上推进：

- 改动触及 `schemas/`
- 改动触及 `scripts/validate_governance_assets.py`
- 改动触及 `examples/smoke-governed-review/`
- 改动需要 `README`、`MIGRATION`、`QUICKSTART` 与 governed pack 一起解释
- 改动会改变 `workflow / object / policy / agent` 合同语义

如果都不命中，并且改动可以独立对外解释，优先回到 `main`。

### 第二步：判断是否需要先同步基线

按下面顺序判断：

1. 如果 `origin/main` 已经前进，而本地 `main` 还没同步：
   - 先同步 `origin/main -> main`
2. 如果 `main` 已有新的稳定提交，而 feature branch 还没吸收：
   - 再同步 `main -> codex/governance-capability-v1`
3. 如果只是 feature branch 内继续迭代：
   - 不要为了“看起来整齐”频繁来回 rebase

原则是：**先稳基线，再稳重构线。**

## 当前两条分支的推荐动作

### 对 `main`

当前 `main` 是否领先 `origin/main`，以实时计数为准。

建议：

1. 把 `main` 上尚未推远端的提交视为“稳定基线候选提交”
2. 如果确认这些提交不依赖治理能力模型 v1 的 schema/validator/example：
   - 可以单独推进到远端 `main`
3. 在这之前，不要继续把 v1 重构内容补到 `main`

### 对 `codex/governance-capability-v1`

当前该分支承载了 v1 的主要实现链，建议继续作为唯一集成分支，直到满足合并门槛。

下一步优先级：

1. 收掉最后的合并前一致性缺口
2. 再做一次 merge-readiness 审计
3. 满足门槛后再决定是否并回 `main`

在上述门槛完成前，不建议直接并入 `main`。

## 合并门槛

只有同时满足下列条件，才考虑把 `codex/governance-capability-v1` 合并回 `main`：

1. smoke pack 校验通过
2. 最小测试集已补齐
3. CI 已能自动跑 validator 和关键测试
4. `README`、`QUICKSTART`、`MIGRATION` 与当前 pack 约定同步
5. 不再存在 schema / validator / example / 文档 四套并行说法

如果只满足 1-4，但第 5 条还没达成，继续留在 feature branch 收口。

## 禁止事项

- 不要在 `main` 和 `codex/governance-capability-v1` 上同时修改同一批 governed pack 合同
- 不要把 feature branch 的零散提交选择性回灌到 `main`，除非它们可以完全脱离 v1 语境独立成立
- 不要在 feature branch 尚未通过最后一轮 merge-readiness 审计前，把它当作稳定公开基线

## 推荐执行顺序

当前建议的实际操作顺序是：

1. 保持 `codex/governance-capability-v1` 作为 v1 的唯一推进线
2. 仅在确认 `main` 上尚未推送的稳定提交可独立公开时，再处理 `main -> origin/main`
3. 先做一次最终 merge-readiness 审计
4. 审计通过后，再决定 `codex/governance-capability-v1 -> main`

这份流程的目的不是增加 Git 仪式，而是避免两条线再次同时演化同一套治理语义。
