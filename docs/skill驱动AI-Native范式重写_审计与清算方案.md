# skill 驱动 AI-Native 范式重写：审计、清算与分支执行方案

> 分支：`rewrite/skill-centric-governance`
> 状态：草案 v1
> 性质：重写前总案卷
> 目标：把 `files-driven` 从“结构治理顾问”重写成“skill 驱动 AI-Native 项目的项目总监替身”

---

## 一、这份文档为什么存在

这不是普通的版本说明，也不是一份零散的改动提案。

这份文档要解决的是一个更根本的问题：

**当前 `files-driven` 的底层逻辑，已经不足以指导 skill 驱动的 AI-Native 项目治理。**

它依然能做很多事：

- 看真源
- 分家族
- 查漂移
- 判投影越权
- 设计恢复链

但它对下面这类问题的回答还不够稳定：

- `Skill` 到底是不是顶层能力主体
- 工具入口、`Skill`、项目规则、项目实体，谁应该先被看见
- 一条规则应该放在哪一层，才能在运行时真正起作用
- 边界原则怎样变成“先判、再落盘、判不清就升级”的动作关口
- 遇到“这个文件放哪”这种高频治理动作时，应该先走哪条流程

这份文档要做的，不是替代后续重写，而是先把：

1. 前因
2. 暴露出的底层偏误
3. 已经收束出的新原则
4. 审计范围
5. 清算对象
6. 分支执行顺序

一次性写清楚。

---

## 二、前因：这次重写是怎么被逼出来的

这次重写不是凭空发生的。

它来自一次真实项目治理纠偏，而且是被连续多轮人工纠偏“顶”出来的。

### 1. 起点不是 files-driven 仓库，而是一个 skill 驱动项目

触发场景来自《折影书》项目。

那个项目不是传统文档驱动项目，而是一个明显的 `skill-driven` 结构：

- 工具入口每轮稳定携带规则
- `mobius skill` 承担作家能力
- 项目文档承担小说业务事实
- agent、workflow、reference 都主要围绕 `Skill` 运转

也就是说，那里真正的主语不是“文件家族”，而是：

- 作家能力怎么被 `Skill` 持有
- 小说内容怎么被项目文档持有
- 两者如何在运行时分边界、分触发、分落位

### 2. 最初问题不是“家族怎么分”，而是“边界怎么生效”

最初讨论的是创作流程、戏剧骨头、实体关系和项目治理。
然后很快进入一个更基础的问题：

**`Skill` 和项目文档的边界，到底怎么定？**

用户后来把这个问题收成一句非常关键的话：

**`Skill` 是作家的大脑、手、眼、身体，是作家的内涵。  
项目文档是作家的外延。**

这个比喻一旦成立，很多原本看起来只是“文档怎么放”的问题，
立刻变成了：

- 什么属于作家能力
- 什么属于小说内容
- 什么规则应该被模型每轮直接看到
- 什么规则只能作为维护长文存在

### 3. 第一次暴露出的偏误：过度依赖解释性引用

一开始，边界规则被抽成一份单独长文。
然后又尝试把它放在：

- 项目根目录
- 工具入口
- `SKILL.md`
- 项目规程

这个过程中暴露出的第一个问题是：

**`files-driven` 更容易把问题收成“解释清楚的真源文档”，而不是“运行时必须直接生效的规则”。**

用户连续指出了几次关键问题：

1. 工具入口是每一轮都携带的系统提示词，不应该依赖“先读某文档”
2. 入口文件不该承担阅读路由，应该直接承载最短稳定规则
3. `边界宪章` 如果只是告诉人“去看它”，那运行时依然会失效

这说明 `files-driven` 有明显的：

- 解释偏置
- 真源位置偏置
- 运行时经济性意识不足

### 4. 第二次暴露出的偏误：把边界原则当成静态说明，而不是动作规则

之后边界规则逐步被拆到了：

- 工具入口
- `SKILL.md`
- 项目规程

但用户继续指出更深一层的问题：

**即便边界原则讲清楚了，它仍然没有明确触发。**

换句话说：

- 规则说了很多“应该”
- 但没有说“什么动作会触发这条规则”
- 更没有说“一级处理在哪里、二级升级在哪里”

真正逼出这个问题的是一个高频治理动作：

**“这个文件放哪？”**

这不是抽象哲学问题，而是项目里最常出现的日常动作之一。
如果一条边界规则不能约束这个动作，它就不是治理规则，只是说明文。

于是才进一步补出了：

- `文件归属工作流`
- “先判归属，再落盘”
- “混写先拆，再继续”
- “一级文件归属工作流，二级边界宪章”

到这一步，问题已经非常明确：

**当前 `files-driven` 缺的不是知识点，而是把原则自动压成动作关口的能力。**

### 5. 第三次暴露出的偏误：八大家族的主轴不适合 skill-driven 项目

在继续回看整个过程时，用户进一步提出了更深的诊断：

**八大家族不是不对，但它不应该是 skill-driven AI-Native 项目的第一判断轴。**

用户的核心判断是：

- `Skill` 是主要治理对象
- `Skill` 是主要实践载体
- `agent / workflow / reference` 主要应从属于 `Skill`
- 业务侧主要应被理解为项目规则和项目实体
- 其余三类运行/投影家族，大多也应解释为项目实体的一部分

也就是说，旧模型的问题不是“家族划错”，而是：

**把结构观察轴误用了治理主轴。**

## 三、后果：为什么会出现反复人工指挥

从项目总监视角看，这次对话之所以需要用户反复推方向，
不是因为信息不足，而是因为 `files-driven` 在底层推理顺序上有偏差。

### 1. 它先问“属于哪一类”，没有先问“谁是能力主体”

`files-driven` 现有心智更像：

1. 先看八大家族
2. 再看四层
3. 再看真源和投影
4. 再给迁移建议

但在 skill 驱动项目里，真正更稳的顺序应该是：

1. 先问谁是能力主体
2. 再问什么是项目事实
3. 再问哪些规则必须运行时携带
4. 最后才问这些东西分别落在什么结构标签下

### 2. 它先想“放在哪一层”，没有先想“哪一层会先触发”

这导致很多建议理论上合理，但运行时不经济。
例如：

- 把规则放在独立长文里
- 让入口文件“先读这个，再读那个”
- 把长文当边界真源

这些在静态治理上说得通，但对每轮都会重新执行的 AI 运行时来说，
成本太高，也太不稳定。

### 3. 它会先给出完整框架，不会自动先补最小关口

这次对话里最典型的例子就是：

- 先有 `边界宪章`
- 后有入口规则
- 再后有 `文件归属工作流`
- 最后才补出一级/二级触发

而一个真正像项目总监的替身，
理论上应该在用户说出“作家能力 vs 小说内容”时，
很快就自动追问：

- 这个边界会在哪些动作上反复出错
- 哪些动作必须前置判定
- 判不清的时候谁来升级

### 4. 它把“讲清楚”当成了治理完成

这次纠偏反复证明：

**讲清楚不等于能执行。**

项目总监关心的是：

- 规则有没有触发点
- 触发点有没有动作顺序
- 动作顺序有没有停手条件
- 停手条件有没有升级口

如果没有，这条规则就还没落地。

## 四、根本诊断：旧 files-driven 的底层逻辑为什么不够用

这次重写不是对若干 reference 做局部修辞修复。
它要求重写的是底层逻辑。

### 1. 旧主轴

旧主轴大体可以概括为：

1. 先定起点
2. 先画四层
3. 再用八大家族解释结构
4. 再判漂移和恢复

这个主轴对以下项目很有效：

- 文档驱动仓库
- 多角色共享真源
- 有明显投影越权问题的项目
- 需要清真源和恢复链的仓库

但它默认的项目画像仍然是：

**“治理对象主要是文档系统本身”。**

### 2. skill-driven 项目的真实主语

在 skill 驱动的 AI-Native 项目里，
更符合现实的画像是：

- 工具入口承载运行时最低层规则
- `Skill` 承载能力、方法、agent、workflow、reference、schema
- 项目规则承载该项目特有的业务法则
- 项目实体承载事实、实例、候选稿、状态、展示

这里治理的主语不是“文档系统如何分层”，而是：

**能力如何被持有，项目如何被承载，运行时如何被约束。**

### 3. 因此，旧主轴会在四个地方持续失效

#### A. 在顶层分类上失效

因为它会把 `skill`、`agent`、`workflow` 并列摆出来，
弱化 `Skill` 作为能力主容器的地位。

#### B. 在入口设计上失效

因为它天然容易把入口文件写成“阅读路由说明”，
而不是“运行时最短规则”。

#### C. 在边界治理上失效

因为它会先产出一份边界说明，
而不是先产出一个边界触发流程。

#### D. 在高频动作治理上失效

因为它不够优先响应：

- 文件放哪
- 规则放哪
- 方法放哪
- 实体放哪
- 混写怎么拆

这些日常动作。

## 五、这次重写冻结的新底层逻辑

### 1. 新第一判断轴

以后 `files-driven` 在 skill-driven 项目里，第一判断轴不再是八大家族。

新的第一判断轴是：

1. `入口规则`
2. `能力规则`
3. `项目规则`
4. `项目实体`

### 2. 新的一句话原则

**具体能力归 Skill，项目内容归实体。**

这句话不是口号，而是以后所有判断的第一过滤器。

### 3. 对四个顶层对象的定义

#### A. 入口规则

定义：

- 每轮稳定携带
- 长度必须克制
- 直接作用于模型本轮行为

典型内容：

- 中文化底线
- 先判归属再落盘
- 派生层不得反写真源
- 工具入口不承载详细 workflow

#### B. 能力规则

定义：

- 某个替身能力本身
- 讨论“怎么工作、怎么判断、怎么验证、怎么迁移”
- 由 `Skill` 持有

它是新的治理重心。

典型内容：

- `SKILL.md`
- workflow
- agent
- references
- schema
- checklist
- 审计规则
- 文件归属规则
- 边界宪章

#### C. 项目规则

定义：

- 只对当前项目成立
- 虽然是规则，但不是通用能力

典型内容：

- 项目规程
- 作品宪法
- 世界规则
- 项目级命名/版本约束

#### D. 项目实体

定义：

- 记录项目现在是什么
- 包含事实、实例、状态和展示

典型内容：

- 实体档案
- 任务实例
- 章节实例
- 候选稿
- 运行实例
- 状态摘要
- 展示页

### 4. 八大家族的降级方式

八大家族不删除，但地位变化如下：

- 不再作为首屏入口
- 不再作为第一判断轴
- 保留为二级观察标签
- 只在需要细化结构责任时再启用

### 5. `Skill` 的重新定位

`Skill` 以后不再只是一个家族标签。

它应被写成：

**能力主容器。**

它内部再承载：

- agent：执行角色面
- workflow：控制路径面
- references：能力资产面
- scripts：执行支持面
- schema/checklist：规范与校验面

### 6. 项目实体的重新定位

`execution_object / status_projection / display_projection`
在观察上仍可存在，
但在治理落位上，优先都作为项目实体体系的一部分解释。

这样做的目的不是否认它们的分层，
而是避免它们与 `Skill` 争顶层主语。

## 六、从这次对话抽出来的项目管理思想、规则和技巧

### 1. 规则不是讲道理，是约束动作

任何规则都必须回答：

1. 什么动作触发
2. 触发后先做什么
3. 判不清时停在哪里
4. 什么时候升级
5. 升级到哪一层

不能回答这五件事的规则，不算落地。

### 2. 运行时可见性优先于静态归属正确性

某条规则理论上归属正确，不代表它运行时有用。

先问：

- 这条规则在哪一层会被模型稳定看到？
- 它会不会每轮都被携带？
- 它会不会增加不必要的阅读成本？

### 3. 工具入口只放最短稳定规则

入口不是长文导览页。

入口应该只放：

- 本轮必须立即生效的约束
- 不看就会直接滑路的底线

不应该放：

- 大段解释
- 先读清单
- 复杂 workflow
- 业务事实

### 4. 长文只做升级口

长文不是不能有。
但它只能处理：

- 二级争议
- 总原则冲突
- 治理重构
- 模型需要统一心智时的维护说明

不能让它承担：

- 日常高频动作的前线入口

### 5. 高频动作必须有一级流程

尤其是这些动作：

- 新建文件
- 迁移文件
- 拆分文件
- 合并文件
- 重命名文件
- 目录重构
- 方法抽离
- 业务事实回写

这些都应该先进入一级动作关口，
不能直接升级到总原则长文。

### 6. 混写视为治理缺陷

以下都不是“稍微有点乱”，而是必须拆分：

- 方法和项目事实写在同一文件
- schema 和实体实例写在同一文件
- A/B 协议和本次 A/B 结果写在同一文件
- 迁移规则和某次迁移执行记录写在同一文件

### 7. 两级触发优先于单一长文

以后凡是边界问题，优先设计成：

- 一级：处理日常动作
- 二级：处理总原则争议

这比一开始就让模型去读一篇总论，更稳。

### 8. 治理改动要做端到端验证

不只检查文案有没有问题，
还要检查：

`入口 -> Skill -> workflow/reference -> 项目落位 -> 上游真源`

这条链是否闭合。

## 七、当前 files-driven 作为“项目总监替身”暴露出的能力偏误

### 1. 结构偏置

过早把注意力放在结构家族，
而不是先判断：

- 这个项目里谁是能力主容器
- 谁是运行时第一入口
- 谁在承载业务事实

### 2. 解释偏置

更擅长产出一篇讲清楚的文档，
不够擅长把原则压成：

- 触发条件
- 动作顺序
- 停手条件
- 升级口

### 3. 真源位置偏置

更关注“应该归哪一层”，
不够关注“哪一层会在运行时被真正看到并执行”。

### 4. 对 Skill 主体性的判断不足

仍然把 `skill` 当成一个可并列的结构标签，
而不是把它视为 AI-Native 项目中的能力主系统。

### 5. 对高频摩擦场景不够敏感

像“这个文件放哪”这种问题，
理论上应该非常早就被识别成一级治理动作。
但旧逻辑更容易先回答结构学问题，
而不是先给动作化关口。

### 6. 缺少“原则 -> 动作关口 -> 升级口”的自动转换

旧逻辑能产出原则，但不会自动补：

- 触发词
- 一级处理流程
- 二级升级流程
- 谁来裁定

### 7. 运行时经济性意识不足

对 skill 驱动项目来说，
入口 token 成本、每轮重复携带成本、解释性路由成本都很高。

旧逻辑对这些成本还不够敏感。

### 8. 把“看清结构”误当成“已经完成治理”

治理完成的标准不是看懂了，
而是：

- 人和模型都会按这条规则做事
- 出现摩擦时知道先走哪条关口
- 争议时知道升级到哪里

## 八、审计目标：这次到底要清算什么

本轮不是平均审一遍整个仓库。
而是围绕底层逻辑重写，做定向清算。

### 1. 要清算的旧主轴

优先清算以下旧说法：

1. 八大家族是第一判断轴
2. `skill` 与 `agent / workflow` 是同级核心家族
3. `object` 比“项目实体”更适合做顶层治理主语
4. `policy_or_rules` 足以同时容纳入口规则、能力规则、项目规则
5. 静态真源归属优先于运行时触发位置

### 2. 要清算的旧文风

优先处理这些写法：

1. 解释性“先读哪里”
2. 首屏大谈家族，却不谈动作触发
3. 先讲结构，再讲关口
4. 先给全套理论，再给日常动作解法

### 3. 要清算的旧例子

重点检查示例是否在误教使用者：

1. workflow 是治理主角
2. agent 是并列顶层对象
3. 状态页和实例页是通用治理对象，而不是项目实体子层
4. 入口文件天然应该承担路由

## 九、审计范围

### A. 主入口层

必须审计：

- `SKILL.md`
- `README.md`
- `QUICKSTART.md`
- `MIGRATION.md`
- `agents/openai.yaml`

审计问题：

1. 是否仍把八大家族放在首屏心智位置
2. 是否仍把 `skill` 和 `agent / workflow` 并列解释
3. 是否仍把入口写成阅读路由
4. 是否已经把 Skill 写成能力主容器

### B. 核心 reference 层

必须审计：

- `references/结构家族定位约定.md`
- `references/基本原则.md`
- `references/场景手册.md`
- `references/输出约定.md`
- `references/治理模式选择对照表.md`
- `references/官方读取顺序.md`
- `references/关口硬化与稳定放行.md`

## 十、清算方法

### 1. 概念清算

先清理词义，再动文件。

具体做法：

1. 把“八大家族”降级成二级分析标签
2. 把“Skill 主容器”提升为首屏概念
3. 把 `policy_or_rules` 拆解成 `入口规则 + 项目规则`
4. 把 `object` 改写成“项目实体”主叙事

### 2. 文档清算

先清主入口，再清核心 reference。

理由：

- 首屏心智不改，后续重写会被拖回旧逻辑
- 主 `SKILL` 不改，其他文档改再多也会失真

### 3. 示例清算

示例只要还在教旧逻辑，
主文档改完之后也会被反向污染。

因此：

- 例子必须和新主轴一致
- 测试必须改成新主轴的回归断言

### 4. 测试清算

不在开头就大动测试，
但必须在第一批主文重写后立刻补回归。

否则旧测试会把新逻辑打回去。
