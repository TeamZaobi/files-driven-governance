# 治理能力模型 v1 下一阶段执行计划

## 1. 目标

这一轮不是重写方法论，
也不是把鲁棒性要求一次性拉满。

目标只有两件事：

1. 把已经冻结的概念边界推进成可执行的最小技术强制层
2. 为后续试点迁移补齐真正阻塞的前置项

## 1.1 当前状态（第二段 P0 后）

已完成：

1. `workflow.state.schema.json` 与 `workflow.event.schema.json`
2. `checks` 协议收口到 `route / evidence / write / stop`
3. `gate_state` 的 canonical 三态冻结为 `blocked / partial / ready`
4. `approver_ref / approval_ref` 的最小语义冻结
5. validator 从“未冻结警告”推进到最小语义校验

当前剩余的 P0 只保留：

1. `status_projection` 的最小技术限制

## 2. 本轮采纳哪些审计结论

采纳：

1. `execution_object` 结构缺失，需要补实例 schema
2. `checks` 协议与 `route / evidence / write / stop` gate 需要收口
3. 需要补引用完整性校验
4. 需要补 Markdown 与 contract 的一致性校验
5. 开放问题、阶段边界和试点标准需要更明确

暂不升格为本轮阻塞：

1. hostile-environment 级别的 `signature`
2. 细粒度访问控制
3. 完整沙箱化
4. 面向生产部署的对抗性防护

这些属于后续高信任环境之外的强化项，
不应反向拖住 v1 的最小实现。

## 3. 执行原则

1. 先补实例与校验，再补示例资产
2. 先冻结最小协议，再做可选外提
3. 能留在现有结构里的，不新开家族
4. 能通过派生校验解决的，不反塞回上游 workflow 合同

## 4. P0：已完成基座与剩余阻塞

### P0-1. `execution_object` 最小实例 schema

补两份 schema：

1. `workflow.state.schema.json`
2. `workflow.event.schema.json`

目标：

1. 冻结 `workflow.state.json` 的最小字段
2. 冻结 `workflow.events.jsonl` 中单条 event 的最小字段
3. 为审计追溯补最小字段，但不提前引入重安全机制

当前建议至少覆盖：

1. `actor_id`
2. `timestamp`
3. `event_type`
4. `workflow_id`
5. `contract_version` 或等价版本键

### P0-2. `checks` 协议收口

把 workflow schema 中的 `checks` 从：

1. `entry`
2. `transition`
3. `stop`

收口到与治理流程一致的：

1. `route`
2. `evidence`
3. `write`
4. `stop`

同时冻结一条权限规则：

- check 只返回检查结果、证据缺口或阻断信号
- check 不单独拥有放行权
- 合法转移仍由 workflow contract + rules contract 决定

最小返回语义默认按：

- `check_ref`
- `gate`
- `result`
- `blockers`
- `missing_evidence`

其中：

- `route` 可补 `resolved_action_class / route_conflicts`
- `write` 可补 `forbidden_output_refs`
- `stop` 可补 `remaining_violations`

### P0-3. 引用完整性与一致性校验

补最小验证机制：

1. contract 间引用完整性校验
2. `workflow state` 对 workflow contract 的实例一致性校验
3. Markdown 与 contract 的最小一致性校验

这轮先保证：

1. 引用不悬空
2. 实例不引用不存在的节点、状态、转移
3. `WORKFLOW.md` 不反向定义与 contract 冲突的状态和关口

当前 validator 优先只判四类错误：

1. `missing target`
2. `kind mismatch`
3. `invalid gate_state`
4. `Markdown introduced forbidden token`

### P0-4. 阶段与开放问题重排

把开放问题分配到阶段，
避免“阶段已进入、问题仍悬空”的倒挂状态。

当前仍留在 P0 的剩余问题只有：

1. `status_projection` 的最小技术限制

## 5. P1：紧随其后

### P1-1. `entry_intents` 是否外提

当前先保持：

- `entry_intents` 内嵌在 `workflow.contract.json`

只有在这两种条件出现时，
才外提 `entry_intents.json` 并单独补 schema：

1. 一个 workflow 需要多个适配层共享同一套意图映射
2. 意图映射的变更频率显著高于 workflow 转移本身

### P1-2. 试点选择标准

最小量化维度按 `0-2` 分桶求和，
不采用纯乘法：

1. 风险分：修改频率、下游影响、历史故障
2. 共享分：参与主体数、工具入口数、交接频率
3. 迁移易度分：待补资产数、参与人数、回退成本

### P1-3. smoke asset 与示例资产

在 P0 完成后，先补一个极小 smoke asset pack，
跑通 validator 的端到端链路；
再扩成最小 project-level 示例资产，
验证：

1. rules -> workflow -> state/event 的引用链
2. checks 协议是否足够表达 gate
3. 状态投影是否还能偷偷越权

高控制试点：

- `风险 >= 4`
- `共享 >= 4`
- `迁移易度 >= 2`

轻量对照试点：

- `风险 <= 2`
- `共享 <= 2`
- `迁移易度 >= 4`
