# 语言体系规范

本文件用于统一 `files-driven` 项目的中文表达，目标是两件事：

1. 让首次接触这个项目的人能直接读懂
2. 让后续文档新增内容不再回到中英混写和工程黑话

## 1. 总原则

### 中文优先

面向人的说明默认全部用中文。
只有在以下情况才保留英文：

- 工具、产品或组织的正式名称
- 文件名、路径、命令、环境变量
- 稳定键名、枚举值、代码标识符
- 已被广泛接受且没有更短中文说法的外部方法名

### 先说人话，再给稳定键名

先写中文概念，再在需要精确对齐时补上稳定键名。

推荐写法：

- 规则与约束层（`policy_or_rules`）
- 状态摘要层（`status_projection`）
- 意图触发约定（`intent-trigger-contract`）

不推荐写法：

- 直接把 `policy_or_rules / status_projection / current-version anchor` 堆给读者
- 一句话里混用多个英文原词而不给中文解释

### 一个概念只保留一个主叫法

不要在不同文档里来回切换：

- `canonical source` / `truth source` / `source of truth`
- `owner` / `authority` / `direct writer`
- `projection` / `adapter surface` / `view`

本项目统一使用一套主叫法，必要时再在第一次出现时补充英文键名。

### 尽量降低抽象门槛

优先用“谁写、谁读、先看什么、出问题怎么回退”这种能直接落地的话。
少用只对工程背景读者友好的压缩表达。

## 3. 固定写法

### 结构家族的写法

第一次出现时：

- 规则与约束层（`policy_or_rules`）
- 对象层（`object`）
- 流程层（`workflow`）
- 技能层（`skill`）
- 角色层（`agent`）
- 过程载体层（`execution_object`）
- 状态摘要层（`status_projection`）
- 展示输出层（`display_projection`）

后文已经没有歧义时，可以只写中文。

### 方法名称的写法

外部方法名第一次出现时用：

- 规格驱动（`Spec-Driven`）
- 看板流（`Kanban`）
- 迭代节奏（`Agile / Sprint-like`）

如果后文重点不是方法论比较，可以只写中文。

### 口令与稳定键名的写法

面向操作人的命令口令可以写中文：

- 继续开发
- 开始审计
- 反思
- 推进

面向系统和文档约束的稳定键名仍保留在反引号中：

- `direct_action_intent`
- `route_intent`
- `sync_pending`

## 4. 推荐句式

推荐：

- 先确认这次到底交付什么，再谈目录和流程。
- 如果多个工具都会碰同一份事实，就先补跨层共享约定。
- 状态摘要页只能总结真源，不能反过来改写真源。

不推荐：

- 先做整体 governance blueprint，再看 source family 和 adapter surface。
- 用 tool bootstrap 承接 canonical workflow definition。
- 直接扩展 multi-layer sync contract 而不先判断 current-version anchor。

## 5. 保留英文的边界

以下内容允许保留英文：

- `SKILL.md`、`README.md`、`agents/openai.yaml`
- `files-driven`
- `OpenClaw`、`Codex`、`Claude Code`、`AntiGravity`
- `MIT`、`Apache-2.0`
- 稳定流程键名、命令、目录、文件路径

但即使保留英文，也应同时给中文解释，不要让英文单独承担语义。

## 6. 维护规则

新增文档或修改现有文档时，默认执行以下检查：

1. 标题是否优先用了中文
2. 首次出现的核心概念是否给了中文主叫法
3. 是否把英文原词当成了默认解释
4. 是否出现多个英文同义词指向同一概念
5. 是否能让不了解本项目的人直接读懂

如果某个新术语确实需要进入项目，请先在本文件补一条主叫法，再扩散到其他文档。
