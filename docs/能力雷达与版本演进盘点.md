# 能力雷达与版本演进盘点

状态：`working audit`

这份文档不是版本说明，也不是能力模型真源。
它的用途只有两个：

1. 把 `files-driven` 当前已经具备的能力簇系统筛出来
2. 把过去各版本到底新增了哪些能力，按统一口径盘清

底层本体仍只认 [项目治理能力模型](./项目治理能力模型.md)。

## 1. 先把两条版本轴分开

这套项目至少同时存在两条版本轴：

| 版本轴 | 当前主叫法 | 作用 | 不该混成什么 |
| --- | --- | --- | --- |
| 能力模型轴 | `v1 -> v2 -> v2.1` | 回答治理世界观、主语、作用域和防变形怎么演进 | 不该和 pack / contract 冻结语义混写 |
| 发布版本轴 | `v0.1.0 -> v0.5.0` | 回答仓库在什么时间把哪些能力做成了公开资产 | 不该拿来代替能力模型判断 |
| governed pack / contract 轴 | `tranche v1` | 回答当前 workflow / harness 最小合同链冻结到了哪里 | 不该被误读成世界观还停在 `v1` |

一句话记法：

- `v1 -> v2 -> v2.1` 是模型怎么长出来
- `v0.x.y` 是仓库什么时候把能力做成公开资产
- `tranche v1` 是 governed pack / contract 当前冻结到哪

## 2. 雷达评分口径

这张雷达不用主观印象打分，统一按资产化程度记：

| 分值 | 含义 |
| --- | --- |
| `0` | 还没有正式提出来 |
| `1` | 只有零散讨论或隐含说法 |
| `2` | 已有正式 reference / 文档，但没有稳定入口 |
| `3` | 已有入口、reference 或 example，能被说明，但执行面还弱 |
| `4` | 已有正式入口 + example / script / 测试中的至少两项 |
| `5` | 已有真源定义、入口暴露、脚本或合同、示例或测试，且边界已被系统回归守住 |

## 6. 过去版本筛下来，能力是怎么一层层长出来的

### 6.1 `v0.1.0 -> v0.2.3`

这一阶段主要在补“能不能讲清、能不能路由、能不能精确输出”：

1. 先有基础治理 reference
2. 再补流程库、质询、读取顺序、工具适配
3. 再补把握度和澄清提问
4. 最后把默认输出压成“核心必答 + 条件展开”

这一段更像 **认知骨架期**。

### 6.2 `v0.2.4 -> v0.2.7`

这一阶段主要在补“起步如何不漂”：

1. 文档生命周期和压缩
2. 方向与边界锚点
3. 说人话需求确认工具包
4. 短口令和意图触发约定

这一段更像 **边界前置期**。

### 6.3 `v0.3.0 -> v0.3.1`

这一阶段主要在补“方法如何进入最小可执行状态”：

1. `项目治理能力模型 v1`
2. schema、validator、smoke pack、CI
3. `BOUNDARY.md`、仓库故事/测试、AI-Native 同构协作前提

这一段更像 **合同化与验证期**。

### 6.4 `v0.4.0`

这一版是明显的拐点。
它把前面的认知骨架、边界前置、最小合同链，第一次合成了 **元 skill + starter + 动作面**：

1. 统一真源
2. files engine starter
3. `install / register / repair / audit`
4. 元 skill 的 scaffold 闭环

### 6.5 `v0.4.1 -> v0.5.0`

这一段主要在纠偏和补控制：

1. 纠正“脚本化至上”的误读，回到“强化控制能力”
2. 把控制强度轴、执行面轴、宿主优先原则讲清
3. 把 `runtime -> candidate -> capability` 正式做成 reference / example / regression
4. 把 self-hosting workflow 改造成可受控 runner
5. 把采用入口翻回“体检、诊断、治疗、复查、随访”

这一段更像 **控制强化与产品收口期**。

## 7. 当前还没彻底长实的角落能力

下面这些不是“不存在”，而是“已经有模型或 reference，但还没完全变成强入口或强执行面”。

1. `宿主化知识工作场景`
   - 例如 `Obsidian / Notion / Docs / Sheets / Slides`
   - 当前模型能解释，且已补宿主化 fixture，但还缺更系统的 host-native runnable operations、反误触发边界和宿主操作回放
   - 这一块的专项补强，见 [宿主化知识工作场景矩阵](./宿主化知识工作场景矩阵.md)

2. `audit` 的分层体检
   - 当前 `audit` 默认是 scaffold 基础体检，`pack` 已有 `manage audit --layer pack` draft 入口
   - `runtime` 已有 `manage audit --layer runtime` draft checker，`governance` 和 `adoption` 也都已有 draft checker
   - `runtime / governance / adoption` 仍是限定问题面的 draft checker，其中 `adoption` 当前只覆盖本仓库入口
   - 当前边界和补完顺序，见 [体检分层矩阵](./体检分层矩阵.md)

3. `记忆链与能力晋升硬校验`
   - `runtime -> candidate -> capability` 已有 official path
   - `runtime` draft checker 已开始查 `recall_note.md / split_decision.md / workflow.events.jsonl`
   - 但 validator 对 `reason_refs / artifact_refs / recall chain` 的硬约束还不够深

4. `工具适配面的负边界`
   - 工具适配、benchmark anchors、projection pages 仍然最容易偷长成 hidden authority surface
   - 这在 self-hosting capability graph 里已经被明确标成主要 drift risk

5. `采用入口的一致触发`
   - 文档入口已经明显改善
   - 但“用户一开口就是宿主名或页面名”时，是否还能自然走到治理问题，而不是被误判成纯工具操作，仍需继续强化

6. `外部 workflow 控制面合同化 benchmark`
   - 当前已经明确吸收了 `human authority layer / machine-readable control plane / hook policy` 的分层变量
   - 也已明确最小合同面可落到 `workflow-control-plane / model-routing / hook-policy / state / events`
   - 但这块仍是外部项目改造的个案 benchmark，不是本仓当前正式 starter 或统一 CLI 子命令
   - 只有跨项目、跨宿主、跨品牌路由反复验证后，才值得考虑晋升

## 8. 这张雷达怎么用

如果你接下来要做系统盘点，建议按下面顺序使用：

1. 先看 [项目治理能力模型](./项目治理能力模型.md)，确认你在谈模型轴还是发布轴
2. 再用本页的“当前能力雷达”确认当前已有哪些能力簇，以及哪些还只是中等成熟
3. 再用“发布版本轴”看某项能力是什么时候收编成仓库资产的
4. 如果要反向检查某个能力今天到底照进了哪些入口、脚本和回归，直接读 [能力覆盖矩阵与历史差分](./能力覆盖矩阵与历史差分.md)
5. 最后用“当前还没彻底长实的角落能力”决定下一 tranche 该补哪一块

如果你接下来要做“能力覆盖审计”，最值得直接检查的是：

1. 某项能力是否同时具备：真源定义、入口暴露、reference/example、脚本或回归
2. 某项能力是否只停在文档里，还没进入动作面
3. 某项能力是否已经有脚本，却还没有用户能理解的触发入口
