# 项目治理能力模型 v1

## 1. 目标

这份设计文档描述的不是 `files-driven` 仓库自己的排版整理，
而是一套可迁移的项目治理设计能力。

当前重构目标是把项目治理从“主要依赖 prose 和聊天习惯”，
推进成“以文件承载的解释资产 + 合同资产 + 实例资产 + 检查执行面”。

`files-driven` 在这里承担的是元技能和交付器角色，
不是唯一使用对象。

## 2. 这次重构不做什么

本次重构明确不做以下事情：

1. 不新增第九类结构家族
2. 不把所有材料都改成 JSON
3. 不把 `files-driven` 退化成只谈 `workflow` 的工具
4. 不把所有项目默认推入重治理
5. 不把这次工作缩成“补几篇 reference”

## 3. 设计边界

本次重构默认保持以下边界不变：

1. 不新增第九类结构家族
2. 继续沿用系统、信息、控制三视角
3. 继续沿用八类结构家族
4. 不要求所有项目默认进入高控制强度
5. 不追求全仓库 JSON 化

本次真正改变的是：

1. 高风险控制语义不再主要寄居在 Markdown prose 中
2. `workflow` 的实现机制要支持合同化和实例化
3. 其他家族要为 `workflow` 提供可引用的结构化接口

### 3.1 四层分层与四类实现资产的关系

这里同时出现两套表达，但它们不是竞争关系：

1. `truth_source / execution_object / status_projection / display_projection`
   负责回答“这份材料在治理分层里处于什么位置”
2. 解释资产 / 合同资产 / 实例资产 / 检查执行面
   负责回答“受控 workflow 是怎么实现出来的”

两者是正交关系，不是替代关系。
默认按下面理解：

1. 解释资产通常落在 `truth_source`，也可能被压缩后投影到 `status_projection / display_projection`
2. 合同资产落在 `truth_source`
3. 实例资产落在 `execution_object`
4. 检查执行面属于 `skill` 或工具适配执行面，不单独占一个文档分层

所以，“四层”讲治理位置；
“四类实现资产”讲受控 workflow 的实现形态。

## 4. 六项设计能力

### 4.1 边界设计能力

回答：

1. 项目服务谁
2. 首批交付物是什么
3. 什么算达标
4. 什么明确不做

核心产物：

- 边界锚点
- 用户故事
- 测试锚点

### 4.2 真源设计能力

回答：

1. 哪些是真源
2. 哪些是投影
3. 当前版本去哪里找
4. 新接手的人先读什么

核心产物：

- 家族定位
- 版本锚点
- 读取顺序

### 4.3 控制设计能力

回答：

1. 状态是什么
2. 转移是否合法
3. 缺什么证据不能升级
4. 什么输出当前禁止
5. 什么时候必须回退

核心产物：

- `rules contract`
- `object schema`
- `workflow contract`

### 4.4 执行设计能力

回答：

1. 谁执行
2. 谁复核
3. 谁能批准
4. 技能如何受 workflow 约束

核心产物：

- `agent contract`
- `skill manifest`
- 运行入口

### 4.5 观测设计能力

回答：

1. 当前状态如何被看见
2. 哪些是实例
3. 哪些只是状态摘要
4. 哪些地方绝不能生成新控制结论

核心产物：

- `workflow state`
- `events`
- `status projection`

### 4.6 恢复设计能力

回答：

1. 如何交接
2. 如何回退
3. 如何在不停工时恢复秩序

核心产物：

- 回退路径
- 恢复链
- handoff 包

## 5. 八类结构家族在能力模型中的职责

### 5.1 `policy_or_rules`

负责稳定规则、阈值、批准条件、回退条件。
只有需要被自动检查的规则部分才进入结构化合同。

### 5.2 `object`

负责状态、证据、输出对象的语义。
`workflow` 不应私造对象字段或状态枚举。

### 5.3 `workflow`

负责控制路径和合法转移。
它是控制论主载体，但不是唯一语义真源。
它应让出：

1. 状态词典
2. 证据对象定义
3. 输出对象定义
4. 批准权限定义
5. 运行实例本身

### 5.4 `agent`

负责执行、复核、批准、回退权限边界。
当权限需要被合同消费时，应暴露结构化接口。

### 5.5 `skill`

负责执行方法和步骤包。
它可以实现 workflow，但不能单独声明 workflow 的放行权。
当 workflow 引入检查器时，
检查器实现属于 `skill` 或工具适配执行面，
不等于 workflow 自己拥有检查真源。

### 5.6 `execution_object`

负责当前运行实例和运行轨迹。
例如：

- `workflow.state.json`
- `workflow.events.jsonl`
- `decision_package`

这些资产可以与 workflow 共置，
但家族归属仍是 `execution_object`，不是 `workflow`。

### 5.7 `status_projection`

负责当前状态摘要与恢复入口。
它必须从合同和实例推导，而不能私自产生放行结论。

### 5.8 `display_projection`

负责展示和汇报。
它的控制权最低，不得定义新状态或新转移。

## 6. Workflow 的实现机制重构

`workflow` 家族不新增结构子类。
真正的变化是：它的 metadata 与合同设计必须支持不同控制强度。

### 6.1 轻量 workflow

适用于：

1. 低风险
2. 单人
3. 无高代价下游动作
4. 回退便宜

允许形态：

1. `WORKFLOW.md`
2. 最小 frontmatter 或 manifest

### 6.2 强控制 workflow

适用于：

1. 高风险
2. 共享执行
3. 直接通向修改、发布、正式放行
4. 已发生 route confusion 或越级动作

最小资产：

1. `WORKFLOW.md`
2. `workflow.contract.json`

与 workflow 共置、但不属于 workflow 家族的最小执行面通常还包括：

1. `workflow.state.json`
2. `workflow.events.jsonl`

`checks/` 常与 workflow 共置，
但它属于 `skill` 或工具适配执行面，
不属于 workflow 家族本身。

### 6.3 不是类型分叉，而是实现强度分层

同一家族，不同合同化程度。
决定条件不靠名字，而靠 metadata：

1. 风险
2. 自主度
3. 共享度
4. 回退成本
5. 下游输出代价
6. 既往失效历史

## 7. 合同化实现原则

### 7.1 合同不是普通 metadata

只要一段 metadata 已经决定：

1. 状态
2. 合法转移
3. 证据要求
4. 输出限制
5. 回退条件

它就不再只是“标签”，而是执行合同。

### 7.2 人读层与机读层分离

对高风险控制语义，建议采用：

1. `Markdown` 负责解释层
2. `JSON` 负责合同层与状态层
3. `JSONL` 负责事件流
4. 脚本、hook、validator 负责执行检查

### 7.3 同一家族支持不同控制强度

不新增新的结构家族类型。
同一 `workflow` 家族内部，通过合同和 metadata 表达不同治理强度。

简单说：

- 轻量 workflow：解释层为主
- 受控 workflow：至少需要解释资产 + 合同资产，并与实例资产、检查执行面对齐

## 8. Markdown / JSON / JSONL 的分工

### 8.1 Markdown

负责：

1. 解释
2. 阅读入口
3. 边界说明
4. 复核说明
5. 例外和背景

### 8.2 JSON

负责：

1. 合同
2. 状态
3. 转移
4. 权限
5. 输出约束

### 8.3 JSONL

负责：

1. 事件流
2. 运行轨迹
3. 阶段记录

## 9. 合同之间的最小引用关系

默认按这个方向设计：

1. `workflow contract` 引用 `object schema`
   - 用于 `state_ref / action_ref / evidence_refs / approval_ref`
2. `workflow contract` 引用 `rules contract`
   - 用于 `guard_policy_refs / output_policy_refs`
3. `workflow contract` 引用 `agent contract`
   - 用于 `agent_refs / approver_ref` 以及批准与执行边界的角色键
4. `workflow state` 作为实例，只引用合同中的节点和状态键
5. `status_projection` 只从合同与实例派生，不反向改写合同

其中最小冻结边界是：

1. `node.approver_ref`
   - 指向 `agent.contract.json` 中 `roles[].role_id`
   - 表示当前节点由谁复核或批准，属于“谁能批”
2. `transition.approval_ref`
   - 指向 `object` 家族中 `kind = approval_type` 的对象
   - 表示这次转移需要什么批准对象或批准凭据，属于“批什么”
   - 它应由相邻节点的 `approver_ref` 通过 `role.can_approve_refs` 覆盖
3. `workflow.state.json` 与 `workflow.events.jsonl` 中的 `gate_state`
   - 最小枚举固定为 `blocked / partial / ready`
   - 运行中、完成、回退等生命周期语义由节点、事件或后续专门字段表达，不并入 gate
4. `status.projection.json`
   - 只允许携带派生字段，例如 `gate_state / current_node_id / missing_evidence_refs / forbidden_output_refs`
   - 不得持有 `allowed_next_step_refs / approver_ref / approval_ref / guard_policy_refs / output_policy_refs`

## 10. 最小资产集合

对进入高控制强度的项目，最小集合建议为：

1. `RULES.md`
2. `rules.contract.json`
3. `OBJECTS.md`
4. `schemas/*.json`
5. `WORKFLOW.md`
6. `workflow.contract.json`
7. `workflow.state.json`
8. `workflow.events.jsonl`
9. 可选：`status.projection.json`
10. `AGENT.md`
11. `agent.contract.json`
12. `SKILL.md`
13. `skill.manifest.json`

不是每个项目都要一次性补齐。
只有当 workflow 已经承担高风险控制语义时，
这套集合才需要逐步启用。

另可与 workflow 共置 `checks/`，
但其家族归属仍在 `skill` 或工具适配执行面。

## 11. 文件存在结构的重构方向

### 11.1 `workflow`

建议形态：

1. `WORKFLOW.md`
2. `workflow.contract.json`
3. 可选：`entry_intents.json`

与 workflow 共置、但归其他家族的资产：

1. `workflow.state.json`
2. `workflow.events.jsonl`
3. `checks/check_workflow.*`

### 11.2 `object`

建议形态：

1. `OBJECTS.md`
2. `schemas/*.json`

### 11.3 `policy_or_rules`

建议形态：

1. `RULES.md`
2. `rules.contract.json`（仅对可执行规则）

### 11.4 `agent`

建议形态：

1. `AGENT.md`
2. `agent.contract.json`

### 11.5 `skill`

建议形态：

1. `SKILL.md`
2. `skill.manifest.json`

### 11.6 `status_projection`

如果项目需要结构化的当前状态入口，
建议最小落成：

1. `status.projection.json`

它最小只应镜像：

1. `current_node_id`
2. `gate_state`
3. `missing_evidence_refs`
4. `forbidden_output_refs`
5. `summary`

它不应自己持有：

1. `allowed_next_step_refs`
2. `approval_ref`
3. `approver_ref`
4. `guard_policy_refs`
5. 新造的 `next_step`
6. 任何新的放行结论

## 12. 读取顺序与工具适配的变化

对轻量结构仍可 prose-first。
对强控制 workflow，默认应改成：

1. 索引或注册表
2. 合同真源
3. 解释真源
4. 最近批准的变更
5. 当前实例
6. 状态摘要
7. 工具适配入口

工具适配层不再只包装 Markdown 真源。
它应：

1. 读取合同
2. 读取实例
3. 执行检查器
4. 生成适合工具的摘要

它仍然不应：

1. 私造状态
2. 私造转移
3. 私造规则

## 13. 验证体系

这次重构至少要补以下回归：

1. `route confusion`
2. `partial / blocked` 下的越级动作
3. `next_step` 非法升级
4. 投影层偷渡放行
5. 工具适配分叉

P0 的最小技术强制还包括：

1. contract 间引用完整性校验
2. `workflow.state.json` 对 workflow contract 的实例一致性校验
3. `WORKFLOW.md` 与 contract 的受控词一致性校验
4. `gate_state` 的三态枚举冻结
5. `approver_ref / approval_ref` 的最小语义冻结
6. `status.projection.json` 只能派生、不准越权

## 14. 执行顺序

### 阶段 0：术语冻结

先冻结：

1. 能力模型主叫法
2. 家族职责边界
3. 合同与解释资产的区分
4. project asset 名称与 schema file 名称的映射

### 阶段 1：合同设计

先设计四个最小合同，并补两份最小实例 schema：

1. `workflow.contract.json`
2. `object schema`
3. `rules contract`
4. `agent contract`
5. `workflow.state.schema.json`
6. `workflow.event.schema.json`

### 阶段 2：Workflow 实现机制重构

把 `workflow` 从单文件说明升级成：

1. 解释资产
2. 合同资产
3. 入口意图绑定

这一阶段同时完成两件事：

1. 把 `checks` 协议从 `entry/transition/stop` 收口到 `route/evidence/write/stop`
2. 冻结“check 只返回观测结果，不授予放行权”的最小执行边界

运行实例与检查执行面在后续阶段分别与 `execution_object`、`skill` 对齐。

### 阶段 3：家族接口对齐

让 `policy`、`object`、`agent`、`skill`
都能被 `workflow` 合同稳定引用。

这一阶段至少补三类 validator：

1. contract ref 校验
2. instance 对 contract 校验
3. Markdown 与 contract 的受控词一致性校验

### 阶段 4：投影层收口

重写 `status_projection` 与 `display_projection` 的纪律，
使其只能从上游实例和合同推导。

### 阶段 5：试点迁移

选取高风险共享项目和低风险轻量项目做对照迁移。

### 阶段 6：回写 files-driven

把能力模型、合同层、迁移模式和验证方法
回写到 `files-driven` 自身。

3. 现有项目的迁移成本怎么评估
4. `entry_intents` 是否需要从 workflow contract 外提

## 16. 成功标准

满足以下条件时，可认为本轮重构成立：

1. 输出不再只停在 Markdown 文件树建议
2. 高风险 workflow 能落出结构化合同
3. `status_projection` 不能再隐性放行
4. 工具适配不能再重写 workflow 语义
5. 试点项目能证明越级动作和 route confusion 显著下降
