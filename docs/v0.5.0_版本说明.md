# v0.5.0 版本说明

发布日期：`2026-04-14`

## 本次版本做了什么

`v0.5.0` 不再继续扩一个更大的世界观，
而是把 `v0.4.0` 之后已经进入主线的几件事正式收口成一个新的小版本：

1. 把当前 tranche 的完成状态、缺口和下一步顺序正式版本化
2. 把“系统诊断后的推进方案”推进成一个可受控执行的 repo treatment rollout runner
3. 把用户侧入口语言压回可理解的 `体检 -> 诊断 -> 治疗 -> 复查 -> 随访`
4. 把 `audit` 的当前能力边界说实：先承认它是“基础体检”，不是全量系统体检

## 主要变化

1. 新增 [当前阶段补完计划](./当前阶段补完计划.md)，把 `runtime -> candidate -> capability` 接入后的当前完成状态、剩余缺口、验收线和非目标显式收成仓库资产。
2. 新增 [运行观察与能力晋升](../references/运行观察与能力晋升.md)、[capture-candidate-activation](../examples/capture-candidate-activation/README.md) 与 [tests/test_capture_promotion_assets.py](../tests/test_capture_promotion_assets.py)，把自迭代能力链冻结成 official reference、official example 和专项回归入口。
3. 新增 [scripts/run_repo_treatment_rollout.py](../scripts/run_repo_treatment_rollout.py) 与 [tests/test_repo_treatment_rollout_runner.py](../tests/test_repo_treatment_rollout_runner.py)，让维护者在 `self-hosting` 场景里可以把既定推进方案编码成受控 runner：
   - 生成 governed run pack
   - 调度 `codex exec`
   - 由脚本维护 `workflow.state.json / workflow.events.jsonl / status.projection.json`
4. 调整 [README.md](../README.md)、[docs/非工程背景起步.md](./非工程背景起步.md)、[docs/使用手册.md](./使用手册.md) 与 [agents/openai.yaml](../agents/openai.yaml)，把采用入口翻译成“体检、诊断、治疗、复查、随访”的用户侧语言，同时保持底层 canonical 术语不变。
5. 调整 [README.md](../README.md) 与 [docs/使用手册.md](./使用手册.md)，明确 `audit` 当前默认是脚手架基础体检：
   - 先检查 scaffold、registry、route、starter profile 这条资产链是否闭环
   - 不再对外暗示自己已经是对下游项目的全量系统体检
6. 调整 [README.md](../README.md)、[docs/完整说明书.md](./完整说明书.md)、[docs/仓库元数据建议.md](./仓库元数据建议.md) 与 [agents/openai.yaml](../agents/openai.yaml)，把当前公开版本锚点、发布元数据和 agent-facing 显示面正式收口到 `v0.5.0 / files-driven`。

## 为什么这次版本重要

因为到 `v0.4.0` 为止，
仓库已经能把 `files-driven` 解释成一个带 starter、registry、route、validator 和动作入口的元 skill，
但还有三类问题没有正式收口：

- 采用者入口还是偏工程内部语言
- `audit` 的产品表述比当前实现更大
- “诊断完以后怎么一次性推进”仍然停在讨论层，而不是受控执行层

`v0.5.0` 的意义，就是把这三类问题从“知道该怎么做”，推进成“有正式发布锚点、有公开文档、有 runner 可执行”。

## 这一版不代表什么

`v0.5.0` 不是 major，也不是本体重写。
它没有推翻 `v1 -> v2 -> v2.1` 的底层能力模型，也没有把 `audit` 一次性升级成平台级全量体检器。

更准确地说，这一版做的是：

- 让当前能力承诺与实现边界重新对齐
- 让 self-hosting 推进有一条正式、受控、可回放的执行线
- 让首次采用者的入口更像门诊接诊，而不是专科病历
