# 文档驱动治理完整说明书

当前版本：`v0.3.0`

## 1. 这是什么

`files-driven` 是一个项目结构治理技能。
它解决的不是“目录怎么摆更整齐”，而是“项目中的事实、过程、状态和展示应该分别落在哪些文档里，并且如何在多人、多代理、多工具环境下保持不漂移”。

这个技能把文档当成治理载体，用来明确：

1. 哪些文件是真源
2. 哪些文件只是过程载体
3. 哪些文件只是状态摘要
4. 哪些文件只是对外展示
5. 谁能写、谁能读、谁负责复核、谁有权回退

## 2. 它主要解决什么问题

这个技能主要应对以下问题：

1. `README`、状态页、任务记录、规则文档和流程文档互相漂移
2. 工具入口被误当成真源
3. 规则、对象、流程、技能、角色和状态页混在一起
4. 多人或多代理协作时，不清楚谁能直接改什么
5. 想借鉴规格驱动、看板流或迭代节奏，但不想把流程做得过重
6. 项目边界还没稳，却太快开始谈结构和目录
7. 活跃文档越来越多，接手成本越来越高
8. 团队想用“继续开发”“开始审计”“推进”这种短口令驱动工作，但这些词在不同工具里含义不稳定

## 3. 本项目统一使用的语言体系

这一版开始，本项目采用“中文优先、必要时补稳定键名”的写法。

统一规则见 [语言体系规范](./语言体系规范.md)。
最重要的几条是：

1. 面向人的说明默认全部用中文
2. 核心概念先写中文，再在首次出现时补稳定键名
3. 同一概念只保留一个主叫法
4. 英文只保留给文件名、工具名、命令、键名和外部正式名称

本说明书中常见概念统一如下：

- 真源：真正定义事实的上游文档
- 结构家族：规则、对象、流程、技能、角色等类别
- 工具适配入口：工具侧的启动入口、包装面或映射面
- 跨层共享约定：谁写、谁读、怎么同步
- 意图触发约定：短口令如何稳定落地
- 运行态叠加模式：持续运行项目在基础治理模式上的叠加治理
- 开发回路：围绕规格、实现、验证和发布前关口的控制回路
- 运营回路：围绕运行、观察、补救、反馈和恢复的控制回路

## 4. 核心结构模型

### 4.1 八类结构家族

本技能默认先把项目中的关键结构分成八类：

1. 规则与约束层（`policy_or_rules`）
2. 对象层（`object`）
3. 流程层（`workflow`）
4. 技能层（`skill`）
5. 角色层（`agent`）
6. 过程载体层（`execution_object`）
7. 状态摘要层（`status_projection`）
8. 展示输出层（`display_projection`）

这个分类先于目录规划。
目录只是承载方式，不应反过来决定概念边界。

### 4.2 先做方向与边界锚点

在谈结构家族、角色边界或工具适配前，先确认方向与边界。
至少要明确：

1. 首批真实使用场景
2. 首批交付物
3. 一到三个核心用户故事
4. 三到八个验收或测试用例
5. 非目标
6. 仅用于校准质量的参考对象
7. 谁来判断这次算不算达标

如果边界还在漂移，先用“说人话需求确认工具包”，不要直接展开治理设计。
相关参考：

- [起步阶段：故事与测试对齐](../references/起步阶段_故事与测试对齐.md)
- [说人话需求确认工具包](../references/说人话需求确认工具包.md)

### 4.3 角色层与技能层的区分

本技能对 `agent` 与 `skill` 的区分很严格：

1. 角色层（`agent`）定义角色职责、权限边界、输入输出和控制位置
2. 技能层（`skill`）定义可复用方法、步骤、清单和适用边界

因此：

1. 不要用工具名来命名长期角色
2. 不要用一次性任务来命名长期角色
3. 不要让技能包冒充组织角色
4. 不要让角色层吞掉大量无关技能

### 4.4 四层文档视角

除了结构家族，本技能还默认从四层看文档系统：

1. `truth_source`：定义事实的真源
2. `execution_object`：承载当前工作的过程载体
3. `status_projection`：帮助快速恢复状态的摘要层
4. `display_projection`：面向外部的展示层

这四层帮助团队判断：

- 哪些文件定义事实
- 哪些文件承载工作过程
- 哪些文件只是帮助接手
- 哪些文件只是做汇报或展示

对开发/运营并存项目，还要额外记住：

- 开发真源和运营状态不要混写
- 运行事件、观察和补救属于过程载体，不属于开发真源
- 状态摘要可以反映运行情况，但不能替开发真源拍板

## 5. 三种判断视角

### 5.1 系统视角

系统视角主要回答：

1. 项目边界是什么
2. 哪些是慢变量，哪些是快变量
3. 哪些结构应该独立分层
4. 哪些对象之间耦合高

它主要用于：

- 划分结构家族
- 设计边界
- 设计责任面
- 决定什么时候才值得谈目录优化

### 5.2 信息视角

信息视角主要回答：

1. 一个事实从哪里产生
2. 哪些文档在引用、总结或展示它
3. 当前版本应该去哪里找
4. 一个新接手的人最短读取路径是什么

它主要用于：

- 设计真源
- 设计引用链
- 设计当前版本锚点
- 设计读取顺序和恢复链

### 5.3 控制视角

控制视角主要回答：

1. 谁观察状态
2. 谁做决策
3. 谁执行
4. 谁复核
5. 谁能授权回退或重分类

它主要用于：

- 设计角色回路
- 设计复核关口
- 设计回退路径
- 设计变更控制强度

默认至少要能画出一条主回路：

1. `observe`
2. `decide`
3. `act`
4. `review`
5. `rollback_or_improve`

如果项目既持续开发，又持续运行，最好再显式补两条子回路：

1. 开发回路（`dev_loop`）
2. 运营回路（`ops_loop`）

这不是为了增加形式感，
而是为了把下面三件事说清楚：

1. 哪些信号只刷新状态
2. 哪些信号只触发局部补救
3. 哪些信号必须升级成开发变更、验证或回退

## 6. 诊断框架

本技能默认从八个维度诊断项目：

1. 项目阶段
2. 变更风险
3. 协作密度
4. 代理自主度
5. 恢复压力
6. 协作拓扑
7. 工具异构度
8. 文档膨胀与读取成本

正式冻结诊断前，还要先判断自己的理解把握度：

- `high`
- `medium`
- `low`

规则很简单：

1. `low`：先提问，不要急着出方案
2. `medium`：如果不确定点会影响结构分层、工具适配或验收边界，就先补问
3. `high`：可以直接给方案，但仍要显式区分证据和推断

如果项目同时存在持续开发与持续运行，再额外判断三件事：

1. 是否存在独立于开发任务的运行观察、事件或补救动作
2. 这些运行信号是否会持续反推上游真源
3. 当前问题究竟是开发面问题、运营面问题，还是两者接口问题

相关参考：

## 7. 治理强度怎么选

本技能默认从“均衡治理”出发，不轻易走极端。

### 7.1 常见治理模式

1. `lean-bootstrap`
   - 适合早期、低风险、协作密度低的项目
2. `balanced-governance`
   - 大多数 AI Agent 项目的默认起点
3. `controlled-delivery`
   - 适合高风险、高耦合、强合规压力的项目
4. `flow-heavy-operations`
   - 适合持续运行、并行流很多的项目
5. `recovery-realignment`
   - 适合已经失真、需要先恢复可信度的项目

### 7.2 方法怎么组合

常见做法不是选一套口号，而是组合方法：

- 规格驱动（`Spec-Driven`）
- 看板流（`Kanban`）
- 迭代节奏（`Agile / Sprint-like`）

简单判断：

1. 慢变量多、耦合强、验收边界重要时，偏向规格驱动
2. 并行流多、状态变化频繁时，偏向看板流
3. 需要阶段收敛和里程碑复核时，偏向迭代节奏
4. 风险、合规或回退成本高时，再加复核和变更控制关口

这里要补一个重要限制：

`flow-heavy-operations` 是运行态叠加模式，不是一个替代开发治理的单独宇宙。
如果运行信号会持续反推规则、对象、流程或验收边界，
就不应只说“状态更多了”，
而应显式拆成开发回路与运营回路。

## 8. 默认流程库

这个技能不只看“结构怎么分”，也会判断“哪些流程要成为默认做法”。

### 8.1 常见默认流程

通常优先考虑这些：

1. `low_token_recovery_chain`
2. `discussion -> decision_package -> task_or_decision`
3. `truth_source -> execution_object -> status_projection -> display_projection`
4. `mechanism_review -> repair_or_split`

### 8.2 常见条件流程

只在需要时才加：

1. `adversarial_inquiry -> defense -> convergence`
2. `isolated_multi_role_deliberation`
3. `proposal -> validation -> shadow/canary -> activation_or_rollback`
4. `skill_seed -> package_contract -> active_package`
5. `contract_gap -> closure_topic -> downstream_resume`
6. `growth_signal -> lifecycle_review -> compact_or_archive`
7. `input_classify -> counterexample_check -> low_bandwidth_package_or_reframe`
8. `ops_signal -> classify -> state_refresh_or_local_fix_or_dev_change_or_rollback`
9. `observation -> mechanism_review -> candidate_or_absorb`

规则是：

1. 只保留当前项目真需要的最小流程集合
2. 不要把重流程当成熟的象征
3. 不要把历史项目里的仪式原样照搬进来

其中有两条只在开发/运营并存项目里特别重要：

1. 运营信号先分流，不要直接改写开发真源
2. 运行观察先进入候选或机制复核，不自动升级成技能真源

## 9. 多工具环境怎么处理

这个技能默认支持多工具环境，但处理原则很明确：

1. 工具入口是适配入口，不应默认是真源
2. 工具品牌不是项目角色
3. 同一治理模型应能跨工具存活
4. 工具可以提供不同入口，但不能私自改写项目语义

如果一个项目只能通过某个工具入口才能找到规则、对象、流程、技能或角色，说明治理还不够稳。

### 9.1 AI-Native 同构团队的默认理解

如果项目满足以下前提：

1. 主要参与者都通过 AI 工具工作
2. 项目级规则文件跟仓共享
3. Git 仓库承担目录级全量共享

那么“多人协作”默认可以先理解成：

`多个执行上下文共享同一份 Git 存储`

这里的执行上下文可以是：

- 一个人在两个工作站上同时工作
- 多个人分别通过 `Codex / Claude Code / AntiGravity` 工作
- 多个 agent 会话并行推进

在这种模式下，默认先看的不是工具品牌差异，
而是：

1. 共享仓库根路径
2. 项目级规则文件集合及优先级
3. 路径级写权
4. branch / worktree 并发隔离方式

这里还要特别区分：

1. `Codex / Claude Code / AntiGravity` 这些程序通常属于 runtime 或适配入口
2. 仓库中的 `AGENT.md / CLAUDE.md / CODEX.md / SKILL.md / RULES.md` 可能是项目真源的一部分

也就是说，
在 AI-Native 同构团队里，
“工具入口不是真源”这句话默认指的是程序和入口本身，
不应误伤仓库里被全团队共同消费的项目级规则文件。

只有当批准权、外部责任、runtime 语义偏差或高成本发布风险明显出现时，
才需要把问题升级成更重的多人协作治理。

## 10. 短口令怎么治理

如果项目希望用“继续开发”“开始审计”“反思”“推进”这类短口令驱动工作，本技能要求把它写成一份显式约定。

这份约定至少要写清楚：

1. 稳定意图编号
2. 直接动作还是路由动作
3. 允许的自然语言别名
4. 可选修饰项
5. 默认读取顺序
6. 流程绑定
7. 执行角色
8. 输出要求
9. 遇到歧义时怎么停、怎么问、怎么回退

核心原则：

1. 易用性放在别名和修饰项层
2. 语义稳定性放在主意图层
3. 不要让短口令只活在聊天习惯里

相关参考：

- [意图触发约定](../references/意图触发约定.md)

## 11. 输出应该长什么样

本技能正式回答前，会先读 [输出约定](../references/输出约定.md)。
默认先输出核心必答区块：

1. 方向与边界锚点
2. 项目画像
3. 当前主要失真或治理压力
4. 推荐治理模式
5. 推荐经典流程库
6. 项目结构家族图
7. 推荐入口与恢复链
8. 推荐下一步实施顺序
9. 明确不建议的做法

只有在确实重要时，才补条件区块：

- 跨层共享矩阵
- 推荐项目结构分层
- 推荐角色控制回路
- 推荐版本与同步纪律
- 对象家族检索与适配策略
- 意图触发与执行约定
- 工具可移植性约束
- 文档生命周期与压缩策略
- 输入分型与低带宽压缩包
- 开发/运营双治理结构

这套设计的目的，是防止技能默认吐出一份又长又重、但对当前问题并不必要的治理全案。

## 11.5 理解型输入怎么处理

这套技能现在默认承认：

1. 有一类摩擦回合主要是 `understanding`
2. 这类摩擦常由投影层不适合人的认知带宽触发
3. 但很多看起来像“请解释一下”的输入，实际仍在补 `constraint / evidence / decision`

所以默认处理顺序不是“先继续解释”，
而是：

1. 先做输入分型
2. 再做反例检查
3. 如果是纯理解同步，优先补低带宽压缩包
4. 如果不是，就明确说出它改写了什么边界

默认低带宽压缩包有四种：

1. `anchor package`
2. `decision package`
3. `evidence package`
4. `route package`

## 11.6 开发/运营并存项目怎么收口

这类项目最容易出的问题，不是文档不够多，
而是把开发真源、运行状态、事件补救和技能沉淀全混在一起。

默认至少要分清四件事：

1. 哪些对象属于开发面，负责定义规则、对象、流程和变更
2. 哪些对象属于运营面，负责承接运行、观察、事件和补救
3. 哪些信号只刷新状态，哪些必须升级成开发变更或回退
4. 哪些经验只保留候选，哪些才吸收到长期资产

最关键的约束是：

1. 运营信号不能直接改写开发真源
2. 自动反思不等于自动晋升技能
3. 运行观察先经机制复核，再决定是候选保留还是正式吸收

## 12. 三种起点的推荐顺序

### 12.1 现有仓库

优先顺序：

1. 找出现有真源和摘要页
2. 划分结构家族与责任边界
3. 先止住漂移
4. 再收拢真源
5. 最后才修目录和命名

### 12.2 新项目

优先顺序：

1. 锁定方向与边界
2. 定义最小真源集合
3. 定义最小结构家族集合
4. 定义一个状态入口和一个恢复链
5. 只启用最小默认流程

### 12.3 恢复与重整

优先顺序：

1. 先止血
2. 选定当前可信事实
3. 降级旧摘要页和旧展示页
4. 恢复责任边界与交接链
5. 稳住之后再做长期重构

建议维护者默认做这几件事：

1. 新增概念前，先在 [语言体系规范](./语言体系规范.md) 固定中文主叫法
2. 改技能真源时，优先改 `SKILL.md` 和 `references/`
3. 改对外包装时，不要反过来让包装文档定义真源
4. 每次新增流程或术语，都检查是否真的降低了认知负担
5. 活跃文档变厚时，及时做生命周期复核和压缩
