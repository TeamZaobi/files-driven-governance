# 当前阶段补完计划

这份文档只回答一件事：
在 `runtime -> candidate -> capability` 这条自迭代能力链正式接入，
入口层、体检分层和外部 workflow benchmark 收口之后，
`files-driven` 当前到底完成到哪，下一 tranche 还差什么，按什么顺序稳妥推进。

它不是底层能力模型真源。
能力模型本体仍以 [项目治理能力模型](./项目治理能力模型.md) 为准。
它也不是变更账本。
新增、调整和删除仍由 [CHANGELOG.md](../CHANGELOG.md) 记录。

如果要看 2026-04-14 这轮入口收口、体检分层补完与外部治理反思吸收后的暂停基线，
读 [阶段状态检查点_2026-04-14](./阶段状态检查点_2026-04-14.md)。

## 1. 当前阶段判断

当前项目已经从“解释型方法论仓库”推进到“最小可执行元 skill”，
并进一步进入“执行面补完期”，
但还没有推进到“下游项目可直接获得完整系统体检与通用外部 workflow control-plane starter”的阶段。

一句话说：

**现在已经有统一真源、starter、动作面、官方自迭代样例、分层体检入口和回归；下一步要把剩下仍停在说明层或 draft checker 的能力，继续推进成稳妥的执行资产。**

## 2. 已完成的主线

1. 底层真源已经统一到 [项目治理能力模型](./项目治理能力模型.md)，并显式分开能力模型 `v1 -> v2 -> v2.1` 与 governed pack / contract `tranche v1`。
2. 元 skill 动作面已经收成 `install / register / repair / audit`，starter、bootstrap、`manage` CLI 和 scaffold validator 已形成最小闭环。
3. 自迭代能力链已经正式接入：
   - 模型层：`runtime -> candidate -> capability`
   - 流程层：`capture_evidence -> recall_history -> split_target -> candidate_trial -> activation_or_rollback`
   - 参考层：[运行观察与能力晋升](../references/运行观察与能力晋升.md)
   - 示例层：[capture-candidate-activation](../examples/capture-candidate-activation/README.md)
4. 项目边界已经有仓库级故事、测试、非目标和验收责任人，不再只靠 `README` 自述。
5. 回归面已经覆盖统一真源、starter 冷启动、元 skill 动作面和自迭代样例链。
6. agent-facing 入口已经开始被显式回归：`agents/openai.yaml -> SKILL.md -> reference/example -> 输出约定` 不再只是隐含关系。
7. `audit` 已从单一 `scaffold` 基础体检，扩到 `scaffold / pack / runtime / governance / adoption` 五层入口：
   - `scaffold` 是正式基础体检
   - `pack / runtime / governance / adoption` 当前都是 draft checker
8. 宿主化知识工作场景已经进入正式说明和 fixture：
   - [宿主化知识工作场景矩阵](./宿主化知识工作场景矩阵.md)
   - [examples/hosted-knowledge-governance](../examples/hosted-knowledge-governance/README.md)
9. 外部 workflow 改造线已经显式吸收一组新变量：
   - `human authority layer`
   - `machine-readable control plane`
   - `hook policy`
   - `workflow-control-plane / model-routing / hook-policy / state / events`
   但当前只按“个案 benchmark”看待，还没有晋升成正式 starter 或统一 CLI 子命令。

## 3. 当前仍缺什么

1. 自迭代链当前主要停在 `reference + example + regression + draft checker`，还没有进入 starter 默认播种物或 `manage` CLI 的直接能力面。
2. 候选包、待试验项和激活/回退对象虽然已经有官方 example，但还没有成为下游 pack 的默认脚手架路径。
3. `governance` draft checker 对 hidden authority surface、入口冒充本体、scope 混写的覆盖还偏窄。
4. 记忆链已开始最小硬化：`runtime` 现在会检查 `recall_note.md`、`split_decision.md` 和 `workflow.events.jsonl` 的关键字段，但 validator 对 `reason_refs / artifact_refs / recall chain` 的更深约束还没有完全收口。
5. `adoption` 当前只检查本仓库入口，不是通用 adoptability audit。
6. 外部 workflow control-plane 合同化这条线，目前还只是个案 benchmark，不足以直接推出通用 starter、统一 CLI 或专项 audit。
7. 本地验证仍依赖显式安装 [requirements-dev.txt](../requirements-dev.txt)；对第一次接手的人来说，零摩擦度还不够。
8. AI-Native / skill-driven 场景已经有多条分散的 deterministic contract / alignment / fixture 回归，但还没有完全收口成统一的 layered E2E 验收矩阵；真实在线模型 + 多 agent + 外部工具的黑盒 runtime 仍不是当前 tranche 的主验收面。

## 4. 当前 tranche 的目标

这一 tranche 不再继续扩世界观。
目标只有四件事：

1. 把自迭代能力链和分层体检，从“模型里成立、入口能解释”推进到“关键高风险点可被 draft checker 直接指出，并收口成 AI-Native / skill-driven 的 layered E2E 验收矩阵”。
2. 把 `governance -> memory chain -> adoption` 的下一阶段研发顺序固定下来，避免又回到“谁想到什么就补什么”。
3. 显式保留“外部 workflow control-plane 合同化”这条个案 benchmark，但不把一次校本化改造直接泛化成仓库级真源。
4. 继续保持 `files-driven = reference implementation + regression fixture` 的定位，不把当前项目误判成平台产品。

## 5. 本 tranche 接受什么

1. 官方 reference、example pack、`README`、`SKILL`、使用手册和输出约定统一使用同一条自迭代主路径。
2. 仓库级计划文档显式写清当前完成状态、剩余缺口、执行顺序、验收线和非目标。
3. 回归测试守住：
   - 入口文档能把读者带到官方 reference 和官方 example
   - self-iterating chain 的 pack 资产可被 validator 验证
   - Agent 请求能够回到正确的 skill 路由与正式输出骨架
   - 当前阶段计划没有脱离项目故事、统一真源和元 skill 动作面
   - 个案 benchmark 仍被明确说成个案，不被偷偷写成通用 starter

## 6. 本 tranche 不接受什么

1. 自动把所有运行观察升级为能力
2. 平台级候选评审系统或完整自动修复器
3. 全量 stale graph、图数据库或知识图谱执行层
4. hostile-environment 级安全、签名、细粒度访问控制
5. 把 repo 自述、reference 和 example 全部一次性升级成厚合同
6. 不把一次外部 workflow 改造经验直接抬成仓库级通用模板
7. 在 stage graph 里直接硬编码品牌或供应商名，而不经过逻辑角色与 model routing

## 7. 补完顺序

### Step 1. 入口统一

状态：本轮已完成

- `README`、`SKILL`、使用手册、输出约定统一回到同一条路径：
  `运行观察 -> 证据包 -> 历史召回 -> 拆分出口 -> 候选试验 -> 激活或回退`
- 对外先说清“当前不能热改真源”，再决定是否进入候选。

### Step 2. 官方执行资产

状态：本轮已完成

- 增加 [运行观察与能力晋升](../references/运行观察与能力晋升.md)
- 增加 [capture-candidate-activation](../examples/capture-candidate-activation/README.md) 及其 pack 资产
- 增加专项回归，冻结这条链的主路径、对象合同和 validator 通过线

### Step 3. 分层体检入口

状态：本轮已完成第一段

- `manage audit` 已具备 `scaffold / pack / runtime / governance / adoption` 五层入口
- 当前要继续守住：
  - `scaffold` 仍是正式基础体检
  - `pack / runtime / governance / adoption` 仍是 draft checker
  - 不把这些入口误讲成系统级全身体检

### Step 4. starter / manage 接入

状态：下一 tranche 待完成

- 让下游项目可以不从 example 反推，而是直接从 starter 或 `manage` CLI 获得候选链的最小脚手架
- 至少明确：
  - 证据包模板从哪里起
  - 候选试验对象如何命名和落盘
  - 激活或回退结果如何回写 pack 实例，而不误写能力真源

### Step 5. governance / memory chain 加深

状态：下一 tranche 待完成，`runtime` 记忆链已开始最小硬化

- 让 `governance` draft checker 继续补：
  - hidden authority surface
  - 入口页 / 状态页 / 展示页冒充真源
  - scope 混写和 self-hosting 绑定遗漏
- 让 `runtime` 继续补：
  - `recall_note.md`
  - `split_decision.md`
  - `workflow.events.jsonl`
  - 候选试验缺 `failure_signals / rollback_path`
- 让 validator 继续补：
  - `reason_refs / artifact_refs`
  - `recall chain`
- 目标不是把 validator 变成平台，而是先守住最容易漂的高风险边界

### Step 6. 采用者体验收口与 E2E 验收

状态：下一 tranche 待完成

- 为第一次接手者补足低摩擦验证路径
- 把 AI-Native / skill-driven 场景收口成统一的 layered E2E 验收矩阵
- E2E 优先以 `CLI / runner` 为真实触发面，并尽量模拟真实生产场景，而不是只验 prose route contract
- 以 [AI-Native 与 Skill 驱动 E2E 验收矩阵](./AI-Native与Skill驱动E2E验收矩阵.md) 作为当前验收路线
- 至少明确：
  - 本地怎样一条命令装起测试依赖
  - 什么时候跑 starter / pack validator
  - 什么时候只看官方 reference，什么时候需要进入 example 或 contract
  - 哪些 lane 只是局部回归，哪些 lane 才算真正的 E2E

### Step 7. 外部 workflow benchmark 是否晋升

状态：下一 tranche 待评估

- 当前先把外部 workflow control-plane 合同化只按个案 benchmark 处理
- 晋升口径要保持“薄共同骨架 + 变量槽位”：共同保留诊断、分工、证据、晋升四件事，其余按 case 裁剪
- workflow 只是这次 benchmark 的实例；下一次可能是 hooks 或 scaffolding，不要把这次的文件组合抬成统一模板
- 晋升前至少要补：
  - 不少于 `2-3` 个外部项目样本
  - 样本之间具备宿主、项目类型和品牌路由差异
  - 逻辑角色优先于品牌路由这条规则被反复证明有效
  - 明确哪些项目不该上这套 control-plane 合同面
- 只有条件满足后，才评估是否进入正式 starter / CLI / audit

## 8. 验收线

这一 tranche 收口后，至少要满足：

1. 新接手的人只读 [README.md](../README.md)、[PROJECT_STORIES_AND_TESTS.md](../PROJECT_STORIES_AND_TESTS.md) 和本计划，就能说出项目当前完成状态、剩余缺口和非目标。
2. 遇到运行中反复纠偏时，维护者能沿官方路径判断它是停在记录、进入候选，还是具备晋升条件。
3. `capture-candidate-activation` 不再只是示意图，而是被正式回归守住的官方 example pack。
4. Agent 作为 skill 使用界面时，常见用户原话能稳定命中正确 reference/example 与正式输出骨架。
5. 当前 tranche 不会因为自迭代链接入，就把仓库错误拉向“自动晋升平台”或“全量知识图谱系统”。
6. 当前 tranche 不会把一次外部 workflow 校本化改造经验，误写成已经泛化成功的通用能力。
7. 当前 tranche 会把 benchmark 维持在“薄共同骨架 + 变量槽位”的层级，不让 workflow 的实例形状变成唯一模板。

## 10. 当前结论

当前项目最强的部分已经不是解释世界观，
而是已经具备：

1. 统一真源
2. 最小元 skill 动作面
3. 官方自迭代样例链
4. 明确项目边界
5. 可跑回归

下一 tranche 最该补的，不是继续扩概念，
而是按顺序补：

1. `governance` draft checker 的 hidden authority / scope drift 覆盖
2. `reason_refs / artifact_refs / recall chain` 这条记忆链硬校验，并收进 AI-Native / skill-driven 的 layered E2E 验收矩阵
3. `adoption` 从本仓入口 draft checker 逐步扩到更通用的 adoptability 体检
4. 外部 workflow 个案 benchmark 是否满足晋升门槛
5. AI-Native / skill-driven layered E2E 矩阵是否能稳定验收主要入口、治理边界和自迭代链

最后才考虑更广的真实 runtime 黑盒级 agent e2e。
