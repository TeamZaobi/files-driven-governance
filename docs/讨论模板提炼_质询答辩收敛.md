# discussion 模板沉淀的质询-答辩-收敛验收

## 这轮要解决的问题

当前仓库已经有 `discussion`、`adversarial_inquiry`、`decision_package`、`process_projection` 这些流程名，但它们主要停留在 reference 层。

直接问题有四个：

1. 用户知道有这些流程名，但不知道什么时候该开正式 `discussion`
2. 用户知道“可以质询”，但不知道什么时候该从讨论升级到 `adversarial inquiry`
3. 用户知道“可以收口”，但不知道什么时候该压成 `decision_package`、什么时候该进 `task / decision`
4. 用户知道“可以补投影”，但不知道什么时候才需要 `process_projection`

如果继续只补 reference，不补官方主路径和条件分支样例，用户仍然要去其他项目逆向理解。

## 候选方案

### 方案 A：直接抽象成多套模板族

- `discussion -> decision_package -> task_or_decision`
- `discussion -> adversarial_inquiry -> defense -> convergence`
- `discussion -> process_projection -> topology_supplement`

优点：

- 结构齐全
- 看起来覆盖面大

风险：

- 首次使用的人仍然不知道先走哪条
- 仓库更像资产面，而不是首轮可用路径
- 容易把源项目里的稳定结构和源项目私货一起抽象上来

### 方案 B：先补一条主路径，再把其余做成条件分支

主路径：

`discussion -> issue_ledger -> decision_package -> task_or_decision -> archive`

条件分支：

- 争议过大时，补 `adversarial_inquiry -> defense -> convergence`
- 多工具过程不透明时，补 `process_projection -> status_projection`

优点：

- 用户第一次就能走通
- 分支什么时候出现更清楚
- 更容易防止入口继续膨胀

风险：

- 分支能力的机器约束暂时还弱
- `decision_package` 和 `process_projection` 这轮还没有 schema/validator

## 质询

### 质询 1

这轮是不是又在把 HQMDClaw 和 AIJournal 的做法拼接进来，只是换了更抽象的名字？

### 质询 2

如果一次补太多 template 和 example，是否会让仓库更难进入，而不是更容易进入？

### 质询 3

如果 `decision_package` 和 `process_projection` 还没有 schema/validator，这轮是不是只是文档增强，不算真正可验收能力？

### 质询 4

README、手册、reference、examples 同时新增后，会不会再次出现“看起来都对，但整体像拼接”的问题？

## 答辩

### 对质询 1

不搬源项目私货，只搬稳定结构。

保留的只有这些：

- `discussion` 的最低字段和出口判断
- `question_id -> response_ref -> closure_authority` 的质询闭环
- `decision_package` 的压缩职责
- `task / decision / archive` 的分流判断
- `process_projection` 作为派生接手面，不升格成控制真源

不带入：

- 源项目角色名
- 领域对象
- 编号体系
- 项目专用 gate

### 对质询 2

不把所有模板平铺成第一入口。

这轮采用一条主路径加两条条件分支：

- 主路径：`discussion -> issue_ledger -> decision_package -> task_or_decision`
- 条件分支一：`adversarial_inquiry -> defense -> convergence`
- 条件分支二：`process_projection -> status_projection`

也就是：

- 先解决“讨论怎么收口”
- 再解决“什么时候必须质询”
- 最后解决“什么时候需要过程投影”

### 对质询 3

这轮先做到“可直接使用 + 可回归验证”，不先做到“全部合同化”。

通过线不是：

- 所有结构都已有 schema

通过线是：

- 用户第一次进入仓库就能沿主路径走通
- 条件分支的触发条件清楚
- 有阶段测试和最终测试守住新增资产

合同化留作下一轮判断项，而不是这一轮的通过前提。

### 对质询 4

通过三个动作压制“拼接感”：

1. 不新增新的独立入口页，只改现有 `README` 和 `使用手册`
2. 主路径先行，条件分支只在对应问题下出现
3. 用阶段测试验证“入口文档是否真的把主路径和分支说清”

这一轮子代理补充指出的具体风险也已经处理：

- `discussion-decision-task` 目录里原先同时存在 `decision-package.md / task.md` 和 `decision_package.md / task_or_decision.md` 两套路径，容易让 example 看起来像拼接资产面
- 当前已经收成单一 canonical path，并在测试里显式防止旧文件名回流
- pack 级 validator 也已经纳入讨论主路径测试，避免只剩字符串存在性检查

## 收敛结论

接受方案 B。

这轮正式范围如下：

1. 新增一份 `discussion` 收口与晋升 reference
2. 新增一个官方主路径 example：`discussion-decision-task`
3. 新增两个条件分支 example：
   - `adversarial-convergence`
   - `multi-tool-process-projection`
4. 把主路径和条件分支接进 `README`、`使用手册`、`PROJECT_STORIES_AND_TESTS`
5. 在阶段中间和方案末尾跑端到端测试

这轮明确不做：

- `decision_package` schema
- `process_projection` schema
- 新增独立入口页
- 把源项目案例整包搬进来
