# files-driven 使用手册

这份手册只解释“怎么用这套共享存储上的项目治理能力解决问题”，不独立定义底层本体。
它是执行解释与落地入口，不平行定义 `files-driven` 的能力模型。
涉及 `v1 / v2 / v2.1` 的统一定义，以 [项目治理能力模型](./项目治理能力模型.md) 为准。

## 先确认前提

按这份手册协作时，默认有 4 个前提：

1. 你的团队主要通过 AI 工具工作。
2. Git 是共享底座。
3. 项目级规则文件跟仓共享。
4. 你希望把协作建立在共享存储上，而不是聊天记录上。

这份手册关心的不是文档写得漂不漂亮，
而是共享存储上的事实、写权、流程和恢复如何保持稳定。

## 如果你的读者几乎没有软件工程基础

不要让他们把这份手册当第一份材料。
这份手册是给要真正理解问题、成因和治理路径的人用的，
不是给第一次上手的人背术语用的。

先让他们读 [非工程背景起步](./非工程背景起步.md)，先只会 3 件事：

1. 哪份文件算数
2. 今天先做什么
3. 哪些文件先别改

等这 3 件事已经能稳定问清，再回来读这份手册。

回到这份手册后，可以把用户侧入口先压成一条翻译链：

1. `体检`：先看哪里漂、哪里混写、哪里先别动
2. `诊断`：判断当前问题落在真源、入口、脚手架还是运行链
3. `治疗`：给最小改动方案，优先止血而不是重构
4. `复查`：改完以后重新跑校验
5. `随访`：在后续迭代里复看是否复发

这条翻译链只服务采用体验。
底层执行时，仍要回到 `scope`、一级关口、四层视角和正式合同语义。

## 1. 先判断你是不是在解决这些问题

如果你的项目反复出现下面这些情况，就按 `files-driven` 来收口：

1. 你说不清哪份文件才是真源。
2. `README`、docs、ppt、状态页慢慢变成事实来源。
3. 不同 AI 工具带出不同口径，项目语义开始漂。
4. 多个执行上下文并行工作，但没有明确写权边界。
5. 新接手的人只能靠聊天记录恢复上下文。
6. 材料越来越多，但你还是说不清“项目现在到底是什么状态”。

你要解决的不是“文档太乱”。
你要解决的是：**同一仓库里的事实、过程、状态和展示被混写以后，项目会持续漂移。**

如果你当前想用 `audit` 开始，不要先把它理解成“全身体检”。
在当前版本里，它默认只表示脚手架基础体检：
先检查 starter、registry、route、manifest、profile 这条最小资产链是否闭环；
如果你明确在查 pack / contract 资产，可以直接运行 `manage audit --layer pack`；
如果你在查官方的 `运行观察 -> 候选试验 -> 激活或回退` 链，可以运行 `manage audit --layer runtime`；
如果你在查入口冒充真源、scope 混写或 hidden authority surface，可以运行 `manage audit --layer governance`；
如果你在查本仓库的新手路径、低带宽解释和宿主名先行分诊是否站得住，可以运行 `manage audit --layer adoption`。
如果你需要把 `scaffold / pack / runtime / governance / adoption` 这五层明确分开，
直接读 [体检分层矩阵](./体检分层矩阵.md)。

```mermaid
flowchart LR
    ctx["多个执行上下文<br/>人 / AI / 并行会话"] --> gen["快速生成大量材料<br/>docs / ppt / 状态页 / 说明稿"]
    gen --> mix["事实、过程、状态、展示混写"]
    mix --> drift["真源漂移<br/>写权不清"]
    drift --> recover["恢复成本升高<br/>交接依赖聊天"]
```

## 2. 为什么这些问题会反复出现

### 2.1 执行上下文越来越多

你现在面对的通常不是一个写入者，而是多个并行写入者：

- 一个人在两个工作站上工作
- 两个人各自开 AI 会话
- 一个主 agent 带多个并行子 agent

结果很直接：

1. 写入点变多。
2. 覆盖风险变高。
3. 恢复成本变高。

### 2.2 生成速度太快

AI 可以很快产出：

- docs
- ppt
- 状态页
- 汇报稿
- 说明稿

真正的问题通常不是“没有材料”，而是“材料太多，但没有固定真源”。

### 2.3 工具品牌容易盖过项目事实

如果团队一直在讨论：

- 谁在用哪个工具
- 哪个工具界面更方便
- 哪个工具自带什么 rules

那你大概率已经偏题了。

你真正要先弄清的是：

1. 仓库里哪些规则文件是项目真源。
2. 这些规则文件和工具入口分别承担什么职责。

### 2.4 靠聊天恢复上下文越来越贵

聊天适合解释过程，不适合稳定承载这些东西：

- 真源入口
- 当前版本锚点
- 当前卡点
- 当前可写边界

如果团队主要靠聊天交接，恢复成本只会越来越高。

## 3. 不要把这些方法当主解

### 3.1 不要让 `README` 长期兼任全部真源

`README` 适合作为入口，不适合长期同时承担：

1. 规则
2. 状态
3. 教程
4. 执行说明

一旦它什么都写，很快就会混层。

### 3.2 不要把聊天、Issue、群消息当真源

这些载体适合讨论，不适合做稳定真源。

原因很简单：

1. 它们更像过程记录。
2. 它们不适合做固定读取顺序。
3. 它们不适合长期承载版本锚点和写权边界。

### 3.3 不要只依赖工具本地 rules

如果项目事实只活在某个工具的本地 rules 里，问题会很快出现：

1. 规则不跟仓，团队不能共享。
2. 工具一切换，项目语义就漂。

你可以用工具 rules，但不能只有工具 rules。

### 3.4 不要以为目录规范就够了

目录只能告诉你“文件放在哪里”，不能替你回答这些问题：

1. 谁能直接写。
2. 哪个是当前版本锚点。
3. 哪个只是摘要。
4. 哪个只是展示。

### 3.5 不要指望 CI/CD 或 `workflow` 自己解决真源问题

自动化很有用，但自动化不会替你判真源。

如果你还没分清：

- 哪份文件定义事实
- 哪份文件只是展示

那自动化只会更快地放大混乱。

## 4. 你要怎么用 Files-Driven

这一节只保留够用的动作顺序，不重新展开完整模型总论。
如果你要系统理解 `scope`、四层、结构家族、三种判断视角和治理强度，
改读 [完整说明书](./完整说明书.md) 第 `4` 到 `7` 节；
如果你要改底层本体，回 [项目治理能力模型](./项目治理能力模型.md)。

### 4.1 先判动作、scope 和一级关口

默认不要一上来先分四层，也不要先分结构家族。
先把这轮动作停在正确入口：

1. 当前动作是什么。
2. 当前站在哪个 `scope` 说话。
3. 当前对象属于能力资产还是项目实体。
4. 当前该停在哪个一级关口。

如果这 4 件事还没说清，后面的分层和分类很容易越判越乱。

### 4.2 再补四层视角

动作、`scope` 和一级关口稳定后，再把材料补到这四层：

1. `truth_source`
2. `execution_object`
3. `status_projection`
4. `display_projection`

这一步回答的是：

1. 哪些文件定义事实。
2. 哪些文件记录这次运行。
3. 哪些文件只是帮助接手和恢复。
4. 哪些文件只是对外展示。

如果你需要稳定键名、schema 或更完整的判法，
不要继续在这份手册里追总论，直接读 [完整说明书](./完整说明书.md) 和相关 `references/`。

### 4.3 再补结构家族标签

只有在责任边界还不够清楚时，才继续补结构家族标签。
这一步的目标不是把目录弄复杂，而是让每类文件只承担自己的职责。
详细判法直接看 [结构家族定位约定](../references/结构家族定位约定.md)。

### 4.4 协作先按共享存储组织

只有当团队已经进入多人、多工具、多会话并行阶段时，
再把协作问题收口到共享存储：

**多个执行上下文共享同一份 Git 存储。**

先把下面 4 件事写清：

1. 哪些路径谁能写。
2. 哪些路径只能生成。
3. 哪些路径只能复核。
4. 并行时用 `branch`、`worktree` 还是独立会话隔离。

这样做的作用很直接：

1. 你不用再围着工具品牌争论。
2. 你能直接按路径和写权组织协作。
3. 你能把交接建立在仓库入口上，而不是聊天回放上。

如果项目还没到这个阶段，不要先把协作抬成共享存储问题；
先稳住真源、下一步和最小写权边界。

### 4.5 把 docs、ppt、状态页都当可再生投影

你可以大量生成：

- docs
- ppt
- 状态页
- handoff

但默认都按下游投影管理。

这意味着：

1. 它们可以生成。
2. 它们可以刷新。
3. 它们可以短期热修。
4. 它们不能长期脱离上游真源独立演化。

这条规则也适用于宿主化知识工作页面：

- `Obsidian` 里的 `MOC`、状态页、`Canvas`、`Bases`
- `Notion` 里的项目主页、database view、状态汇报页
- `Docs / Sheets / Slides` 里的汇总页、dashboard、deck

默认都先按索引、状态或展示处理，
不要因为它们“最常被打开”就直接升格成真源。
如果这次问题正好是“这些页面里谁算数、谁能直接改、谁只是投影”，
直接读 [宿主化知识工作场景矩阵](./宿主化知识工作场景矩阵.md)。

### 4.6 每轮改动都挂回用户故事和测试锚点

每次改动前，先回答 3 个问题：

1. 这次改动对应哪个真实用户故事。
2. 这次改动要让哪个测试或验收锚点更稳。
3. 这次改动解决的是哪个真实问题，而不是哪个抽象概念。

这样做的目的只有一个：让治理始终贴着真实问题走。

```mermaid
flowchart LR
    problem["真实问题"] --> stories["用户故事"]
    stories --> tests["测试 / 验收锚点"]
    tests --> truth["真源与写权设计"]
    truth --> exec["执行与协作"]
    exec --> feedback["真实反馈"]
    feedback --> stories
```

### 4.7 把落地过程看成一条连续收口链

项目开始失真时，不要同时解决所有问题。
按下面这条顺序收口，整套方法会更稳：

1. 先稳住协作面：让所有执行者先共享同一套仓库事实，不再按工具品牌各说各话。
2. 再稳住边界面：让每个 pack 或子项目先说清服务场景、用户故事、测试锚点和失败边界。
3. 再稳住控制面：把高风险 `workflow` 从说明文本推进成机器和人都能稳定消费的控制载体。
4. 最后稳住迭代面：把每轮升级重新挂回用户故事、测试锚点和培训入口，避免能力再次漂成零散材料。

你可以把这个仓库里现有材料理解成这条链上的不同承载点：

1. [AI-Native 同构团队协作](../references/AI-Native同构团队协作.md) 主要服务协作面。
2. `BOUNDARY.md`、smoke pack 和 validator 主要服务边界面。
3. `workflow.contract.json`、`workflow.state.json`、`workflow.events.jsonl` 和 `checks` 主要服务控制面。
4. [PROJECT_STORIES_AND_TESTS](../PROJECT_STORIES_AND_TESTS.md) 和这份手册主要服务迭代面。

```mermaid
flowchart LR
    s1["协作面<br/>共享同一套仓库事实"] --> s2["边界面<br/>先有场景、故事、测试和失败边界"]
    s2 --> s3["控制面<br/>workflow 可执行、可验证、可恢复"]
    s3 --> s4["迭代面<br/>升级持续回到故事、测试和培训入口"]
```

### 4.8 运行观察该停在记录、候选还是晋升

当你在运行中反复遇到纠偏、补救或临时替换时，先不要把它直接写回能力真源。先按下面三步判断：

1. 只是这一次运行的记录，还是已经反复出现、值得保留？
2. 还没验证够、也还不能跨场景复用的，是否先停在候选层？
3. 只有经过复核、验证和回退确认，能稳定复用的，才晋升到能力真源？

判断口径可以很直接：

1. 只影响这一次执行、下一轮未必复现的，停在运行记录。
2. 已经反复出现，但还没验证够的，先进候选保留。
3. 经过机制复核、能解释回退路径、也能跨场景复用的，才进入能力晋升。
4. 运行观察、临时纠偏和候选结论都可以写进说明与状态，但不要把它们直接当成已完成晋升。

这类问题优先按下面顺序回看：

1. 先读 [运行观察与能力晋升](../references/运行观察与能力晋升.md)，明确这次到底停在记录、候选还是晋升判断。
2. 再看 [capture-candidate-activation](../examples/capture-candidate-activation/README.md)，把 `运行观察 -> 证据包 -> 历史召回 -> 拆分出口 -> 候选试验 -> 激活或回退` 走完整。

### 4.9 用户说“用 files-driven 治理知识管理”时

如果用户直接说“帮我治理项目里的知识管理”，
默认不要先讲术语，也不要先改页面；
先按这条最短路径执行：

1. 先分诊：这是治理问题，还是工具操作问题
2. 再盘点：当前有哪些规则页、规格页、`MOC`、状态页、`Canvas`、dashboard、deck、database view
3. 再分类：哪些是 `truth_source`，哪些是 `status_projection / display_projection`
4. 再定边界：哪些页面可以直接改，哪些只能回写上游
5. 再定顺序：先读真源，再读实例，再读状态，再读展示
6. 最后才做宿主操作：把剩余编辑任务交给对应宿主能力

当场景是“wiki 化知识管理 + Obsidian 兼容”时，额外守住两条：

1. `MOC / Hub` 默认是导航或状态投影，不默认升格成真源
2. 会落盘进 vault 的 `.md -> .md` 链接，内部引用用 `[[笔记名]]` 或 `[[相对路径/笔记名|显示名]]`

推荐最低交付物：

1. 一份真源清单
2. 一份投影清单（状态/展示）
3. 一条写权边界
4. 一条读取顺序
5. 一条回写路径（状态页、汇报页、`MOC` 不能反写真源）

对应参考：

1. [宿主化知识工作场景矩阵](./宿主化知识工作场景矩阵.md)
2. [官方读取顺序](../references/官方读取顺序.md)
3. [输出约定](../references/输出约定.md)
4. 还要再回到总模型时，再读 [经典治理流程库](../references/经典治理流程库.md) 和 [项目治理能力模型](./项目治理能力模型.md)。

## 5. 逐类处理常见问题

### 5.1 真源漂移

如果你看到这些现象：

1. `README` 开始变成规则来源。
2. 某个状态页更新得比上游更快。
3. 讨论里定了规则，但没人回写。

你就按这个顺序处理：

1. 先判谁是真源。
2. 再判谁只是摘要或展示。
3. 固定读取顺序。
4. 固定版本锚点。

### 5.2 工具入口冒充项目规则

如果你看到这些现象：

1. 工具一切换，团队对项目的理解也跟着变。
2. 本地 rules 和仓库规则互相打架。
3. 工具品牌名慢慢变成角色名。

你就按这个顺序处理：

1. 区分 runtime / 工具入口 与 项目级规则文件。
2. 明确 `AGENT.md / CLAUDE.md / CODEX.md / SKILL.md / RULES.md` 的真源地位。
3. 让工具入口回指真源，而不是自己再定义一套项目语义。

如果你这次不是只想判清楚，而是要真的修改多套 rules，
继续读 [多规则工具治理与精准修改协议](../references/多规则工具治理与精准修改协议.md)。

### 5.3 docs、ppt、状态页反向定义事实

如果你看到这些现象：

1. ppt 已经改了，上游还没改。
2. docs 写了新的承诺，规则里却没有。
3. 状态页已经默认放行下一步。

你就按这个顺序处理：

1. 把这些材料明确归到下游投影。
2. 允许生成和刷新。
3. 禁止它们长期脱离上游真源独立演化。

### 5.4 多人并行互相覆盖

如果你看到这些现象：

1. 两个人同时改同一块真源。
2. 多个 agent 并行，但没有隔离。
3. 合流时只能靠口头解释。

你就按这个顺序处理：

1. 先按路径分工。
2. 并行时用 `branch`、`worktree` 或独立会话隔离。
3. 合流时以上游真源裁决。

### 5.5 交接只能靠聊天回放

如果你看到这些现象：

1. 新接手的人先翻聊天记录。
2. 看了很久还是不知道先读什么。
3. 能说出发生过什么，却说不清现在该做什么。

你就按这个顺序处理：

1. 交接只压最少必要入口。
2. 明确真源入口、当前状态、当前卡点和下一步。
3. 不让聊天记录承担主要恢复职责。

### 5.6 团队把协作误解成工具选择

如果你看到这些现象：

1. 讨论一直停在“哪个 AI 工具更好用”。
2. 项目级规则文件被当成某个工具的包装文案。
3. 团队没人说得清哪些路径才是真正共享面。

你就按这个顺序处理：

1. 先区分 runtime / 工具入口 与 项目级规则真源。
2. 把协作改写成共享存储问题。
3. 让项目级规则文件跟仓共享，并成为统一读取入口。

### 5.7 pack 看起来完整，但没有边界入口

如果你看到这些现象：

1. pack 里已经有合同文件，但说不清服务谁。
2. 只有 schema 和状态文件，没有显式场景和失败边界。
3. 新接手的人不知道从哪里判断是否越界。

你就按这个顺序处理：

1. 先用 `BOUNDARY.md` 锁定场景、故事、测试和失败边界。
2. 再补 `workflow.contract.json`、对象合同和其他最小结构资产。
3. 用 smoke pack 和 validator 守住最小边界入口。

### 5.8 `workflow` 只会解释，harness 读不稳

如果你看到这些现象：

1. `workflow` 主要靠 Markdown 或口头说明来解释。
2. runtime 要靠猜状态、猜关口、猜下一步。
3. 检查、恢复和自动化执行依赖额外聊天补充。

你就按这个顺序处理：

1. 把 `workflow.contract.json` 固定成控制语义真源。
2. 把 `workflow.state.json` 和 `workflow.events.jsonl` 固定成实例层。
3. 把 `checks` 与 validator 接成最小可执行合同链。
4. 用“减少运行时歧义、稳定检查、降低恢复成本、提升 harness 可消费性”评估这类改动。

### 5.9 讨论一直停在聊天层

如果你看到这些现象：

1. 一件事已经反复讨论，但还是没有正式落点。
2. 大家都在发意见，但没人说得清这件事下一步去哪。
3. 争议点、证据缺口和执行动作全混在一段长聊天里。

你就按这个顺序处理：

1. 先把它开成正式 `discussion`，不要继续留在聊天里。
2. 先把未决点压成 `issue_ledger`，不要继续把所有问题混写。
3. 需要裁定时先压 `decision_package`。
4. 只剩执行时进 `task`，只剩长期边界时进 `decision`。
5. 争议很大时，再补 `adversarial_inquiry -> defense -> convergence`。

第一次看样例时，先看：

- [讨论收口与晋升](../references/讨论收口与晋升.md)
- [discussion-decision-task](../examples/discussion-decision-task/README.md)

## 6. 你今天怎么开始

### 6.1 新项目第一天

先做这 5 件事：

1. 确定共享仓库。
2. 确定哪些项目级规则文件跟仓共享。
3. 确定哪些路径谁能直接写。
4. 确定并行时的隔离方式。
5. 确定最小交接格式。

然后立刻补这 3 个锚点：

1. 当前最重要的 1 到 3 个用户故事。
2. 当前最重要的 3 到 8 个测试或验收锚点。
3. 至少一个失败或越界边界。

不要第一天先做这些事：

1. 大量目录设计。
2. 长篇 docs。
3. 复杂 `workflow` 合同。
4. 大量展示页。

只有在下面这些问题已经出现时，再补更重的 `workflow`、实例状态和检查链：

1. harness 或 validator 需要稳定读取控制语义。
2. 任务需要跨会话恢复，而不是一次性完成。
3. 项目已经有明确 `gate`、证据和 `stop` 条件。
4. 人能看懂说明，但机器和接手者都读不稳。

### 6.2 已经出现问题的项目，先这样开始

如果你的项目已经出现真源漂移、状态混写、交接困难或工具口径打架，不要一上来做大全量重构。

先按这 5 步止血和收口：

1. 先停掉最容易继续制造漂移的写入面。
2. 先指定一个当前真源入口和一个当前状态入口。
3. 先把最痛的 1 到 3 个问题压成用户故事和测试锚点。
4. 先补最小写权边界，明确哪些文件还能写、哪些只能生成、哪些只能复核。
5. 再决定要不要补 `BOUNDARY.md`、`workflow` 合同、实例状态和检查链。

你可以按这个顺序落：

1. 先找出哪几份 docs、ppt、状态页或说明稿正在反向定义事实。
2. 先把这些文件降回投影层，不再允许它们直接改写真源。
3. 先明确一个恢复入口，让新接手的人知道先读哪里、当前卡在哪里、下一步是什么。
4. 先守住最小边界和最小测试，而不是同时收拾所有目录。

不要先做这些事：

1. 一次性重写全部文档。
2. 一次性重排整个目录树。
3. 一次性把所有 `workflow` 都合同化。
4. 一次性补齐所有展示层和自动化。

### 6.3 如果一个议题还没到 task 或 decision

先按这个顺序处理：

1. 先开正式 `discussion`
2. 再补 `issue_ledger`
3. 再判断是否需要 `decision_package`
4. 只在 owner、object 和 stop condition 都清楚时，再进 `task`
5. 只在长期边界已经收敛时，再进 `decision`

如果你已经出现下面这些信号，再加一层正式质询：

1. 这件事会直接改变公开口径、治理边界或高风险放行。
2. 这件事反复兜圈，但没有逐点应答链。
3. 这件事如果判错，代价明显高于普通讨论。

如果多工具过程已经不透明，再补：

1. `process_projection`
2. 必要时再补 `topology_supplement`

### 6.4 一次任务开始前

开工前先写清这 7 件事：

1. 这次任务的目标是什么。
2. 这次任务的真源在哪里。
3. 当前版本锚点在哪里。
4. 这轮可写边界是什么。
5. 这轮改完后要刷新哪些下游投影。
6. 这次任务对应哪个用户故事。
7. 这次任务要让哪个测试锚点更稳。

如果一个改动既对不上当前问题，也对不上当前故事和测试，先把它当成范围外动作处理。

### 6.5 执行时

默认按这个顺序推进：

1. 先改上游真源。
2. 再补运行状态。
3. 最后刷新 docs、ppt、状态页和汇报页。

### 6.6 完成前

至少复核这 4 件事：

1. 有没有越过真源直接改投影。
2. 有没有让状态摘要越权。
3. 有没有让工具包装文案冒充项目规则。
4. 有没有留下难以恢复的断点。

## 7. 团队协作纪律

把下面这些纪律写清并长期执行：

1. 不用工具品牌名命名长期角色。
2. 不让 `README` 长期兼任全部真源。
3. 不让 docs、ppt、状态页长期独立演化。
4. 不让交接主要依赖聊天记录。
5. 不让并行工作在同一写入面无隔离进行。
6. 不让治理结构脱离当前用户故事和测试锚点长期自转。

## 8. 如果你要给团队做培训

### 第零段：先降到工作语言

如果听众里有很多人几乎没有软件工程基础，
不要一上来先讲 `scope`、`projection`、`contract tranche` 或 `harness`。

第一段只讲 3 件事就够了：

1. 哪份文件算数
2. 今天先做什么
3. 哪些文件先别改

这一步的目标不是让他们理解完整模型，
而是先让他们能安全开工，不会把状态页、README 或说明稿误改成真源。

### 第一段：先讲项目到底在解决什么问题

讲清 3 件事：

1. 真源为什么会漂。
2. 为什么 AI 生成越强，越容易把投影写成真源。
3. 为什么协作的主问题是共享存储和写权，而不是工具品牌。

### 第二段：再讲问题怎么挂回用户故事和反馈

讲清 3 件事：

1. 每个治理问题都要能回到真实用户故事。
2. 每个关键修正都要有测试或验收锚点。
3. 每轮反馈都要推动真源、过程或投影再次校准。

### 第三段：再讲为什么常见办法不够

讲清 4 件事：

1. 只靠 `README` 不够。
2. 只靠聊天不够。
3. 只靠工具 rules 不够。
4. 只靠目录规范不够。

### 第四段：再讲你们怎么解决

讲清 4 件事：

1. 先判动作、`scope` 和一级关口。
2. 再补四层与结构家族，不让它们抢默认起手式。
3. AI-Native 团队先按共享存储理解协作。
4. 下游投影默认可再生。

需要举例时，不要按“最近多了哪些文件”去讲。
按一条连续能力链讲：

1. 先讲共享存储协作。
2. 再讲边界入口。
3. 再讲 `workflow` 控制载体。
4. 最后讲用户故事、测试锚点和培训入口怎么把迭代钉回真实问题。

仓库里的对应材料只用来承载这条链，不单独当成四个零散亮点来讲。

### 第五段：最后讲团队今天怎么执行

按操作顺序讲就够了：

1. 新项目怎么起步。
2. 已经出现问题的项目怎么止血和收口。
3. discussion 什么时候开、什么时候晋升。
4. 一次任务怎么推进。
5. 多人并行怎么分工。
6. 交接怎么写。

## 9. 让团队至少记住这句话

**你们共享的不是聊天，也不是工具界面；你们共享的是同一套仓库里的项目事实。**
