# 宿主化知识工作场景矩阵

状态：`working matrix`

这份文档不是 `Obsidian`、`Notion`、`Google Docs / Sheets / Slides` 的使用手册。
它只回答一个问题：

**当用户先说宿主名、页面名或视图名时，怎样把问题翻译回 `files-driven` 的治理判断。**

底层本体仍只认 [项目治理能力模型](./项目治理能力模型.md)。

## 1. 什么时候该用这份矩阵

只有在下面两类情况才需要打开这份矩阵：

1. 用户先说宿主名或页面名，但真正想问的是：
   - 哪份页面算数
   - 谁能直接改
   - 哪些只是索引、状态、展示或视图
   - 改坏后怎么恢复
   - 当前读取顺序应该怎样定
2. 你担心把工具操作问题误判成治理问题，或反过来把治理问题误判成纯工具操作

如果用户只是要：

- 建一个 `Canvas`
- 改一个 `Base`
- 调一个 `database view`
- 修改单篇 `Doc`
- 调整 `Slide` 版式

这份矩阵不该抢活。
那是宿主操作问题，不是 `files-driven` 的治理主问题。

## 2. 默认判定规则

先按这四条判：

1. 页面类型不等于治理身份
2. 先看它是否实际定义事实、规则、写权或放行
3. 再看它是 `truth_source`、`execution_object`、`status_projection` 还是 `display_projection`
4. 如果它只是索引、视图、看板、汇报或包装入口，默认先降为投影或适配面

一句话总规则：

**先看页面承担什么责任，不要先看它长得像什么工具页面。**

## 3. 先分“治理问题”还是“工具操作问题”

| 用户当前在问什么 | 默认路由 |
| --- | --- |
| 哪份页面算数、哪些只是索引/状态/展示、当前真源在哪里 | `files-driven` |
| 哪个页面可以直接写、哪个只能总结、哪个需要回写上游 | `files-driven` |
| 当前页面之间互相打架、命名漂移、口径不一致、恢复困难 | `files-driven` |
| 如何创建、编辑、排版、联动某个宿主对象 | 宿主专用 skill 或工具能力 |
| 既有治理问题，又有宿主操作 | 先 `files-driven` 判边界，再把剩余操作转给宿主专用能力 |

## 4. 宿主矩阵

### 4.1 Obsidian

| 宿主对象 | 默认治理身份 | 什么时候可能升格 | 常见误判 | 红旗信号 |
| --- | --- | --- | --- | --- |
| 项目规则笔记 / 规格笔记 | 可成为 `truth_source` | 项目显式指定其为当前规则或事实真源，且有稳定回指 | 觉得“只是笔记软件，所以不算真源” | 规则只存在于零散对话，不落到稳定笔记 |
| `MOC` / 索引页 / Hub 页 | 默认 `status_projection` 或导航入口 | 只有项目明确把它定义成唯一入口规则页时才可升格 | 把索引页当成事实真源 | `MOC` 写出了上游没批准的新规则或新事实 |
| 日报 / 周报 / 状态页 | 默认 `status_projection` | 几乎不应升格成真源 | 把状态页当作当前事实裁决面 | 状态页比上游规格更新得更快 |
| `Canvas` | 默认 `display_projection` 或过程压缩面 | 只有极少数显式声明为正式设计对象时才例外 | 把视觉编排当成事实定义 | `Canvas` 独自定义了关口、权限或下一步 |
| `Bases` / 查询视图 | 默认 `display_projection` 或索引视图 | 只有底层数据本身承担对象真源时，视图仍不升格 | 把视图结果当成对象真源 | 过滤结果替代了上游对象定义 |

### 4.2 Notion

| 宿主对象 | 默认治理身份 | 什么时候可能升格 | 常见误判 | 红旗信号 |
| --- | --- | --- | --- | --- |
| 项目主页 | 默认入口或 `status_projection` | 只有项目明确把它定义成总入口规则页时才升格 | 把项目主页当成所有事实真源 | 首页写了底层规则页没有的要求 |
| 规则页 / 规格页 | 可成为 `truth_source` | 页面承担正式项目规则或事实定义，并有稳定链接关系 | 觉得“数据库旁边的说明页都只是注释” | 真正约束项目的规则只活在评论或数据库字段里 |
| database schema / 对象库 | 可成为 `truth_source` 或 `execution_object` | 当它实际承担对象定义或实例主表时 | 把任何 database 都当真源 | 视图或汇总字段反过来改写对象定义 |
| database view / board / calendar | 默认 `status_projection` 或 `display_projection` | 几乎不应单独升格 | 把视图当成对象真源 | 某个 view 独自决定下一步或写权 |
| 状态汇报页 | 默认 `status_projection` | 不应升格成真源 | 把周报/总结页当批准页 | 汇报页给出上游未批准的结论 |

### 4.3 Google Docs / Sheets / Slides

| 宿主对象 | 默认治理身份 | 什么时候可能升格 | 常见误判 | 红旗信号 |
| --- | --- | --- | --- | --- |
| `Docs` 规格文档 / 规则文档 | 可成为 `truth_source` | 项目明确指定其为当前规格或规则真源 | 觉得在线文档天然只是协作草稿 | 真正规则在汇报文稿里，而不在规格文档里 |
| `Docs` 会议纪要 / 讨论记录 | 默认 `execution_object` 或证据材料 | 一般不升格成真源 | 把讨论记录当最后拍板 | 纪要里有正式规则，但没有回写上游 |
| `Sheets` 系统主表 / 对象主表 | 可成为 `truth_source` 或 `execution_object` | 当表格本身就是正式对象系统或事实主表 | 把所有表格都当报表 | 聚合列或手工备注替代了正式对象字段 |
| `Sheets` 仪表盘 / 统计视图 | 默认 `status_projection` 或 `display_projection` | 一般不升格 | 把图表或 dashboard 当事实裁决面 | dashboard 比主表先宣布状态变化 |
| `Slides` 汇报页 / 路演页 | 默认 `display_projection` | 几乎不应升格成真源 | 把 deck 当项目当前事实 | 汇报稿比上游规格更像真源 |

## 5. 这些宿主放回四层时怎么理解

| 宿主对象特征 | 默认落位 |
| --- | --- |
| 正式定义规则、事实、对象结构 | `truth_source` |
| 承载运行实例、会议纪要、过程记录、对象主表 | `execution_object` |
| 状态页、周报、看板、dashboard、项目主页 | `status_projection` |
| `Canvas`、`Slides`、对外展示页、汇报页 | `display_projection` |
| 程序入口、插件面板、查询视图、包装命令 | 工具适配面，不单独升格成真源 |

## 6. 这些宿主放回四分法时怎么理解

| 宿主对象特征 | 默认四分法归属 |
| --- | --- |
| 首页必读、总入口说明 | `入口规则` |
| 项目内某个 skill / agent 的操作规则 | `能力规则` |
| 只对当前项目成立的业务法则、命名法、版本法 | `项目规则` |
| 规格、实例、对象、状态、展示、候选稿 | `项目实体` |

注意：

- `Canvas / Bases / database view / dashboard / deck` 这些名字本身，不决定它属于哪一类
- 真正决定归属的是：它是否承担事实定义、执行记录、状态摘要或展示输出

## 7. 反误触发边界

下面这些请求，默认不该由 `files-driven` 单独吞掉：

- “帮我新建一个 Obsidian Canvas”
- “帮我调整 Obsidian Bases 视图”
- “帮我改一个 Notion database view”
- “帮我排版这份 Google Doc”
- “帮我优化这页 Google Slides”

下面这些请求，默认应先触发 `files-driven`：

- “Obsidian vault 里 `MOC`、项目主页、状态页和 `Canvas` 互相打架，哪份算数？”
- “Notion 里项目主页、database view 和状态汇报页互相不一致，应该听谁的？”
- “共享的 Docs / Sheets / Slides 里，哪些是真源，哪些只是汇报材料？”
- “这些宿主页面里，哪些能直接改，哪些必须回写上游？”

## 8. 推荐最小回答形状

当用户先说宿主名时，默认先回四句：

1. 当前你在问的是治理问题，还是工具操作问题
2. 这几类页面里，哪一类默认更接近真源
3. 哪几类页面默认只是索引、状态或展示
4. 这次先去看哪份上游文件，哪些页面先别直接改

只有当这四句站稳后，再继续展开：

- 四层
- 四分法
- 读取顺序
- 写权边界
- 恢复链
