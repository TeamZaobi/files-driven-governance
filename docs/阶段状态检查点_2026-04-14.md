## 5. 外部治理反思吸收

本次暂停前，额外吸收了一次外部项目治理反思：

- 对话来源：`019d8991-467a-74b1-9941-3a5d19c63807`
- 主题：`诊断Workflow脚本化与Hook治理`

这次外部反思的结论，不直接改写本仓当前真源；
但它提供了一组很重要的“新变量 / 外部对照基线”。

### 5.1 提炼出的新变量

1. `human authority layer` 与 `machine-readable control plane` 应显式分层，不能继续混成同一层 prose 说明。
2. `skills/*/SKILL.md` 更适合承担叶子能力包和高召回说明，不适合继续充当控制面真源。
3. 如果要把 prose-first workflow 升级成 machine-readable control plane，最小合同面通常至少需要：
   - `workflow-control-plane`
   - `model-routing`
   - `hook-policy`
   - 运行时 `state / events`
4. 模型分工不应直接把品牌名硬编码进阶段图；
   更稳的做法是先定义逻辑角色，再在单独的 model routing 合同里绑定具体模型。
5. 如果控制面开始合同化，统一 CLI 入口和 hook 注册表通常要一起考虑；否则合同、执行和门禁会继续分裂。

### 5.2 这些收获如何作用于本仓

当前把它们先视为：

1. 外部项目治理诊断时的重要 benchmark
2. 未来本仓继续深化 `control plane / hook / runtime contract` 时的参考变量
3. 下一轮引入新修改变量时必须显式评估的一组候选约束

当前不把它们直接升级为本仓既有真源，原因也要说清：

1. 本仓当前已经有自己稳定的 `capability_scope / project_scope / runtime_scope` 和 governed pack 语义。
2. 如果直接把外部项目的 control plane 合同面照搬进来，容易混写“本仓当前能力模型真源”和“外部项目 workflow 改造 benchmark”。
3. 更稳的做法是把它们先作为下一轮设计变量，显式评估后再决定是否进入本仓合同面。
