# 体检分层矩阵

状态：`working matrix`

这份文档只回答一个问题：

**`files-driven` 现在的“体检”到底能查到哪一层，还不能查到哪一层。**

它不是能力模型真源，也不替代 [项目治理能力模型](./项目治理能力模型.md)。
它只把当前 `audit` 的能力边界说实。

## 1. 先把“体检”拆层

当前最容易混的，不是有没有 `audit`，
而是把不同层级的体检混成一个词。

这里统一拆成五层：

| 体检层 | 主要回答什么 | 当前代表问题 |
| --- | --- | --- |
| `scaffold` | starter、registry、route、profile 这条最小资产链是否闭环 | tracked 文件没注册、route 漏绑、starter profile 漂移 |
| `pack` | governed pack / contract 的边界、合同、对象、状态、事件是否对齐 | `BOUNDARY.md` 缺口、legacy `schemas/`、合同和实例打架 |
| `runtime` | 运行中的状态、事件、观察、候选、激活链是否越级或失真 | 运行观察越级写真源、展示页冒充激活、候选缺回退 |
| `governance` | 真源、作用域、写权、工具适配面是否被混写或偷权 | `README` 冒充真源、工具入口冒充规则、hidden authority surface |
| `adoption` | 第一次接手的人是否真能起步，而不是读完还不知道做什么 | 非工程入口过重、宿主名先行时无法自然落路由 |

一句话记忆：

- `scaffold` 查“装得稳不稳”
- `pack` 查“合同链齐不齐”
- `runtime` 查“运行链越不越级”
- `governance` 查“真源和写权稳不稳”
- `adoption` 查“人能不能接住”

## 3. 当前 `audit` 命令到底等于什么

当前 [scripts/manage_files_engine.py](../scripts/manage_files_engine.py) 里的 `audit`，
默认只等于：

**默认 `audit = scaffold 基础体检`**

如果显式运行 `manage audit --layer pack`，
当前会进入 pack validator 的 draft 统一入口。
如果显式运行 `manage audit --layer runtime` 或 `manage audit --layer governance`，
当前会进入对应问题面的 draft checker。

它当前主要检查：

1. `manifest`
2. `hooks`
3. `starter_profile`
4. `registry`
5. `routes`
6. `scaffold`

也就是说，它现在能稳定指出的是：

- tracked 文件和 `files.registry.json` 是否对齐
- starter profile 和 manifest 是否漂移
- route 是否消费了应消费的入口
- hooks 适配面是否缺项目级 hook 真源

它现在还不等于：

- 通用 adoption 黑盒体检
- 通用 runtime 黑盒体检
- 全量 governance 系统体检

## 4. 每一层的体检问题长什么样

### 4.1 `scaffold`

典型问题：

- 新增 tracked 文件后，没进 `files.registry.json`
- `intent.routes.json` 还在引用旧 `file_id`
- starter profile 和 manifest 形状打架
- hooks adapter 存在，但 `governance/hooks.policy.md` 不存在或没注册

当前通过线：

- 现有 `manage audit` 能查到这层

### 4.2 `pack`

典型问题：

- `BOUNDARY.md` 缺故事、测试或失败边界
- `workflow.contract.json`、`objects/*.json`、`workflow.state.json`、`workflow.events.jsonl` 不对齐
- pack 里仍保留 legacy `schemas/*.json`
- `status_projection` 偷带放行语义

当前通过线：

- validator 已能守住不少边界
- `manage audit --layer pack` 已能把这层包装成 draft 体检入口
- 但它仍不是全平台统一体检器

### 4.3 `runtime`

典型问题：

- 运行观察直接热改能力真源
- 展示页冒充正式激活
- 候选试验缺 `failure_signals`
- 候选试验缺 `rollback_path`
- `runtime -> candidate -> capability` 跳步

当前通过线：

- 已有官方 reference、example 和专项回归
- `manage audit --layer runtime` 已能检查官方 observation -> candidate -> activation 链
- 但它仍不是通用 runtime 黑盒体检器

### 4.4 `governance`

典型问题：

- `README`、状态页、汇报页反向定义事实
- `project_scope` 四分法误套到 `capability_scope`
- 工具入口、包装命令或投影页偷长成 authority surface
- `repo.files-driven / skill.files-driven / meta-skill capability / downstream project instance` 混写

当前通过线：

- 已有模型真源、入口结构回归、专项 reference
- `manage audit --layer governance` 已能检查入口回指真源、投影/适配面不偷权、self-hosting scope 绑定
- 但它仍不是全量 governance 系统体检器

### 4.5 `adoption`

典型问题：

- 首次接手的人读完入口仍不知道先做什么
- `audit` 被误解成“全身体检”
- 宿主名先行时，用户问题落不到治理判断
- 说明层和入口层分工失稳

当前通过线：

- 已有入口压缩和翻译层
- 已有宿主化知识工作场景矩阵
- `manage audit --layer adoption` 已能检查本仓库的新手路径、低带宽解释和宿主名先行分诊
- 但它仍不是通用 adoptability 体检器

## 5. 当前最准确的能力口径

如果要对外说实，当前应该这样说：

1. `audit` 当前默认是 `scaffold` 基础体检
2. `manage audit --layer pack` 已提供 pack 级 draft 体检入口
3. `manage audit --layer runtime` 已提供 runtime 级 draft 体检入口，但当前只覆盖官方 observation -> candidate -> activation 链
4. `manage audit --layer governance` 已提供 governance 级 draft 体检入口，但当前只覆盖入口回指真源、投影/适配面不偷权、self-hosting scope 绑定
5. `manage audit --layer adoption` 已提供 adoption 级 draft 体检入口，但当前只覆盖本仓库的新手路径、低带宽解释和宿主名先行分诊

不该再这样说：

1. “`audit` 已经是全量系统体检”
2. “只要跑了 `audit` 就算做完体检”
3. “当前已经有完整的 runtime / adoption 黑盒体检器”

## 6. 下一步该怎么补

这块最合理的补法不是一次性做平台，而是按顺序收：

1. 先补 `pack` 体检包装
   - 让 pack validator 能被更稳定地路由和理解
2. 再补 `runtime` 体检最小检查
   - 先守住越级写回、假激活、缺失败信号、缺回退路径
3. 再把 `governance` 体检从 draft checker 扩到更广覆盖
   - 重点是 hidden authority surface、scope 混写、入口冒充本体
4. 再把 `adoption` 体检从当前 draft checker 扩到更广覆盖
   - 重点是首次接手路径和宿主名先行路径
