# 能力边界速查

## 能做

- 判断哪份文件算数，哪份只是说明、状态或展示
- 判断今天先做哪一步，以及当前是否应该停手
- 区分入口规则、能力规则、项目规则、项目实体
- 判断当前问题属于 `scaffold / pack / runtime / governance / adoption` 哪一层
- 在需要时把问题升级到更强控制，例如 `capability-improve`、self-hosting 或 `AI-Native E2E`

## 不能直接替代

- 单纯的工具操作说明
- 单篇文档代写或格式微调
- 任何宿主品牌专属功能的详细操作手册
- 把所有 workflow 一上来都脚本化
- 把 README、metadata 或 quickstart 写成并列本体

## 最小判断句式

面对一个问题，优先用下面的句式回答：

1. 现在哪份文件算数
2. 今天先做哪一步
3. 哪些地方先别动
4. 这次需不需要更强控制

## 什么时候该升级

如果出现下面这些情况，就不该停留在轻解释层：

- 规则、状态和事实页面已经明显漂移
- 需要把判断编码成受控执行
- 正确性已经依赖模型解释、路由、边界或恢复判断
- self-hosting 需要 `conversation/replay E2E` 来验证可回放、可复核、可恢复

## 常见误吞场景

- 用户说了 `Obsidian / Notion / Docs / Sheets / Slides`，不等于一定是治理问题
- 用户要改 `Canvas`、database view、单篇 Doc，不等于一定要启动 `files-driven`
- 用户要一个目录结构，不等于真正需要治理建模
