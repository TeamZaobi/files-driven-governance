# files-driven 能力边界速查

状态：`working draft`

这份文档只回答一件事：
**`files-driven` 现在到底能稳定做什么、不能承诺什么、案例要怎么讲才不跑偏。**

它不是新的能力模型真源，不替代 [项目治理能力模型](./项目治理能力模型.md)。

## 1. 一句话定位

`files-driven` 不是“替你完成所有项目工作”的通用代理，
而是“在 AI-Native / AI-Driven 协作里，先稳住真源、写权、恢复，再决定下一步动作”的治理能力。

## 2. 现在可稳定承诺的能力

| 能力面 | 典型触发 | 最小输出（你至少会拿到什么） |
| --- | --- | --- |
| 真源/投影判定 | “哪份文件算数？”“这页是不是越权改写？” | `truth_source / execution_object / status_projection / display_projection` 的最小判定结果 |
| 动作分诊 | “我现在该先干什么？” | `动作触发 -> 一级关口 -> 二级升级` 的最小动作链 |
| 作用域绑定 | “这是在改能力仓还是改项目实例？” | `capability_scope / project_scope / runtime_scope` 的显式归位 |
| 顶层归属判断 | “这段内容该放哪类资产？” | `入口规则 / 能力规则 / 项目规则 / 项目实体` 的归属结论 |
| 下游项目四动作入口 | “怎么把 files engine 装进项目？” | `install / register / repair / audit` 的执行入口与顺序 |
| `audit` 分层判断 | “体检到底在查哪层？” | `scaffold / pack / runtime / governance / adoption` 的层级判定与下一步 |
| governed pack / harness 收口 | “怎么从 prose 变最小合同链？” | `BOUNDARY.md -> workflow.contract.json -> objects/*.json` 的最小路径 |
| gate 失效硬化 | “为什么 blocked 还在往下写？” | `route / evidence / write / stop` 的失效定位与补强点 |
| discussion 晋升 | “讨论要不要升到质询/决策？” | `discussion -> adversarial inquiry -> decision package -> task_or_decision` 的停位建议 |
| 官方读取顺序/恢复 | “新接手从哪读？” | 可执行的短读取顺序与恢复入口 |
| 宿主化知识工作分诊 | “Obsidian/Notion/Docs 页面冲突怎么判？” | 先判治理问题 vs 工具操作问题，再给真源/写权边界 |
| self-hosting 受控执行 | “把 files-driven 自己改成受控 run” | `capability-improve` + runner 入口和最小 run artifact 路径 |

## 3. 现在明确不承诺的事

| 不承诺项 | 边界说明 |
| --- | --- |
| 代替业务技能完成领域产出 | `files-driven` 管治理，不替代领域 `Skill` 产出业务内容 |
| 把普通工具操作一律当治理问题接管 | 纯编辑 `Canvas / Bases / database view / 单篇 Doc` 默认是工具操作，不是治理收口 |
| 把 `audit` 当“全量系统体检”一次兜底 | 当前 `audit` 是分层能力，不是单次覆盖全系统 |
| 运行中直接热改能力真源 | 默认遵守 `runtime -> candidate -> capability`，不边执行边改真源 |
| 自动补齐所有宿主/品牌工具的深层执行细节 | 当前优先给治理归位和边界，不承诺每个宿主都提供完整操作回放 |
| 用一个案例就冻结长期本体 | 个案先形成候选与证据，不直接改写统一模型真源 |

## 4. 案例讲解的最小句式（强制）

以后讲任何案例，至少按下面 6 句讲完：

1. 这次触发动作是什么（例如：`修改规则` / `回写业务事实` / `audit`）
2. 当前在哪个 `scope` 说话（`capability_scope / project_scope / runtime_scope`）
3. 先停在哪个一级关口（先止血、先归位、先校验，哪一个）
4. 当前真源是谁，哪些只是投影（并标出是否越权）
5. 这轮最小输出是什么（1-3 个可执行动作）
6. 什么条件下升级到二级处理（质询、合同化、runner、回归）

只要这 6 句讲不全，就不要宣称“这个案例已经说明了 skill 能力”。

## 5. 三类常见误判

1. 把“案例过程”当“能力定义”
2. 把“工具品牌差异”当“治理问题本体”
3. 把“说明层文档写得很全”当“执行面已经闭环”

**先把案例翻译回能力句式，再谈案例细节；否则案例越多，边界越模糊。**
