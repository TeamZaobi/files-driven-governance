# files-driven 三层信息架构复盘

当前版本：`v0.5.1`

这份复盘只回答一个问题：
仓库里的说明材料，哪些应该负责入口，哪些只负责解释，哪些才有资格当真源。

如果这个边界不稳，常见后果通常是：

1. `README` 越写越像压缩版本体
2. `使用手册` 和 `完整说明书` 互相重复
3. `reference` 专题页慢慢抢首屏
4. 代理入口和公开入口各说各话

## 1. 当前建议的稳定分层

### 1.1 入口层

入口层只负责三件事：

1. 先回答用户主诉
2. 给出最小下一步
3. 把读者路由到下一层

当前属于入口层的文件：

- [README.md](../README.md)
- [SKILL.md](../SKILL.md)
- [agents/openai.yaml](../agents/openai.yaml)
- [docs/非工程背景起步.md](./非工程背景起步.md)
- [QUICKSTART.md](../QUICKSTART.md)

入口层不要负责：

- 重写底层能力模型
- 长篇铺陈理论来源
- 平行定义术语本体
- 抢走专题 reference 的职责

入口层的通过线：

1. 新手能先听懂“哪份文件算数 / 今天先做什么 / 哪些先别改”
2. 用户能在一屏内知道自己下一份该读什么
3. 入口文件不会把自己写成并列本体

### 1.2 说明层

说明层负责把入口里说不下的东西讲清楚，但默认仍然先讲问题，再讲方法，再讲机制。

当前属于说明层的文件：

- [docs/使用手册.md](./使用手册.md)
- [docs/完整说明书.md](./完整说明书.md)
- [docs/能力雷达与版本演进盘点.md](./能力雷达与版本演进盘点.md)
- [docs/能力覆盖矩阵与历史差分.md](./能力覆盖矩阵与历史差分.md)
- [docs/宿主化知识工作场景矩阵.md](./宿主化知识工作场景矩阵.md)
- [docs/体检分层矩阵.md](./体检分层矩阵.md)
- [references/](../references)
- [docs/v0.5.1_版本说明.md](./v0.5.1_版本说明.md)
- [docs/files引擎脚手架工程.md](./files引擎脚手架工程.md)
- [MIGRATION.md](../MIGRATION.md)

说明层不要负责：

- 重新冻结 `v1 / v2 / v2.1` 本体
- 把专题页写成通用首屏
- 在不同文件里各自发明新的第一判断轴

说明层内部也应再分工：

1. [docs/使用手册.md](./使用手册.md)：从问题到动作，解释今天怎么用
2. [docs/完整说明书.md](./完整说明书.md)：系统化解释为什么这样设计
3. [docs/能力雷达与版本演进盘点.md](./能力雷达与版本演进盘点.md)：盘点能力簇、版本演进和当前缺口
4. [docs/能力覆盖矩阵与历史差分.md](./能力覆盖矩阵与历史差分.md)：反筛模型能力今天照进了哪些入口、脚本和回归
5. [docs/宿主化知识工作场景矩阵.md](./宿主化知识工作场景矩阵.md)：把宿主名、页面名和视图名翻译回治理判断
6. [docs/体检分层矩阵.md](./体检分层矩阵.md)：把 `audit` 的当前层级边界说实
7. [references/](../references)：只做专项下钻，不抢主路径

### 1.3 真源层

真源层只负责冻结本体，不负责做采用体验包装。

当前真源层核心文件：

- [docs/项目治理能力模型.md](./项目治理能力模型.md)

历史兼容入口：

- [docs/项目治理能力模型_v1.md](./项目治理能力模型_v1.md)

真源层不要负责：

- 替用户做首屏分诊
- 承担教程式起步说明
- 和 `README`、`使用手册`、`完整说明书` 竞争入口地位

## 3. 当前发现的主要重叠面

### 3.1 README 仍然偏厚

当前 `README` 已经比之前稳很多，但它仍然同时承载：

1. 首屏分诊
2. 方法学摘要
3. 世界观轴摘要
4. 文件职责表
5. 脚手架入口

这意味着它依旧有继续长回“压缩版总论”的风险。

### 3.2 使用手册与完整说明书边界还需要持续守

当前二者已经初步分工：

- `使用手册` 更偏“问题为什么出现、今天怎么处理”
- `完整说明书` 更偏“系统解释和专题总览”

但它们仍然可能在以下部分重新重叠：

1. 为什么会漂
2. 默认判断顺序
3. 四层与结构家族
4. 多工具与共享问题

### 3.3 reference 最容易再次抢首屏

现在已经收住的几个专题页包括：

- `结构家族定位约定`
- `跨层共享约定`
- `官方读取顺序`
- `工具适配对照表`
- `AI-Native同构团队协作`

它们当前都已经回到“第二层解释”的位置，但后续新增专题页时很容易再次犯同样的问题。

## 4. 建议的稳定职责

以后如果继续维护文档，优先按下面这组问题判断要改哪一层：

1. 这是在回答“用户现在先做什么”吗？
   - 是：改入口层
2. 这是在解释“为什么会这样判断、为什么这样设计”吗？
   - 是：改说明层
3. 这是在修改 `v1 / v2 / v2.1`、作用域、四分法或受控合同链本体吗？
   - 是：改真源层
4. 这是在更新当前 tranche 边界、样例、脚手架、schema 或脚本执行面吗？
   - 是：改侧车资产，不要偷改说明主线

## 5. 下一轮裁剪建议

### P0

1. 继续压缩 `README` 中会重复解释模型的段落，只保留入口必要摘要
2. 守住 `使用手册 = 怎么处理问题`、`完整说明书 = 为什么这样设计`
3. 所有新 reference 开头都先回答“当前不该拿这份专题页做首屏”

### P1

1. 给 `README`、`使用手册`、`完整说明书` 做一个更明确的互链导航
2. 审 `docs/` 内其他公开专题材料，避免它们平行讲一套入口
3. 把“入口层 / 说明层 / 真源层 / 侧车资产”加入维护检查清单

## 6. 最短记忆版

一句话记忆：

- `README / SKILL / metadata` 先解决“现在怎么办”
- `使用手册 / 完整说明书 / references` 再解释“为什么这样办”
- `项目治理能力模型` 才负责定义“本体到底是什么”
- stories / examples / schemas / scripts 是侧车，不抢说明主线
