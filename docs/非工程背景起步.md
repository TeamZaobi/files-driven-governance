# 给非工程背景读者的起步页

这份起步页是给第一次接触 `files-driven`、且几乎没有软件工程基础的读者准备的。
它不要求你先理解 `scope`、`projection`、`contract tranche`、`harness` 这些词。

你现在只要先会 3 件事：

1. 哪份文件算数
2. 今天先做什么
3. 哪些文件先别改

更完整的底层定义，仍以 [项目治理能力模型](./项目治理能力模型.md) 为准。
但第一次上手，不要先去读完整模型。

## 这套 skill 先帮你做什么

先把它当成一个“项目现场翻译器”就够了。
它默认先帮你回答：

1. 这个项目里，哪份文件才是真正算数的
2. 你这次要改的内容，应该改规则、项目内容，还是工具入口
3. 如果现在直接动手，最容易改错哪一层

你不用先理解完整方法论，先把这 3 件事问清就能开始工作。

## 第一次使用时只问这 4 类话

你可以直接这样开口：

1. “先别讲理论，告诉我这个项目现在哪份文件算数。”
2. “我这次只想改这个问题，请告诉我先看哪两个文件。”
3. “这份 README / 状态页 / 说明稿能不能直接改？”
4. “如果我今天只能做一步，最先做哪一步？”

如果回答还是太抽象，就继续追问：

1. “把上面改成工作语言，不要用术语。”
2. “只告诉我：该看哪里、先做什么、先别碰什么。”
3. “如果我做错了，最可能错在哪个文件层级？”

## 收到回答后，你只检查这 4 件事

1. 有没有明确指出哪份文件是真源
2. 有没有告诉你哪些文件只是说明、状态或展示
3. 有没有把下一步压成 1 到 3 个动作
4. 有没有指出哪些地方现在先不要改

只要这 4 件事说清了，你就已经拿到了可执行答案。

## 你暂时不用先学的词

这些词是给维护者、设计者或重度治理场景用的。
第一次上手时，看见它们不用慌，也不用先背：

1. `capability_scope / project_scope / runtime_scope`
2. `truth_source / execution_object / status_projection / display_projection`
3. `contract tranche`
4. `harness`
5. `source_kind`

你只要知道一件事：

- 如果回答里出现这些词，它们应该最终被翻译回“哪份文件算数、今天先做什么、哪些先别改”。

## 什么时候要找维护者或更懂工程的人

出现下面这些情况时，不要自己硬扛：

1. 同一个问题同时牵涉多个工具的 rules
2. 你要改的是 `schema`、validator、starter 或脚本
3. 你发现两个文件都在自称真源
4. 你不确定自己改的是“项目事实”还是“工具包装”
5. 讨论已经不只是说明，而是在改决定、改边界或改验收条件

这时候要把问题升级给维护者，而不是继续只靠聊天猜。

## 给非工程背景读者的最短工作流

默认照这个顺序走：

1. 先说清你这次想解决的具体问题
2. 让 skill 告诉你哪份文件算数
3. 只改被允许改的那一层
4. 改完后再确认这次有没有动到真源、状态页或展示页

如果只记一句话，就记这一句：

**先找算数的文件，再动手；先分清真源和说明，再谈怎么改。**

## 看懂以后再往下读什么

如果你已经能稳定问出“哪份文件算数、今天先做什么、哪些先别改”，
再按需要往下读：

1. 想知道为什么会漂：读 [使用手册](./使用手册.md)
2. 想让团队统一说法：读 [README](../README.md)
3. 想执行这个 skill：读 [SKILL](../SKILL.md)
4. 想看完整底层模型：读 [项目治理能力模型](./项目治理能力模型.md)
