# 对外一页能力地图

## 一句话介绍

`files-driven` 是一个面向 AI-Native / AI-Driven 项目的项目治理能力。
它不先替团队堆流程，而是先帮团队回答三件事：

1. 哪份文件算数
2. 今天先做哪一步
3. 哪些地方先别动

当项目进入多人、多代理、多工具协作阶段时，它再继续帮助团队稳住真源、控制写权边界，并让协作过程可检查、可恢复。

## 它解决什么问题

典型失控信号包括：

- README 被慢慢写成真源
- 讨论页、任务页、状态页各自长出新规则
- 工具入口各自包一层解释，口径越来越飘
- 上下文断掉后，项目只能靠猜恢复
- 明明已经 `blocked`，流程还是继续往前滑

## 它怎么启动

对下游项目，默认先用四个动作起步：

1. `install`
2. `register`
3. `repair`
4. `audit`

对第一次采用者，先不强推术语，先把行动压成工作语言：

- 先确认哪份文件算数
- 再确认今天先做什么
- 最后确认哪些文件先别改

## 它交付什么

`files-driven` 交付的不是一套固定模板，而是一套可复用的治理判断和最小执行面：

- 真源与投影边界
- 入口规则、能力规则、项目规则、项目实体的归属判断
- starter、registry、route、profile 这一条最小脚手架资产链
- 需要时可升级的 pack、runtime、governance、adoption 体检入口
- self-hosting / AI-Native E2E 场景下的回放验证口径

## 它的边界

它适合治理问题，不适合吞掉所有工具操作问题。

- 如果用户在问真源、写权、漂移、投影、恢复或读取顺序，应该由 `files-driven` 接管
- 如果用户只是想改单篇文档、单个视图、单个 canvas 或数据库字段，先按工具操作问题处理
- 如果宿主原生 `workflow / subagents / approvals` 足够稳，优先复用宿主能力，不要为了“更高级”而强上外部控制面
