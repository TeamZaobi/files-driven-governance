# files-driven 对外一页能力地图

状态：`public-facing draft`
更新时间：`2026-04-16`

这页给对外沟通、团队对齐、培训宣讲使用。
它不重写模型本体；底层定义仍以 [项目治理能力模型](./项目治理能力模型.md) 为准。

## 1. 我们到底在卖什么能力

`files-driven` 不是“替你做业务内容”的生成器，
而是“让 AI 协作项目不漂移、可恢复、可审计”的治理能力。

一句话版本：
**先把真源、写权、恢复链稳住，再推进执行。**

## 2. 五个能力产品面

| 能力产品面 | 典型触发 | 你会拿到的最小交付 | 可验证证据 |
| --- | --- | --- | --- |
| `A1` 真源与边界判定 | “到底哪份文件算数？”“这页能不能直接改？” | 一份可执行的真源/投影/写权判断 | [SKILL.md](../SKILL.md) 默认主路径、[输出约定](../references/输出约定.md) |
| `A2` 漂移止血与修复 | README、状态页、规则页互相打架 | `动作触发 -> 一级关口 -> 二级升级` 修复链 | `manage repair`、[关口硬化与稳定放行](../references/关口硬化与稳定放行.md) |
| `A3` 合同化与校验 | prose workflow 扛不住、harness 无法稳定消费 | `BOUNDARY -> workflow.contract -> objects` 最小合同链 | [QUICKSTART.md](../QUICKSTART.md)、`validate_governance_assets.py` |

## 3. 三档服务形态（对外可直接报价/承诺）

| 服务档位 | 适用场景 | 交付边界 |
| --- | --- | --- |
| `L1 基础收口` | 项目开始漂移，但先求稳住 | 真源判定、动作分诊、最小修复顺序、基础体检 |
| `L2 合同化收口` | 需要把流程变可检查可执行 | `BOUNDARY` 锚点、合同链、validator 校验、最小回归 |
| `L3 受控运行收口` | 多代理/高风险流程，需要可恢复可审计 | runner 编排、节点证据、晋升/回退规则、E2E 验收线 |

当前动作面现状：`L1` 已稳定；`L2` 稳定到最小合同链；`L3` 已有官方路径，但部分能力仍在 draft checker 阶段。

## 4. 四步启动方式（外部团队）

1. 给一句真实问题，而不是先给工具名
2. 先拿到“哪份文件算数 + 这轮先做什么 + 哪些先别改”
3. 再决定走 `install / register / repair / audit` 哪条动作面
4. 需要时才升级到合同化、runner、E2E 验收

## 5. 我们承诺什么，不承诺什么

### 承诺

- 把“说不清”变成“可执行下一步”
- 把“靠聊天记忆”变成“可恢复链路”
- 把“流程口号”变成“可校验合同”
- 把“经验”沉淀成“可回归资产”

### 不承诺

- 不替代业务领域 Skill 产出业务内容
- 不把普通工具编辑请求一律误判成治理请求
- 不把 `audit` 夸大成一次全量系统体检
- 不在运行中直接热改能力真源

边界详见 [能力边界速查](./能力边界速查.md)。

## 6. 当前成熟度一句话

- 半闭环：`pack/runtime/governance/adoption` 分层体检仍有 draft 范围限制

详见 [流程工具方法经验全景盘点](./流程工具方法经验全景盘点.md) 和 [能力覆盖矩阵与历史差分](./能力覆盖矩阵与历史差分.md)。
