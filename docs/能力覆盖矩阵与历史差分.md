# 能力覆盖矩阵与历史差分

状态：`working audit`

这份文档不是新的能力模型真源，也不替代 [能力雷达与版本演进盘点](./能力雷达与版本演进盘点.md)。
它只做两件事：

1. 反向检查 `v1 -> v2 -> v2.1` 里长出来的能力，今天到底照进了哪些入口、脚本、合同和回归
2. 把 `v0.2.0 -> v0.5.0` 每一版真正点亮了哪些能力，再按统一口径筛一遍

底层本体仍只认 [项目治理能力模型](./项目治理能力模型.md)。

## 1. 先把三条轴分开

这张矩阵同时会用到三条轴，但不能混写：

| 轴 | 当前口径 | 回答什么 |
| --- | --- | --- |
| 模型轴 | `v1 -> v2 -> v2.1` | 某个能力点最早属于哪一层世界观演进 |
| 发布轴 | `v0.2.0 -> v0.5.0` | 某个能力是什么时候被做成公开资产 |
| contract 轴 | `tranche v1` | 某个 governed pack / harness 语义冻结到了哪里 |

一句话记法：

- 模型轴回答“能力本来是什么”
- 发布轴回答“仓库什么时候把它做出来”
- contract 轴回答“workflow / harness 这条机读链目前冻结到哪”

## 2. 覆盖判定口径

这张表不用主观印象，统一按下面四档记：

| 覆盖档位 | 含义 |
| --- | --- |
| `真源` | 只在 canonical model 里有定义 |
| `说明` | 已进入 README / SKILL / metadata / 手册 / reference 等说明层 |
| `执行` | 已进入 schema、脚本、contract、example 或 runner 至少一项 |
| `闭环` | 已同时具备真源、入口说明、执行资产和回归守护 |

## 5. 发布版本历史差分

下面这张表只回答“每一版把什么能力点亮成公开资产”，不重复定义模型本体。

| 版本 | 这一版真正点亮的能力 | 点亮了模型里的哪一块 | 当前回看还缺什么 |
| --- | --- | --- | --- |
| `v0.2.0` | 流程库、质询与收敛回路 | 把治理从“只谈结构”推进到“结构 + 流程” | 这一阶段还没有合同、schema 和最小执行面 |
| `v0.2.1` | 结构定位、官方读取顺序、工具适配 | 把 `v1` 的结构法和读取顺序变成外露入口 | 工具适配负边界当时还不够硬 |
| `v0.2.2` | 理解把握度与按需澄清提问 | 给后续的低带宽入口和分诊规则打底 | 还没有说人话工具包和门诊式入口翻译 |
| `v0.2.3` | 输出约定改成“核心必答 + 条件展开” | 让 `v2` 以后的问题导向表达有了最小输出骨架 | 还没正式绑定故事、测试和边界确认 |
| `v0.2.4` | 文档生命周期与压缩 | 把恢复成本和膨胀问题抬成治理问题 | 还没进入合同化或验证链 |
| `v0.2.5` | 起步阶段：故事与测试对齐 | 把“先确认方向与边界”前置 | 还没有标准化的确认工具包 |
| `v0.2.6` | 说人话需求确认工具包 | 把早期边界确认变成显式动作 | 还没上升成意图触发约定 |
| `v0.2.7` | 意图触发约定 | 让短口令不再只活在聊天习惯里 | 仍然缺统一模型真源和最小合同链 |
| `v0.3.0` | `项目治理能力模型 v1`、schema、validator、smoke pack、CI | 正式点亮 `v1` 的分层、合同和验证能力 | 还没统一 `v2 / v2.1`，也还没有元 skill 动作面 |
| `v0.3.1` | AI-Native 同构团队协作、stories/tests、pack 级 `BOUNDARY.md` | 把协作前提、项目入口和 pack 边界补成正式资产 | 这一阶段仍偏 prose-heavy，元 skill 动作面尚未闭环 |
| `v0.4.0` | 统一 `v1 -> v2 -> v2.1`、files engine starter、`install / register / repair / audit` | 把治理主语、作用域约束和元 skill 动作面合成可执行入口 | `audit` 还只是 scaffold 基础体检 |
| `v0.4.1` | 强化控制能力、宿主优先、控制强度轴、执行面轴 | 给 `v2.1` 之后的控制补强路径重新定标 | 这版是方向纠偏，不是独立的强执行版本 |
| `v0.5.0` | self-hosting control-capability 脚手架、`capability-improve`、repo treatment runner、运行观察晋升链、体检入口翻译层 | 把控制强化、自迭代路径和入口翻译层一起收口 | layered audit、宿主化场景和记忆链校验仍未完全执行化 |

## 6. 反筛下来，目前最明显的残缺

不是每个残缺都属于模型缺失。
更常见的情况是：模型里已经有了，但今天还没有完全照进入口、执行面或回归。

### 6.1 已经基本闭环的能力

1. `v1` 的四层、受控合同链和 validator / smoke / CI
2. `v2` 的治理主语与能力 / 实体区分
3. `v2.1` 的三作用域、四分法默认边界和 `self-hosting` 显式声明
4. `v0.4.0` 之后的 `install / register / repair / audit` 首屏动作面
5. `v0.5.0` 的 self-hosting runner 和 control-capability 执行面

### 6.2 还没有完全照进去的能力

1. `audit` 五层体检
   - 文档、`SKILL`、metadata 已经能解释
   - 但统一动作面仍只覆盖 `scaffold / pack / runtime` draft 层

2. 宿主化知识工作场景
   - 入口已经能判断 `Obsidian / Notion / Docs / Sheets / Slides`
   - 但还缺 host-native 的操作回放和更细的误吞回归

3. 记忆链 / 晋升链硬校验
   - `runtime -> candidate -> capability` 已经有真源、reference 和 example
   - 但 validator 对 `reason_refs / artifact_refs / recall chain` 仍未完全硬化

4. adoption 级体检
   - 采用体验已经进入门诊化语言
   - 现在已有 `manage audit --layer adoption` draft checker，但它还只覆盖本仓库入口，不是通用 adoptability 审计

5. 反向能力覆盖审计的脚本化入口
   - 目前已经有 `SKILL` 路由和回归
   - 但还没有单独的脚本命令专门跑这张能力覆盖矩阵

## 7. 这张表怎么用

如果你接下来要继续审这套 skill，推荐顺序是：

1. 先回 [项目治理能力模型](./项目治理能力模型.md)，确定你在谈的是模型轴还是发布轴
2. 再看这份矩阵，判断某个能力今天到底停在 `真源 / 说明 / 执行 / 闭环` 哪一档
3. 需要看全仓能力簇和当前缺口优先级时，再回 [能力雷达与版本演进盘点](./能力雷达与版本演进盘点.md)
4. 需要继续补角落能力时，再去：
   - [宿主化知识工作场景矩阵](./宿主化知识工作场景矩阵.md)
   - [体检分层矩阵](./体检分层矩阵.md)

一句话收口：

**当前 `files-driven` 的主问题已经不是“模型有没有”，而是“哪些能力已经闭环，哪些能力只停在说明层，哪些能力还没真正进入动作面”。**
