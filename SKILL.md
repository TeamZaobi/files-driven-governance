# 文档驱动治理（files-driven）

## 核心定位

把文档当成项目治理的承载介质，而不是被动备注。
这个技能主要回答三类问题：

1. 项目的事实到底写在哪
2. 哪些文档在定义事实，哪些只是在总结、搬运或展示
3. 多人、多代理、多工具并行时，谁能写、谁先读、谁负责复核

适用场景：

- 现有仓库开始出现规则、流程、状态页和说明页互相漂移
- 新项目需要一套够用但不过重的治理结构
- 项目已经失真，需要先止血、再收口、再重整

## 主流程

### 1. 先定起点与边界

先判断当前问题属于哪个起点：

- `existing_repo`
- `greenfield`
- `recovery_or_realignment`

如果情况是混合的，优先按用户眼下最急的问题判断。

在讨论目录、角色或流程之前，先锁一个最小“方向与边界锚点”。
至少要明确：

1. 首批真实使用场景
2. 首批交付物
3. 一到三个核心用户故事
4. 三到七个验收或测试用例，至少包含一个失败或越界边界
5. 明确不做的范围
6. 仅用于校准质量的参考对象
7. 由谁来判断这次算不算达标

提问规则：

- 不要只问一个模糊问题就继续设计
- 对边界还在漂移的项目，先问四到六个短问题
- 先问使用场景、交付预期、成功标准和非目标，再问工具和目录
- 提完问题后，先起草故事和测试，再进入治理设计

边界还不稳时，优先读：

- [起步阶段：故事与测试对齐](references/起步阶段_故事与测试对齐.md)
- [说人话需求确认工具包](references/说人话需求确认工具包.md)

### 2. 诊断并判断把握度

先做八维诊断：

1. 项目阶段
2. 变更风险
3. 协作密度
4. 代理或自动化的自主度
5. 漂移、歧义或交接失效带来的恢复压力
6. 人、代理、系统之间的协作拓扑
7. 工具异构度与跨工具通用需求
8. 文档膨胀与读取成本

然后判断自己的理解把握度：

- `high`
- `medium`
- `low`

处理规则：

- `low`：先问问题，再给方案
- `medium`：如果不确定点会改动结构分类、工具适配、控制回路或验收边界，就先补一到三个问题；否则明确假设后继续
- 边界还在漂移：优先使用起步确认问题集，而不是提前给完整蓝图

如果项目既持续开发，又持续承接运行、反馈、补救或观察信号，
再补一层判断：它是不是“开发/运营并存项目”。
至少看三件事：

1. 是否存在独立于开发任务的运行观察、局部补救或持续值守
2. 这些运行信号是否会反推规则、对象、流程或验收边界
3. 当前项目是否不能只按阶段交付理解，还必须按运行状态持续恢复

如果三者都明显存在，后续默认按“开发回路 + 运营回路”设计，
而不是只画一条单一执行链。

如果对话里出现解释型摩擦，例如“讲解一下”“再检查一下”“我还是担心”，
先做输入分型和反例检查：

1. `understanding`
2. `constraint`
3. `evidence`
4. `decision`

不要因为一句话听起来像追问，就立刻把它当成纯理解同步。
如果确认主要是 `understanding`，再补低带宽压缩包；
详细规则读 [理解型输入与低带宽压缩包](references/理解型输入与低带宽压缩包.md)。

需要时读：

- [基本原则](references/基本原则.md)
- [治理模式选择对照表](references/治理模式选择对照表.md)
- [理解把握度与澄清规则](references/理解把握度与澄清规则.md)

### 3. 划分结构家族与四层

先分八类结构家族：

1. `policy_or_rules`
2. `object`
3. `workflow`
4. `skill`
5. `agent`
6. `execution_object`
7. `status_projection`
8. `display_projection`

对每一类都要判断：

1. 真源是什么
2. 哪些摘要页、展示页或工具入口在投影它
3. 谁负责维护
4. 与相邻文档的同步顺序是什么
5. 哪些地方需要复核或回退关口

对五类核心结构还要额外判断：

1. 当前版本锚点
2. 官方读取顺序
3. 工具适配入口

同时把文档系统按四层看清：

1. `truth_source`
2. `execution_object`
3. `status_projection`
4. `display_projection`

必要时继续区分过程载体：

- `discussion`
- `task`
- `review`
- `decision`
- `handoff`

如果不同工具各自保留自己的 `walkthrough / trace`，
但人和下游 agent 需要一套统一的过程可见性或交接压缩面，
可以从上述过程载体派生一个 `process_projection`。
它不是新的结构家族，也不是新的分层；
它以上游 `execution_object` 为来源，
默认服务于 `status_projection` 或 `display_projection`，
只负责把“这次实际跑了什么”压成统一语义，
不替代原始过程记录，也不冒充真源。
默认不为简单单 agent 直线执行额外补这一层；
只有多工具过程已经不透明时才启用。
如果复杂性主要来自显式 `workflow` 或多 agent 协作，
再条件补一个 `topology_supplement`，
用来说明节点、流转、关口、并行、交接和收敛。

对开发/运营并存项目，必要时再补运行态过程载体：

- `run`
- `rollout`
- `incident`
- `observation`
- `remediation`

这几条边界必须保持清楚：

- `Agent` 定义角色职责与边界
- `Skill` 定义可复用做法
- `Workflow` 定义路径、流转与关口
- `Object` 定义对象结构与状态语义

默认同时用三种视角做判断：

1. 系统视角：看结构边界、分层、耦合、责任面
2. 信息视角：看真源、引用链、版本锚点、同步顺序、恢复链
3. 控制视角：看观察、决策、执行、复核、回退的闭环

控制设计至少要落一条主回路：

1. `observe`
2. `decide`
3. `act`
4. `review`
5. `rollback_or_improve`

如果项目同时持续开发和持续运行，至少再补两条子回路：

1. 开发回路（`dev_loop`）：围绕规格、实现、验证和发布前关口
2. 运营回路（`ops_loop`）：围绕运行、观察、局部补救、反馈和恢复

两条回路之间至少明确三类接口信号：

1. 哪些信号只刷新状态摘要
2. 哪些信号只触发运营侧局部修复
3. 哪些信号必须升级成开发变更、提案验证或回退

这里要记住：

- `status_projection` 只做摘要和导引，不冒充真源
- `display_projection` 不只是展示层，也常是人的低成本接手面
- 运营信号不能直接改写开发真源
- 运行观察默认先归类，不自动升级成技能真源

需要时读：

- [结构家族定位约定](references/结构家族定位约定.md)
- 真源入口已大致判清、但要固定接手顺序时，再读 [官方读取顺序](references/官方读取顺序.md)

### 4. 只在需要时补共享约定与执行路由

多人、多代理或多工具并行时，不要默认大家会自动共享同一套事实。
重要结构要明确跨层共享约定，至少写清：

1. 谁生产
2. 谁消费
3. 哪一层能直接写
4. 哪一层只能总结或展示
5. 谁能看见
6. 什么时候同步
7. 冲突时按什么规则处理
8. 交接时最少要带哪些信息

如果项目希望用“继续开发”“开始审计”“反思”“推进”这类短口令驱动工作，
就把它设计成显式约定，而不是聊天默契。
每个口令至少写清：

1. 稳定意图编号
2. 它是直接动作还是路由动作
3. 允许的自然语言别名
4. 可选修饰项
5. 默认读取哪些状态页和真源
6. 绑定哪条流程
7. 由谁执行
8. 期望输出什么
9. 歧义时怎么停、怎么问、怎么回退

核心规则：

- 工具入口是适配面，不是真源
- 工具品牌不是项目角色
- 易用性放在别名和修饰项层，不放在偷偷变义上
- 一个常用短语只对应一个主意图

需要时读：

- [跨层共享约定](references/跨层共享约定.md)
- 工具入口已经明显影响读取、启动或协作语义时，再读 [工具适配对照表](references/工具适配对照表.md)
- 项目已经决定把短口令做成稳定入口时，再读 [意图触发约定](references/意图触发约定.md)

### 5. 选择治理组合与最小流程集合

默认从“均衡治理”出发，再根据证据决定要不要更轻或更重。

组合规则：

- `Spec-Driven`：适合慢变量多、耦合强、验收边界稳定
- `Kanban`：适合并行流多、状态变化频繁、需要持续可见性
- `Agile / Sprint-like`：适合需要阶段收敛和里程碑复核
- `flow-heavy-operations`：适合持续运行、信号密集、需要显式区分开发面与运营面
- 决策关口、复核关口、变更控制只在风险、合规、自主度或回退成本足够高时加

先选最小流程集合，不要默认全开。

常见默认流程：

- `low_token_recovery_chain`
- `discussion -> decision_package -> task_or_decision`
- `truth_source -> execution_object -> status_projection -> display_projection`
- `mechanism_review -> repair_or_split`

常见条件流程：

- `adversarial_inquiry -> defense -> convergence`
- `isolated_multi_role_deliberation`
- `proposal -> validation -> shadow/canary -> activation_or_rollback`
- `skill_seed -> package_contract -> active_package`
- `contract_gap -> closure_topic -> downstream_resume`
- `growth_signal -> lifecycle_review -> compact_or_archive`
- `input_classify -> counterexample_check -> low_bandwidth_package_or_reframe`
- `ops_signal -> classify -> state_refresh_or_local_fix_or_dev_change_or_rollback`
- `observation -> mechanism_review -> candidate_or_absorb`

不要让每个项目都背上整套流程库，只推荐当前项目真需要的最小集合。
对开发/运营并存项目，优先判断“运营信号分流”和“运行观察晋升关口”两条流程要不要启用。

需要时读：

- [经典治理流程库](references/经典治理流程库.md)
- 争议已经影响关键决策，或需要制度化反方视角时，再读 [反方质询与收敛回路](references/反方质询与收敛回路.md)
- 问题重心是多工具协作、并行分工、证据入口或工具切换成本时，再读 [跨工具团队实践](references/跨工具团队实践.md)

### 6. 按输出约定收尾

正式回答前，先读 [输出约定](references/输出约定.md)。
默认做法：

1. 先按输出约定组织答案
2. 只启用当前诊断真正需要的条件区块
3. 用能解决当前问题的最小答案形状
4. 如果把握度仍是 `medium`，明确写出关键假设
5. 给出有先后顺序的实施动作，不停在原则

常见实施顺序：

- `existing_repo`
  - 先找真源与摘要页
  - 再划结构家族与责任边界
  - 先止漂移，再收真源
  - 最后才修命名和目录
- `greenfield`
  - 先锁方向与边界
  - 再定最小真源和最小结构家族
  - 定一个状态入口和一条恢复链
  - 只启用最小默认流程
- `recovery_or_realignment`
  - 先止血
  - 选定当前可信事实
  - 降级旧摘要页和旧展示页
  - 恢复责任边界与交接链

## 边界约束

- 先诊断，再开药方
- 真源清晰度优先于目录外观
- `status_projection` 必须保持轻、保持可追溯
- 不要让投影层反向改写真源
- 不要让 `discussion` 长期兼任任务桶
- 不要让 `Agent`、`Skill`、`Workflow`、`Object` 互相改义
- 不要让工具品牌变成项目角色
- 没到必要程度，不要默认上重变更控制
- 不要把看似解释型追问自动归成低价值理解同步
- 仓库证据不完整时，要显式区分证据和推断

## 示例请求

- “分析这个多代理仓库的现有文档体系，判断哪些文件是真源，哪些只是状态摘要，并给出收口方案。”
- “我要做一个 AI Agent 驱动的 OpenClaw 项目，请给我一套早期够用、后续还能扩展的文档治理结构。”
- “这个项目的 README、状态页、任务和流程说明已经互相漂移，请判断主要问题和修复顺序。”
- “这个项目很多机制争议都讲不清，请判断哪些话题需要进入反方质询、最小决策包或验证链，哪些保持轻量处理。”
